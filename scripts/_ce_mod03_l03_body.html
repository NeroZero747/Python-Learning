01

Store Member Data

Beginner Members __init__ Attributes

What This Does

This script defines a class with three attributes and creates one object from it. Every attribute is set inside __init__ the moment the object is created, and reading any attribute back is a single dot-access.

member_record.py 
class HealthRecord:                            # define the class -- this is the blueprint
    def __init__(self, member_id, plan_type):  # __init__ runs the moment an object is created
        self.member_id = member_id             # attach member_id to this specific object
        self.plan_type = plan_type             # attach plan_type to this specific object
        self.is_active = True                  # every new record starts active by default

record = HealthRecord("M-001", "Gold")         # create one object -- __init__ runs immediately
print(record.member_id)                        # read the member_id attribute from this object
print(record.plan_type)                        # read the plan_type attribute from this object
print(record.is_active)                        # read the is_active attribute from this object
Terminal $ python member_record.py
M-001
Gold
True

Every new object you create gets its own separate copy of every attribute — change the values you pass in and you get a completely independent record.

02

Check Claim Status

Beginner Claims Methods Updating Attributes

What This Does

This script defines a class with two methods — one that changes an attribute and one that reads it. Calling the second method before and after the first shows how a method can update an object's stored data in place.

claim_status.py 
class Claim:                                       # blueprint for one insurance claim
    def __init__(self, claim_id, amount):          # set attributes at object creation time
        self.claim_id = claim_id                   # store the claim reference number
        self.amount = amount                       # store the dollar amount billed
        self.is_approved = False                   # all new claims start as not approved

    def approve(self):                             # method to approve this specific claim
        self.is_approved = True                    # update the is_approved attribute

    def status_label(self):                        # method to return a plain-English status word
        if self.is_approved:                       # check this object's own is_approved value
            return "Approved"                      # return this if the claim was approved
        return "Pending"                           # return this if the claim is still waiting

claim = Claim("C-4821", 350)                       # create one claim; is_approved starts False
print(claim.status_label())                        # read status before approving
claim.approve()                                    # call approve() -- updates the attribute
print(claim.status_label())                        # read again -- attribute has changed
Terminal $ python claim_status.py
Pending
Approved

A method that changes an attribute and a method that reads it work as a natural pair — this "change then read" pattern appears in almost every real Python script that manages data.

03

Multiple Provider Records

Beginner Providers One Class, Many Objects List + Loop

What This Does

This script creates three separate objects from the same class and stores them in a list. A loop then calls the same method on every object — showing how one blueprint can produce as many independent records as you need.

provider_records.py 
class Provider:                                    # one blueprint for any healthcare provider
    def __init__(self, provider_id, specialty):    # set each provider's data at creation time
        self.provider_id = provider_id             # store the unique provider ID
        self.specialty = specialty                 # store the medical specialty

    def describe(self):                            # method to print one line about this provider
        print(self.provider_id + " -- " + self.specialty)  # reads this object's own attributes

providers = [                                      # build a list of three separate objects
    Provider("P-01", "Cardiology"),               # first object -- has its own data
    Provider("P-02", "Radiology"),                # second object -- completely independent
    Provider("P-03", "Oncology"),                 # third object -- same blueprint, different data
]

for provider in providers:                         # loop through every object in the list
    provider.describe()                            # call describe() on this specific object
Terminal $ python provider_records.py
P-01 -- Cardiology
P-02 -- Radiology
P-03 -- Oncology

You only write the class once — add more items to the list and the loop handles every new object automatically, with no extra code needed.
