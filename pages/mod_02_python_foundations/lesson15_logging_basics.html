Overview

A high-level summary of the topic

Py

Logging is Python’s built-in system for recording what your program does — automatically, while it runs.

Think of Python logging like a security guard’s logbook at a building entrance — every significant event gets written down with the time it happened and a note on how serious it was. A logging setup is that logbook, and Python’s logging module is the system that writes in it for you.

Log severity levels

The urgency stamp — INFO note or CRITICAL alarm

Filters which entries are important enough to write down.

The Logger

The guard — watches one wing of the building

Passes each approved entry to the right handler for filing.

Log handlers

The destination — logbook, email, or both

Files each entry into the logbook or sends an alert.

Log formatters

The template — timestamp, level, and message

Sets the exact layout every entry in the logbook must follow.

If you already track issues in an Excel sheet with a date column and a severity column, you’ll recognize the pattern — logging does the same job automatically, so your scripts document themselves as they run.

Key Concepts

The logging module, severity levels, basicConfig(), and log handlers.

The logging Module

Python’s built-in recording system

Built-in

The logging module is Python’s built-in tool for recording events in your script. You add import logging at the top once — no installation needed.

print() vs logging

print() Terminal only
→
logging File + terminal
→
Audit trail
Python 
import logging                               # built-in — no pip install needed
logging.basicConfig(level=logging.DEBUG)     # configure once, before any log call
logging.info("Script started")               # record a routine event
logging.warning("Config file missing")       # record a concern worth noting
logging.error("File not found")              # record a failure that needs attention

Call basicConfig() before your first log call. If you skip it, Python uses silent defaults and your DEBUG and INFO messages will never appear.

Severity Levels

Five levels, least to most urgent

5 Levels

A severity level labels how urgent a message is. You set a minimum level when configuring logging — only messages at that level or above get recorded.

Severity Levels Reference

Level When to use Rank
DEBUG Detailed steps — for diagnosing problems 10
INFO Normal progress — everything is working 20
WARNING Unexpected — script still runs but watch out 30
ERROR A failure occurred — something needs fixing 40
CRITICAL Serious failure — the program may not continue 50
Python 
logging.debug("Checking row count: %d", len(df))    # dev only — very detailed
logging.info("Dataset loaded successfully")           # normal progress
logging.warning("Missing values in column 'price'")  # something to watch
logging.error("Could not connect to database")        # a real failure
logging.critical("Out of memory — stopping run")      # the program cannot continue

WARNING is the default minimum level. If you don’t call basicConfig() first, your DEBUG and INFO messages are silently discarded.

basicConfig()

One function call to set up logging

Setup

basicConfig() is the function that configures your logging. You pass it the minimum level, the file to write to, and the message layout — all in a single call at the top of your script.

Key Parameters

level= logging.DEBUG shows all messages ✓
filename= "app.log" writes messages to a file ✓
format= add %(asctime)s for timestamps ✓
filemode= "a" keeps old entries ✓  "w" overwrites ✗
Python 
logging.basicConfig(
    level=logging.DEBUG,                         # record all messages
    filename="pipeline.log",                     # write to this file
    filemode="a",                                # append — keeps previous entries
    format="%(asctime)s %(levelname)s: %(message)s"  # timestamp + level + text
)
logging.info("Pipeline started")                 # test that the setup works

Call basicConfig() once, at the very top of your script. Calling it a second time has no effect — Python ignores repeated calls because logging is already configured.

Log Handlers

Control where messages are sent

Output

A handler decides where your log messages go. Python gives you two main options: the terminal (StreamHandler) and a file on disk (FileHandler). You can attach both to the same logger.

StreamHandler vs FileHandler

StreamHandler FileHandler
Destination Terminal screen File on disk
Survives close ✗ Gone on close ✓ Stays on disk
Good for Live debugging Production audits
Python 
logger = logging.getLogger(__name__)          # named logger for this file
logger.setLevel(logging.DEBUG)                # accept all messages
stream_h = logging.StreamHandler()            # sends messages to the terminal
file_h   = logging.FileHandler("app.log")     # writes messages to a file

logger.addHandler(stream_h)                   # attach the terminal handler
logger.addHandler(file_h)                     # attach the file handler

logger.info("Handlers attached")              # appears in terminal AND in app.log

Use logging.getLogger(__name__) instead of the root logger in any file others might import. It keeps each module’s logs separate and easier to trace.

Code Examples

Hands-on code snippets to explore the concepts

01

Log Your First Message

Beginner Script basicConfig()

What This Does

This script sets up Python’s logging module with one line and records a message. You’ll see how a logged message looks different from a plain print() call.

first_log.py 
import logging                              # load the built-in logging module

logging.basicConfig(level=logging.DEBUG)   # accept all messages, including DEBUG

logging.debug("Checking setup")            # detailed trace — for diagnosing problems
logging.info("Script started")             # routine progress message
logging.warning("Config file not found")   # something unexpected — still running
Terminal $ python first_log.py
DEBUG:root:Checking setup
INFO:root:Script started
WARNING:root:Config file not found

Notice the format: Python automatically adds the level name and logger name before your message — you get that extra context for free without writing any extra code.

02

Logging Levels in Action

Beginner Data Pipeline Level filter

What This Does

This script records messages at all five severity levels, then raises the minimum level to WARNING so lower-priority messages disappear without any other code changes.

level_demo.py 
import logging

logging.basicConfig(level=logging.WARNING)    # only WARNING and above appear

logging.debug("Loading config")               # hidden — below the threshold
logging.info("Pipeline started")              # hidden — below the threshold
logging.warning("Missing values detected")    # shown — at the threshold
logging.error("Database connection failed")   # shown — above the threshold
logging.critical("Out of memory — stopping") # shown — most urgent level
Terminal $ python level_demo.py
WARNING:root:Missing values detected
ERROR:root:Database connection failed
CRITICAL:root:Out of memory — stopping

Change logging.WARNING to logging.DEBUG and run again — all five messages appear instantly, with no other edits needed.

03

Write Logs to a File

Beginner Sales Report FileHandler Timestamps

What This Does

This script saves every log message to a file on disk, with a timestamp on each line. Each run adds to the file rather than overwriting it, so you build up a full history of every time the script ran.

sales_log.py 
import logging

logging.basicConfig(
    filename="sales_report.log",                      # write to this file
    filemode="a",                                     # append — keeps previous entries
    level=logging.INFO,                               # record INFO and above
    format="%(asctime)s %(levelname)s: %(message)s"   # timestamp + level + text
)
logging.info("Sales report started")                  # first entry in the log file
logging.warning("Missing data in Q3 column")          # flagged for follow-up
Terminal $ python sales_log.py
# output written to sales_report.log:
2026-03-30 09:14:02 INFO: Sales report started
2026-03-30 09:14:02 WARNING: Missing data in Q3 column

Use filemode="a" (append) rather than "w" (write) — append mode keeps every previous run, so you never accidentally erase your audit trail.

The DEBUG:root: prefix tells you the severity level and the logger name. Changing level=logging.INFO hides all debug() calls in a single edit — you do not need to delete or comment them out.

02

Filter by Level

Beginner Reports logging.WARNING Level threshold

Your Task

You are building a weekly report script that logged debug messages while you were testing. Now that it runs in production, you want to suppress those noisy lines. Set basicConfig() to WARNING level, then add one log call each at debug, info, and warning. Run the script and notice that only the warning line appears in the console.

report_filter.py 
import logging                               # import the logging module
logging.basicConfig(level=logging.WARNING)   # only show WARNING and above
logging.debug("Debug details")               # blocked — below the threshold
logging.info("Report started")               # blocked — below the threshold
logging.warning("Missing column")            # shown — at or above WARNING
Terminal $ python report_filter.py
WARNING:root:Missing column

Setting the level to WARNING acts like a noise filter — it silently ignores every message below that threshold. You can write as many logging.debug() calls as you like during development, then suppress them all with one setting change when you deploy.

03

Log to a File

Intermediate Analysis filename= filemode="a"

Your Task

You are analysing monthly sales data and want to keep a record of what your script did, even after the console closes. Configure the logging module to write messages to a file called analysis.log. Set filemode to "a" so the file grows with each run instead of being overwritten. Add one info and one warning message, then open the file to verify your entries are there.

file_logger.py 
import logging                                # import the logging module
logging.basicConfig(                          # configure file-based logging
    filename="analysis.log",                  # write messages to this file
    filemode="a",                             # append — keep old entries
    level=logging.INFO,                       # record INFO and above
)
logging.info("Analysis started")              # written to analysis.log
logging.warning("Input has blank rows")       # also written to analysis.log
Terminal $ python file_logger.py
INFO:root:Analysis started
WARNING:root:Input has blank rows

Using filemode="a" (append) means each run adds to the log rather than starting fresh. This gives you a complete history across runs — if something breaks on Friday, you can read back through the file to see exactly what every previous run recorded.

Common Mistakes

Frequent errors and how to avoid them

Using print() Instead of the Logging Module

Print output disappears when the terminal closes — a log entry persists in a file and can be reviewed later.

Pitfall

Using print() works during quick testing but has no severity level, no timestamp, and no way to redirect to a file without extra code. Replace it with logging.info() (or the appropriate level) so that every message carries structured metadata from the start.

Wrong — disposable console output 

print("Loading data")
print("Cleaning rows")
print("Export complete")

Correct — structured log entries 

logging.info("Loading data")    # severity level + timestamp
logging.info("Cleaning rows")   # can redirect to a file later
logging.info("Export complete") # searchable, filterable record

Think of print() as telling someone something out loud — once spoken, it is gone. logging.info() is the written note you can come back to days later when something in production breaks.

Calling logging Functions Without Configuring the Module First

Without basicConfig(), the default level is WARNING — DEBUG and INFO calls produce no output and no error.

Pitfall

The logging module's default level is WARNING, so calling logging.debug() or logging.info() without first running basicConfig() produces no output and no error message — your log calls are simply swallowed silently. Add logging.basicConfig(level=logging.DEBUG) as the very first logging statement in your script.

Wrong — no output appears 

import logging
logging.info("Script started")  # silent — level not configured
logging.debug("Reading file")   # silent — filtered out by default

Correct — configure before calling 

import logging
logging.basicConfig(level=logging.DEBUG)  # configure first
logging.info("Script started")            # now visible
logging.debug("Reading file")            # now visible

basicConfig() is like plugging in a speaker — without it, your log messages play into nothing and you get no error to warn you. One call at the very top of your script turns the whole system on.

Calling basicConfig() a Second Time Has No Effect

Python's logging module ignores every basicConfig() call after the first — settings from the first call stay in force for the entire run.

Pitfall

Once any handler is attached to the root logger — even by the first basicConfig() call — every subsequent basicConfig() is silently ignored. Writing a second call to change the level or add a filename does not override the first. Put all your logging configuration in a single basicConfig() call at the top of your script.

Wrong — second call ignored 

logging.basicConfig(level=logging.WARNING)  # first call wins
logging.basicConfig(level=logging.DEBUG)    # ignored silently
logging.debug("Detail")                     # still blocked

Correct — single call at the top 

logging.basicConfig(level=logging.DEBUG)  # one call, set correctly
logging.debug("Detail")                   # now visible

Treat basicConfig() like signing a contract — only the first signature counts. Collect every option you need (level, filename, format) into one call and place it at the very top of your script, before any log messages.

Real-World Use

How logging is used across real-world workflows

Imagine you are part of a data team that runs automated scripts overnight. Something breaks at 2 am — and by morning, all you have is a blank screen. The logging module lets every script write a permanent, timestamped record of what it did, so you can trace exactly where things went wrong without being there to watch.

An Overnight Data Pipeline Run

What the log file recorded while you were asleep

Below is a real analysis.log file produced by a Python script using the logging module.

analysis.log
2024-01-15 02:00:01 INFO Pipeline started — loading raw_sales_jan.csv
2024-01-15 02:00:03 INFO 1,247 rows loaded successfully
2024-01-15 02:00:04 WARNING Row 892: price is negative (-12.50), row skipped
2024-01-15 02:00:04 WARNING Row 1103: missing product_id, row skipped
2024-01-15 02:00:07 INFO Cleaning complete — 1,245 rows retained, 2 skipped
2024-01-15 02:00:09 ERROR Database connection failed: timeout after 30s
2024-01-15 02:00:09 ERROR Pipeline stopped — see above for details

What Each Level Told the Team

INFO — progress checkpoints that confirm everything is working

logging.info()

The two INFO lines showed that the file loaded correctly and cleaning finished. Without them, a silent success looks identical to a silent failure — you would not know if the script even started.

WARNING — bad data rows caught and named before they reach the report

logging.warning()

The two WARNING lines identified the exact row numbers with bad data. The analyst opened the log, went straight to row 892 and row 1103 in the source CSV, and fixed them in 10 minutes instead of scanning 1,247 rows manually.

ERROR — the exact failure point without running the script again

logging.error()

The ERROR lines showed that the cleaning steps all passed — the problem was the database connection at the very end. That single detail cut the investigation from hours to minutes because the team knew immediately where not to look.
