Overview

A high-level summary of the topic

Py

When Python encounters a problem in a program, it displays an error message — a structured report that tells you exactly what went wrong and where.

Think of a Python error message like feedback from a teacher who just marked your essay with a red pen: they circle the problem spot (line number), write the category of mistake in the margin (error type), and leave a brief note explaining what went wrong (description).

Error Type

The red annotation — names the category of mistake

Python labels every error with a type such as SyntaxError, NameError, or TypeError so you know immediately what category of problem to investigate.

Line Number

The circled line — shows exactly where to look

Python reports the line number where it detected the problem. This is your starting point — go straight to that line in your code rather than reading from the top.

Error Description

The written note — tells you what specifically went wrong

The description after the error type explains the specific problem in plain language. Reading it carefully is often enough to understand exactly what needs to be corrected.

Debugging

The revision process — correct and resubmit

Debugging is the process of reading the error, locating the problem, fixing the code, and running the program again. It is a normal, expected part of writing any program.

Error messages are your most reliable tool for debugging. Experienced developers read them first, every time — they rarely guess or try random fixes.

Key Concepts

SyntaxError, NameError, and TypeError — what each one means and how to fix it.

SyntaxError

Python found a code-structure problem

SyntaxError

A SyntaxError means Python found code that breaks the grammar rules of the language — it could not run a single line, because it could not understand what you wrote. Think of it like a sentence with a missing full stop: the reader stops before they can work out what you meant.

Common Causes

Missing closing ) print("hello" ✗ → print("hello")
Missing colon : after if if x > 0 ✗ → if x > 0:
Mismatched quotes "hello' ✗ → "hello"
Python 
# Python checks the whole file for syntax before running anything
# print("Hello"   # SyntaxError: '(' was never closed — Python stops here

# Fix: close every bracket you open
print("Hello")    # now Python can run this — prints: Hello
print("World")    # this line runs too, because the syntax is correct

Your script produces no output with a SyntaxError — Python refuses to run any line at all until the structure is valid. Fix the highlighted line first, then run your script again.

NameError

Python found a name it does not recognize

NameError

A NameError means Python found a word it has never seen before — you used a variable name before you created it, or you made a spelling mistake in the name. Python reads top to bottom, so any name must exist before you use it.

Why Names Fail

Used before assigned print(total) ✗ before total = 0
Typo in variable name totel ✗ → total
Wrong capitalization Total ✗ ≠ total
Python 
total = 0                   # define the variable before using it

for price in [10, 20, 30]:
    total += price          # Python knows what total is — it was defined above

print(total)                # prints: 60 — no NameError because total exists

The error message tells you the exact name Python could not find — copy that name and search your script for it. Compare spelling and capitalization letter by letter.

TypeError

Two incompatible data types were mixed

TypeError

A TypeError means you asked Python to do something with two values whose types do not match — the most common example is trying to join a piece of text and a number with +, which Python cannot do without help.

str vs int

str int
Example "42" 42
Has quotes ✓ Yes ✗ No
Join with + strings only numbers only
Python 
age = 25

# TypeError: Python cannot join a str and an int with +
# print("Age: " + age)          # TypeError — mixed types

# Fix option 1: convert the int to str first
print("Age: " + str(age))       # prints: Age: 25

# Fix option 2: use a comma — Python handles the space automatically
print("Age:", age)               # prints: Age: 25

The error message names both types that clashed — for example, can only concatenate str (not 'int') to str tells you exactly where to look. Wrap the number in str() to fix it.

Decision Flow

A visual walkthrough of the logic and process

When you run a Python script, Python reads your code from top to bottom, one line at a time. When it hits a problem it stops immediately and prints a message that tells you exactly what went wrong and where. Your job is to read that message — not guess.

What Happens When Python Finds an Error

1

Python runs your code top to bottom

Each line runs in the order you wrote it. Lines 1, 2, 3 … in sequence — no lines are skipped.

2

It hits a problem — and stops

Nothing after the broken line runs. Python halts at that exact point and prints an error message in your terminal.

3

You read the message and fix the line

The message always contains three clues: the error type, the line number, and a description. Together they point you straight to the fix.

Here is a real example. The script below runs two lines successfully, then fails on line 7. Notice how the error message gives you all three clues.

debug_me.py 
name = "Alice"              # line 1: defines a text variable
age = 30                    # line 2: defines a number variable

print("Name:", name)        # line 4: runs fine — prints: Name: Alice
print("Age:", age)          # line 5: runs fine — prints: Age: 30

print("Score: " + 95)       # line 7: TypeError — can't join str and int
Terminal $ python debug_me.py
Name: Alice
Age: 30
Traceback (most recent call last):
  File "debug_me.py", line 7, in <module>
    print("Score: " + 95)
TypeError: can only concatenate str (not "int") to str

The Three Clues in That Error Message

1

Error Type — TypeError

The category of problem. This tells you that two incompatible data types were used together — that is your starting point for the fix.

2

Line Number — line 7

The exact line where Python stopped. Go directly to that line — you do not need to read the whole script to find the problem.

3

Description — can only concatenate str (not "int") to str

A plain-English explanation of what went wrong. This one tells you the fix: convert 95 to a string first, or use a comma instead of +.

Reading the error message is always the first step — not scrolling through your whole script guessing. With the type, the line number, and the description, almost every fix takes under a minute.

Code Examples

Hands-on code snippets to explore the concepts

01

Fix a SyntaxError

Beginner Budgets if / else Missing colon

What This Does

This script compares a budget to an amount spent using an if statement. The broken line is shown as a comment so you can see exactly what was missing — then the fixed line runs successfully.

budget_check.py 
budget = 1000               # the monthly budget amount
spent = 750                 # total amount spent so far

# if spent > budget         # ← BROKEN: missing colon — causes SyntaxError
if spent > budget:          # ← FIXED: colon added — Python reads the block below
    print("Over budget!")   # runs only when spent exceeds budget
else:
    print("On track")       # runs when spending is within budget

remaining = budget - spent  # calculate how much money is left
print("Remaining:", remaining)  # prints the leftover amount
Terminal $ python budget_check.py
On track
Remaining: 250

Python requires a colon at the end of every if line — it is the signal that an indented block follows, and SyntaxError is almost always the result when it is missing.

02

Fix a NameError

Beginner Products Variables Typo check

What This Does

This script stores a product name and price as variables, then calculates a sale price. The broken line has a typo in the variable name — one missing letter causes a NameError because Python cannot find a variable by that name.

product_label.py 
product_name = "Laptop"      # the name of this product
product_price = 999           # the original price in whole dollars

# print(prodct_price)         # ← BROKEN: typo missing 'u' — NameError
print(product_price)          # ← FIXED: correct spelling — Python finds the value

discount = 100                # discount amount in dollars
final_price = product_price - discount  # subtract to get the sale price
print("Final price:", final_price)     # prints the discounted price
Terminal $ python product_label.py
999
Final price: 899

The error message always prints the exact name Python could not find — copy it, search your script for it, and compare the spelling letter by letter, including capitalization.

03

Fix a TypeError

Beginner Reports print() Type mismatch

What This Does

This script builds a short report summary from a title, a page count, and an item count. The broken line tries to join a piece of text and a number with + — which causes a TypeError — then shows the two-character fix.

report_summary.py 
report_title = "Monthly Summary"   # the title of the report
page_count = 12                     # number of pages in the report
item_count = 47                     # number of items recorded

# print("Pages: " + page_count)    # ← BROKEN: str + int causes TypeError
print("Pages:", page_count)        # ← FIXED: comma joins str and int cleanly

total_entries = item_count * 2      # each item appears twice in the report
print("Total entries:", total_entries)  # prints the total entry count
print(report_title, "— complete")       # prints the title with a status note
Terminal $ python report_summary.py
Pages: 12
Total entries: 94
Monthly Summary — complete

Replacing + with a comma in print() is the fastest TypeError fix — Python adds the space and converts the number automatically, so you never need str() there.

SQL / Excel Comparison

How this topic compares across Python, SQL, and Excel

If you already use SQL or Excel, you have seen error messages before — a query that fails with a syntax error, a cell showing #VALUE!. Python works the same way: when something goes wrong, it stops and prints a message that names the error type and tells you exactly which line to check.

Python
SQL
Excel
syntax / grammar error
Python SyntaxError

Python raises this when your code breaks a grammar rule — the most common cause is a missing colon after an if or for.

SQL syntax error

SQL returns this when a keyword is missing, misspelled, or placed in the wrong order in your query.

Excel formula error

Excel shows an alert when a formula cannot be parsed — usually a typo in a function name or a misplaced bracket.

missing name or column
Python NameError

Python raises this when you use a variable that has not yet been assigned a value in your script.

SQL column not found

SQL returns this when a column name in your query does not exist in the table you referenced.

Excel #NAME?

Excel shows #NAME? when a formula references a named range or function name that it cannot find.

type or value mismatch
Python TypeError

Python raises this when an operator is applied to incompatible types — such as joining text and a number with +.

SQL data type mismatch

SQL returns this when you compare or combine columns of incompatible types — such as text alongside a number.

Excel #VALUE!

Excel shows #VALUE! when a formula performs a calculation on a cell that contains the wrong type of data.

Python, SQL, and Excel all signal the same three error categories — grammar, missing name, and type mismatch. Once you can name which one you are looking at, you already know where to start the fix — and that skill transfers directly between all three tools.

Guided exercises to reinforce your learning

01

Spot a SyntaxError

Beginner Members SyntaxError

Your Task

A member eligibility script has a broken if statement that causes Python to stop with a SyntaxError. Read the error message Python prints, find the line number it reports, and fix the one-character mistake on that line. Then run the script again to confirm it prints the eligibility status correctly.

eligibility_check.py 
member_name = "Rosa Chen"    # name of the member being checked
plan_type = "Gold HMO"       # the member's health plan
deductible_met = True         # whether the annual deductible has been paid

if deductible_met             # ← fix: a colon is missing at the end of this line
    print(member_name, "is eligible for full cover under", plan_type)
else:
    print(member_name, "must pay the deductible before claims are covered")
Terminal $ python eligibility_check.py
Rosa Chen is eligible for full cover under Gold HMO

Every if, else, and for line in Python must end with a colon. A missing colon is one of the most common causes of SyntaxError in real eligibility and claims scripts.

02

Fix a NameError

Beginner Claims NameError

Your Task

A claim copay calculation script raises a NameError because one variable is used before it is assigned. Read the error message to find the name of the missing variable, then add one line to define it before it is used. The script should print the member's copay amount.

copay_calc.py 
claim_id = "CLM-8812"         # unique identifier for this claim
copay_rate = 0.20              # member pays 20% of the total billed amount

# ← fix: add  total_billed = 350  here before the next line uses it
member_owes = total_billed * copay_rate   # calculate the member's share
print("Claim", claim_id, "— member copay:", member_owes)  # display the result
Terminal $ python copay_calc.py
Claim CLM-8812 — member copay: 70.0

In a real claims system, total_billed would be read from a database record rather than typed in by hand. A NameError here often means a variable was renamed in one place but not updated everywhere it is used.

03

Fix a TypeError

Beginner Providers TypeError

Your Task

A provider payment summary script raises a TypeError because it tries to join a text string and a number with +. Fix the print() line so the output reads Total paid to provider: 1450. The accordion shows two correct approaches — try both.

Approach 1 — comma (recommended for beginners)

provider_payment_a.py 
provider_name = "Riverside Clinic"  # name of the provider being paid
claim_1 = 800                        # first approved claim amount
claim_2 = 350                        # second approved claim amount
claim_3 = 300                        # third approved claim amount
total_paid = claim_1 + claim_2 + claim_3  # sum all three claims

print("Total paid to provider:", total_paid)  # comma — no TypeError
Terminal $ python provider_payment_a.py
Total paid to provider: 1450

Approach 2 — str() conversion

provider_payment_b.py 
print("Total paid to provider: " + str(total_paid))  # str() converts int to text
Terminal $ python provider_payment_b.py
Total paid to provider: 1450

Provider payment reports often combine text labels with numeric totals. Using a comma in print() is the simplest fix; wrapping with str() gives you more control over spacing when you need the number embedded inside a sentence.

Common Mistakes

Frequent errors and how to avoid them

Skipping the Line Number in the Traceback

Python prints the exact line where it stopped — scanning the script from line 1 wastes time and usually leads to the wrong fix.

Pitfall

Every traceback ends with the file name, line number, and error type. Go to that line number first — not to the top of the script.

Terminal — example traceback 
Traceback (most recent call last):
  File "claim_report.py", line 6, in <module>   # <-- open the file at line 6
    member_owes = total_billed * copay_rate
NameError: name 'total_billed' is not defined    # <-- this tells you the category

Treat the reported line number like a GPS pin — open the file there and check that line and the one directly above before touching anything else.

Editing the Reported Line When the Real Mistake Is One Line Above

Python reports where it stopped, not always where the mistake is — a missing closing bracket or colon on one line causes the error to surface on the next.

Pitfall

A missing closing parenthesis on line 4 may cause a SyntaxError on line 5, because Python only detects the broken structure when it hits the next statement. If the reported line looks correct, check the line immediately above it.

Wrong — editing line 5 (not the problem) 

claim_id = "CLM-7701"
copay_rate = 0.20
claim_amount = 350
print("Claim:", claim_id    # missing ) — real mistake here
print("Copay:", copay_rate) # SyntaxError reported HERE

Correct — closing bracket added on line 4 

claim_id = "CLM-7701"
copay_rate = 0.20
claim_amount = 350
print("Claim:", claim_id)   # ) added — line 4 fixed
print("Copay:", copay_rate) # now runs without error

When a line looks fine and still raises a SyntaxError, check the line above it for a missing closing parenthesis, quote, or colon before making any changes.

Making Random Changes Instead of Reading the Error Type

Commenting out code or rewriting sections at random rarely fixes the bug and often creates new ones — the error type tells you exactly which category of fix to apply.

Pitfall

The error type — SyntaxError, NameError, or TypeError — names the exact problem category. Read it first, then apply only the change that addresses that category.

Wrong — commenting out code at random 

member_id = "M-4420"
claim_amount = 350
# copay_rate = 0.20      # commented out at random
member_owes = claim_amount * copay_rate  # still NameError
print("Owes:", member_owes)

Correct — define the variable NameError named 

member_id = "M-4420"
claim_amount = 350
copay_rate = 0.20           # NameError named this — define it
member_owes = claim_amount * copay_rate  # now works
print("Owes:", member_owes) # prints: Owes: 70.0

Read the error type the same way you would check a patient's chart before prescribing treatment — it tells you what is wrong before you decide what to change.

Real-World Use

Practical applications in real-world workflows

Reading error messages is a daily skill in real data and analytics work. Every professional Python script eventually fails — knowing how to read what Python tells you is what separates fast debugging from guesswork.

Pipeline Failure

A scheduled data pipeline crashes overnight. In the morning you open the log and find a Python traceback. Instead of reviewing every line of the script, you read the last error — it names the file, the line number, and the problem in three lines of text.

Dashboard TypeError

A reporting dashboard stops updating with a TypeError. The error message names the exact variable where a text value was used where a number was expected — you fix it in under a minute without touching anything else.

Script NameError

A script to process customer data fails with a NameError. The message names the undefined variable and the line number. You search the file for that variable name, find the missing assignment, and the script runs cleanly.

Without vs. With error message skills

Situation Without reading the error Reading the error first
Pipeline crashes Browse every line looking for the problem Go directly to the line Python reported
Dashboard TypeError Guess which column or variable changed The message names the exact variable and both types involved
Script NameError Ask a colleague to find the missing assignment Search for the named variable — add the assignment in seconds
