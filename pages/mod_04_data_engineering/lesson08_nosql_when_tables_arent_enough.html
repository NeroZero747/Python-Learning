Overview

A high-level summary of the topic

Py

SQL databases are exceptional for structured, consistent data. But not all data is structured, consistent, or relational. When you’re storing user activity events, product catalogs with irregular attributes, or sensor readings by the billions — SQL tables become an uncomfortable fit.

Think of the difference between SQL and NoSQL like two kinds of office filing. An SQL database is like a rigidly organized file cabinet where every document must fit the section’s template. A NoSQL database is like a flexible storage room where documents can have different structures and new fields can be added without redesigning the cabinet.

Document DB

One envelope per record — holds everything, any shape

A document database stores each record as a self-contained JSON (or BSON) document, meaning one product can have a specs field while another has an isbn field — no column migration required. You use document databases like MongoDB or Firestore whenever your data has variable or evolving attributes, such as a product catalog, user profiles, or a content management system.

Key-Value DB

A wall of numbered lockers — instant access by number

A key-value database is the simplest possible store: you hand it a key (like "user:42") and it returns the associated value in sub-millisecond time, with no parsing of row structure or execution of a query plan. You use key-value stores like Redis or DynamoDB for session data, real-time leaderboards, rate-limiting counters, and any lookup where the tiny latency of a traditional database is unacceptable.

Column-Family DB

A notebook with infinitely expanding rows

A column-family database (like Cassandra or HBase) groups related columns together and allows each row to have a completely different set of columns, making it exceptionally efficient for append-heavy workloads where you rarely update existing records. You use column-family databases for IoT sensor readings, click-stream event logs, and time-series metrics — situations where you write billions of rows but only ever query a narrow slice.

Graph DB

A pinboard with thread connecting related items

A graph database stores data as nodes (entities like users or products) and edges (named relationships like FOLLOWS or PURCHASED), making multi-hop relationship traversal orders of magnitude faster than joining tables in SQL. You use graph databases like Neo4j or Amazon Neptune for social network friend-of-friend queries, fraud detection rings, and product recommendation engines where the connections matter as much as the data itself.

NoSQL doesn’t mean “no structure” — it means “not only SQL.” Most production systems use SQL and NoSQL together.

Key Takeaways

The most important ideas to remember

NoSQL Schema Flexibility Has a Hidden Cost

Because document databases let every record have a different shape, your application code becomes responsible for enforcing consistency — and if you forget to validate incoming data, a missing field silently produces None rather than an error. The gotcha is that schema-less does not mean schema-free: you still need to document and validate your data contract, you just do it in code instead of the database.

Schema-less Validation Silent failure

Scaling Out Beats Scaling Up for Petabyte Data

When a SQL server runs out of capacity, your only option is a bigger, more expensive machine — you cannot simply split a relational table across two servers without breaking its foreign-key guarantees. NoSQL databases are designed from day one to shard data across dozens or hundreds of commodity nodes, so adding capacity means adding cheap servers rather than upgrading a single expensive one.

Sharding Horizontal scale Distributed

Access Pattern, Not Data Volume, Decides the Database

Picking NoSQL purely because you have “a lot of data” is the single most common mistake: if your queries always join two entities together, a document store will be slower and harder to maintain than a well-indexed SQL table. The correct question is “how do I read this data most often?” — if the answer is “by key alone”, use key-value; “as a whole record”, use document; “by traversing connections”, use graph.

Access pattern Query shape Data modeling

MongoDB CRUD Looks Like SQL — Until the Schema Changes

Inserting and querying documents with pymongo feels familiar — you use dictionaries instead of rows, and filter syntax like {"status": "active"} instead of WHERE status = ’active’. The real payoff comes when a new product type needs a new field: you simply insert it in the next document without running an ALTER TABLE or migrating millions of existing rows.

pymongo insert_many find_one

Key Concepts

Categories, decision rules, and MongoDB CRUD operations.

4 Categories

Document, Key-Value, Column-Family, Graph

Reference

NoSQL is an umbrella term for four distinct database families. Each family is optimized for a different access pattern that relational tables handle poorly.

NoSQL Category Reference

Category Best For Tools
Document Variable-schema records MongoDB, Firestore
Key-Value Caching, sub-ms lookups Redis, DynamoDB
Column-Family Append-heavy writes, IoT Cassandra, HBase
Graph Relationship traversal Neo4j, Neptune
Python 
from pymongo import MongoClient          # Document DB driver
client = MongoClient("mongodb://localhost:27017/")
db = client["ecommerce"]                 # select database
col = db["products"]                     # select collection (no fixed schema)

import redis                             # Key-Value DB driver
r = redis.Redis(host="localhost", port=6379)
r.setex("cart:42", 900, "item_data")    # key, TTL in seconds, value

Before picking a NoSQL category, write down your most common query. The query shape — not data volume — tells you which family to use. Choosing the wrong one forces you to do joins in Python code instead.

When SQL Wins

Decision criteria

4 Signals

SQL databases are the right choice when your data is uniform and relationships matter. Reach for SQL whenever your queries join multiple entities or require strict guarantees that every write either completes fully or not at all.

Use SQL When…

Every record has the same columns use SQL ✓
Queries JOIN two or more tables use SQL ✓
Partial writes must roll back use SQL ✓
GROUP BY or window functions use SQL ✓

# SQL checklist — answer YES to any of these → use PostgreSQL / MySQL
# 1. Every employee record has the exact same 10 fields
# 2. Payroll queries JOIN employees JOIN departments
# 3. Payments must commit fully or roll back (ACID transaction)
# 4. Monthly totals use SUM + GROUP BY department

import psycopg2                                    # PostgreSQL driver

SQL has survived 50 years because it solves a real, common problem very well. NoSQL does not replace SQL — it extends your toolkit for cases where the relational model creates friction rather than safety.

When NoSQL Wins

Decision criteria

4 Signals

NoSQL databases shine when your data is irregular, high-volume, or retrieved by a single key. Reach for NoSQL when the relational model creates more friction than safety.

Use NoSQL When…

Records have variable fields Document DB ✓
Lookup is always by a single key Key-Value DB ✓
Billions of append-only writes Column-Family ✓
Queries traverse connections Graph DB ✓
Python 
# NoSQL checklist — answer YES to any of these → use NoSQL
# 1. Products have variable fields (laptops have RAM, books have ISBN)
# 2. Session lookup is always "get session by ID" — no joins needed
# 3. Sensor writes arrive 1 million rows/minute — SQL cannot keep up
# 4. "Friends of friends" query traverses 5 relationship hops

# Each signal maps to a different category:
# Variable fields     → MongoDB     (Document)
# Single-key lookup   → Redis       (Key-Value)
# Append-heavy writes → Cassandra   (Column-Family)
# Relationship hops   → Neo4j       (Graph)

Picking NoSQL because you have “a lot of data” is the single most common mistake. Data volume alone does not decide the database — your read pattern does. A 10 GB SQL table with good indexes outperforms the wrong NoSQL choice at any scale.

MongoDB CRUD

Python mechanics with pymongo

pymongo

pymongo is Python’s official MongoDB driver. The four CRUD operations map directly to method calls on a collection object, and every query filter is a plain Python dictionary.

CRUD Method Reference

Op Method What It Does
C insert_one / insert_many Add one or many documents
R find_one / find Return first match or all matches
U update_one / update_many Change fields using $set
D delete_one / delete_many Remove matching documents
Python 
col = db["products"]                              # collection handle

# CREATE — add a new document
col.insert_one({"sku": "A1", "price": 9.99, "stock": 100})

# READ — find documents matching a filter dictionary
col.find_one({"sku": "A1"})                       # returns one dict
list(col.find({"price": {"$lt": 20}}))            # returns list of dicts

# UPDATE — always wrap new values in $set
col.update_one({"sku": "A1"}, {"$set": {"price": 12.99}})   # one document
col.update_many({"stock": 0}, {"$set": {"status": "sold_out"}})  # many

# DELETE — remove matching documents
col.delete_one({"sku": "A1"})                     # first match only

Never omit $set in an update call. Writing update_one(filter, {"price": 12}) without $set replaces the entire document with only {"price": 12}, silently deleting all other fields.

SQL vs MongoDB Comparison

Side-by-side mapping of concepts you already know

If you already know SQL, every concept has a MongoDB analog. The biggest shift is that MongoDB stores data as Python dictionaries and replaces SQL keywords with Python method calls.

SQL → MongoDB
Data Model
Table vs Collection
SQL TABLE users

A fixed-schema container where every row shares the same typed columns.

MongoDB db.users

A schema-free collection. Documents in the same collection can have completely different fields and still coexist.

Row vs Document
SQL ROW

One record; every field must match the table's column definitions.

MongoDB {"name": "Alice", "age": 30}

A Python dictionary stored as JSON. Each document can have unique keys and even nested sub-documents.

Column vs Field
SQL col_name TYPE NOT NULL

Declared once at table creation; changing the schema later requires ALTER TABLE.

MongoDB field_name

Added per-document with no schema migration required. You can start using a new field in any new document immediately.

CRUD Operations
Create
SQL INSERT INTO users VALUES (...)

Values must match the column order and data types defined at table creation.

MongoDB col.insert_one({"name": "Alice"})

Pass any Python dictionary. MongoDB automatically adds a unique _id field if you do not include one.

Read
SQL SELECT * FROM users WHERE age > 25

Returns a result set of matching rows; column projections use explicit SELECT col syntax.

MongoDB col.find({"age": {"$gt": 25}})

The filter is a dictionary. Operators like $gt, $in, and $regex replace SQL WHERE keywords.

Update
SQL UPDATE users SET age = 31 WHERE name = 'Alice'

Updates only the named column; all other columns remain unchanged.

MongoDB col.update_one({"name": "Alice"}, {"$set": {"age": 31}})

The $set wrapper is required. Omitting it replaces the entire document with only the fields you specified.

Delete
SQL DELETE FROM users WHERE name = 'Alice'

Permanently removes all matching rows. An empty WHERE removes the entire table's contents.

MongoDB col.delete_one({"name": "Alice"})

Removes the first matching document. To clear all documents, use delete_many({}) with an empty filter — equivalent to SQL's TRUNCATE.

Advanced Queries & Indexing
GROUP BY vs Aggregation Pipeline
SQL SELECT city, COUNT(*) … GROUP BY city

Declarative: you describe the result you want and the database decides how to compute it.

MongoDB aggregate([{"$group": …}, {"$sort": …}])

A pipeline of stages — $match, $group, $sort — where data flows through each stage sequentially.

Indexing
SQL CREATE INDEX idx ON users (age)

Speeds up WHERE, JOIN, and ORDER BY clauses on the indexed column.

MongoDB col.create_index([("age", 1)])

The 1 means ascending; use -1 for descending. Without an index, every find() scans the entire collection.

The mapping is intentionally one-directional: every SQL concept has a MongoDB analog, but MongoDB has concepts — like embedded documents, the aggregation pipeline, and flexible fields — that have no direct SQL equivalent. These are the features you reach for when SQL starts to feel unnatural.

01

MongoDB CRUD

Beginner

What This Does

Perform all four CRUD operations on a MongoDB collection using pymongo — with realistic e-commerce product data.

mongodb_crud.py 
from pymongo import MongoClient

# Connect to MongoDB running locally on the default port
client = MongoClient("mongodb://localhost:27017/")
db = client["ecommerce"]           # select (or create) the database
products = db["products"]           # select (or create) the collection

# CREATE — insert two documents; they can have completely different fields
products.insert_many([
    {"sku": "LAPTOP-001", "name": "Pro Laptop 15", "price": 1299.99,
     "specs": {"ram_gb": 16, "storage_tb": 1}},   # nested sub-document is valid
    {"sku": "BOOK-042",   "name": "Python for Engineers", "price": 39.99,
     "author": "Jane Smith"}      # no "specs" field — flexible schema allows this
])
print(f"Inserted {products.count_documents({})} products")

# READ — find_one() returns the first matching document as a Python dict (or None)
laptop = products.find_one({"sku": "LAPTOP-001"})
print(f"Found: {laptop['name']} @ ${laptop['price']}")

# UPDATE — $set modifies only named fields; omitting $set replaces the WHOLE document
products.update_one({"sku": "LAPTOP-001"}, {"$set": {"in_stock": True}})

# DELETE — delete_many({}) with an empty filter removes the entire collection
products.delete_many({"discontinued": True})
print("Update and delete applied")
Terminal $ python mongodb_crud.py
Inserted 2 products
Found: Pro Laptop 15 @ $1299.99
Update and delete applied

The $set operator in update_one() is not optional. Writing update_one({"sku": "X"}, {"price": 999}) without $set replaces the entire document with only {"price": 999}, silently deleting every other field.

02

MongoDB Query Operators

Beginner

What This Does

Filter documents using MongoDB query operators — exact match, range ($lt/$gt), list membership ($in), and field projection.

mongo_queries.py 
from pymongo import MongoClient

client = MongoClient("mongodb://localhost:27017/")
products = client["ecommerce"]["products"]

# Exact match — returns all documents where category equals "laptop"
laptops = list(products.find({"category": "laptop"}))

# Range query — $lt (less than) and $gt (greater than)
affordable = list(products.find({"price": {"$lt": 500, "$gt": 50}}))

# $in — matches any value in the list (equivalent to SQL's IN clause)
selected = list(products.find({"brand": {"$in": ["Dell", "Lenovo", "HP"]}}))

# Projection — second arg picks which fields to return (1 = include, 0 = exclude)
names_only = list(products.find(
    {"in_stock": True},
    {"name": 1, "price": 1, "_id": 0}   # return name + price; exclude _id explicitly
))

# count_documents() is faster than len(list(find(...))) for large collections
count = products.count_documents({"price": {"$gt": 1000}})
print(f"Premium products: {count}")
print(f"Affordable options: {len(affordable)}")
print(f"First result: {names_only[0] if names_only else 'none'}")
Terminal $ python mongo_queries.py
Premium products: 3
Affordable options: 12
First result: {'name': 'Wireless Mouse', 'price': 29.99}

find() returns a cursor, not a list. Wrapping it in list() loads all results into memory at once. For large collections, iterate with for doc in col.find(...) instead so Python processes one document at a time without exhausting RAM.

Nested Documents

Beginner

What This Does

Embed related data directly inside a MongoDB document and query into nested fields using dot notation — no JOIN required.

nested_docs.py 
from pymongo import MongoClient

client = MongoClient("mongodb://localhost:27017/")
orders = client["ecommerce"]["orders"]

# Embed all related data in ONE document — no separate tables needed
order = {
    "order_id": "ORD-2024-001",
    "customer": {                          # sub-document — replaces a JOIN to a customers table
        "name": "Alice Johnson",
        "email": "alice@example.com"
    },
    "items": [                             # array of sub-documents — replaces an order_lines table
        {"sku": "LAPTOP-001", "qty": 1, "unit_price": 1299.99},
        {"sku": "MOUSE-007",  "qty": 2, "unit_price": 29.99}
    ],
    "total": 1359.97,
    "status": "shipped"
}
orders.insert_one(order)

# Dot notation queries into nested fields — "customer.name" is the path
shipped = list(orders.find({"status": "shipped", "customer.name": "Alice Johnson"}))
print(f"Alice's shipped orders: {len(shipped)}")

# $elemMatch ensures ONE array element satisfies ALL conditions simultaneously
laptop_orders = list(orders.find({
    "items": {"$elemMatch": {"sku": "LAPTOP-001", "qty": {"$gte": 1}}}
}))
print(f"Orders containing LAPTOP-001: {len(laptop_orders)}")

# Read nested values from a result document like any Python dict
if laptop_orders:
    doc = laptop_orders[0]
    print(f"Customer : {doc['customer']['name']}")
    print(f"Line items: {len(doc['items'])}")
Terminal $ python nested_docs.py
Alice's shipped orders: 1
Orders containing LAPTOP-001: 1
Customer : Alice Johnson
Line items: 2

Embed data you always read together and that doesn't grow without bound. Use a separate collection with a reference _id only when the sub-document can grow indefinitely (e.g., a chat message history) or is shared by many parent documents — MongoDB has a hard 16 MB per-document limit.

Indexes & Performance

Intermediate

What This Does

Create single-field, compound, and text indexes in MongoDB, then use explain() to verify that queries are using indexes instead of scanning the entire collection.

mongo_indexes.py 
from pymongo import MongoClient, ASCENDING, DESCENDING, TEXT

client = MongoClient("mongodb://localhost:27017/")
products = client["ecommerce"]["products"]

# Single-field index — speeds up any find() that filters on "category"
products.create_index([("category", ASCENDING)], name="idx_category")

# Compound index — supports queries filtering on BOTH fields in order
products.create_index(
    [("category", ASCENDING), ("price", DESCENDING)],
    name="idx_cat_price"
)

# Text index — enables full-text search with $text operator
products.create_index([("name", TEXT)], name="idx_name_text")

# Full-text search — returns documents where "name" contains these words
results = list(products.find({"$text": {"$search": "laptop pro"}}))
print(f"Text search results: {len(results)}")

# explain() shows the query execution plan — IXSCAN means index used
plan = products.find({"category": "laptop"}).explain()
stage = plan["queryPlanner"]["winningPlan"]["stage"]
print(f"Query plan stage: {stage}")   # FETCH/IXSCAN = good; COLLSCAN = no index!

# List all indexes on the collection to confirm they were created
for idx in products.list_indexes():
    print(f"  {idx['name']:20} keys={dict(idx['key'])}")
Terminal $ python mongo_indexes.py
Text search results: 3
Query plan stage: FETCH
_id_ keys={'_id': 1}
idx_category keys={'category': 1}
idx_cat_price keys={'category': 1, 'price': -1}
idx_name_text keys={'_fts': 'text', '_ftsx': -1}

For compound indexes, put the most selective field first — the field that eliminates the most documents early. An index on (status, created_at) where status has only 3 distinct values is far less useful than (created_at, status). Run .explain("executionStats") and look at totalDocsExamined vs nReturned — a good index makes those numbers equal.

05

MongoDB Aggregation Pipeline

Intermediate

What This Does

Use the MongoDB aggregation pipeline to compute GROUP BY-style analytics — filter completed orders, sum revenue by region, sort descending, and return the top 5.

mongo_agg.py 
from pymongo import MongoClient

client = MongoClient("mongodb://localhost:27017/")
orders = client["ecommerce"]["orders"]

# Aggregation pipeline — data flows through each stage sequentially
pipeline = [
    # Stage 1: $match filters early — less data to process in later stages
    {"$match": {"status": "completed"}},

    # Stage 2: $group acts like SQL GROUP BY; _id is the grouping key
    {"$group": {
        "_id": "$region",                    # group documents by their "region" field
        "total_revenue": {"$sum": "$amount"},# sum the "amount" field for each group
        "order_count":   {"$sum": 1}         # count documents per group (like COUNT(*))
    }},

    # Stage 3: $sort orders results (-1 = descending, highest revenue first)
    {"$sort": {"total_revenue": -1}},

    # Stage 4: $limit keeps only the top 5 groups
    {"$limit": 5}
]

results = list(orders.aggregate(pipeline))   # execute and materialise the cursor
for row in results:
    print(f"{row['_id']:12}  revenue=${row['total_revenue']:,.2f}  orders={row['order_count']}")
Terminal $ python mongo_agg.py
North East revenue=$482,310.00 orders=1203
West Coast revenue=$371,845.50 orders=988
Midwest revenue=$298,410.00 orders=754
South revenue=$241,620.75 orders=623

Always place $match as the first stage. MongoDB can use an index to satisfy a leading $match, but if $group runs first it must read every document in the collection before any filter is applied. Putting $match last on a 10-million-document collection can turn a 0.3-second query into a 45-second one.

02

NoSQL Categories

Covers: Document, Key-Value, Column-Family, Graph

Tasks

  1. 1For each scenario, name the NoSQL category that fits best: (a) an e-commerce catalog where each product type has different attributes; (b) a session cache that maps a token to a user object for fast lookup; (c) billions of IoT sensor readings arriving every second; (d) a social network that needs to find friends-of-friends within three hops.
  2. 2For each of your four answers, name one real-world tool that implements that NoSQL category (e.g., MongoDB, Redis, Cassandra, Neo4j).
  3. 3In one sentence each, explain why a standard SQL database would be a worse choice for scenarios (c) and (d).

Answers

(a) Document — MongoDB / Firestore. Each product is a self-contained JSON document, so electronics can have a wattage field that clothing products simply omit — no schema migration needed.

(b) Key-Value — Redis / DynamoDB. A single-token lookup returns the full session object in under a millisecond; there are no joins and no complex queries to perform.

(c) Column-Family — Apache Cassandra / HBase. SQL write throughput is bounded by a single coordinator node; Cassandra distributes writes across a cluster with no bottleneck, making it purpose-built for time-series and sensor workloads.

(d) Graph — Neo4j / Amazon Neptune. Finding friends-of-friends in SQL requires chaining multiple self-JOIN operations on a massive table; as the network grows these joins become exponentially slower, whereas a graph database traverses relationships in constant time per hop.

Why this matters: Choosing the wrong database category for a use case is extremely costly to fix once data is in production — thinking through your access patterns before you write a single line of code saves months of re-architecture later.

03

Query Operators

Covers: $gt, $lt, $in, array matching, $regex

Tasks

  1. 1Seed the store.products collection with 5 products (Laptop Pro, USB Cable, Mouse Pro, Desk Lamp, Keyboard) each having a category, price, and tags array.
  2. 2Use $gt and $lt inside a single filter dict to find products with price between 10 and 50 (exclusive).
  3. 3Use $in to find products in category "electronics" or "accessories" in a single query, then find all products whose tags array contains "sale".
  4. 4Use $regex with $options: "i" to find all products whose name contains the word "Pro" regardless of case.
solution_03.py 
from pymongo import MongoClient

client = MongoClient("mongodb://localhost:27017/")
products = client["store"]["products"]

# Task 1: seed data
products.drop()
products.insert_many([
    {"name": "Laptop Pro",  "category": "electronics", "price": 999, "tags": ["sale", "featured"]},
    {"name": "USB Cable",   "category": "accessories", "price": 12,  "tags": ["sale"]},
    {"name": "Mouse Pro",   "category": "accessories", "price": 45,  "tags": ["new"]},
    {"name": "Desk Lamp",   "category": "furniture",   "price": 35,  "tags": ["featured"]},
    {"name": "Keyboard",    "category": "electronics", "price": 80,  "tags": ["new", "sale"]},
])

# Task 2: both conditions in the same dict = implicit AND
mid = list(products.find({"price": {"$gt": 10, "$lt": 50}}))
print("Price 10–50:", [p["name"] for p in mid])   # USB Cable, Mouse Pro, Desk Lamp

# Task 3a: $in matches any value in the list
multi = list(products.find({"category": {"$in": ["electronics", "accessories"]}}))
print("Elec/Access:", [p["name"] for p in multi])

# Task 3b: querying an array field with a scalar checks for membership
on_sale = list(products.find({"tags": "sale"}))
print("On sale:", [p["name"] for p in on_sale])   # Laptop Pro, USB Cable, Keyboard

# Task 4: $regex with $options "i" = case-insensitive
pro = list(products.find({"name": {"$regex": "Pro", "$options": "i"}}))
print("'Pro' items:", [p["name"] for p in pro])   # Laptop Pro, Mouse Pro

Why this matters: Query operators are the building blocks of every MongoDB filter — once you understand how $gt, $in, and $regex compose inside a single filter document, you can express virtually any read query without writing a single SQL JOIN.

Aggregation Pipeline

Covers: $match, $group, $sort, $limit

Tasks

  1. 1Add a $match stage to filter the sales.orders collection to EMEA orders only.
  2. 2Add a $group stage that groups by product, sums the amount field as total, and counts orders using $sum: 1.
  3. 3Add a $sort stage to order by total revenue descending (-1).
  4. 4Add a $limit stage to keep the top 3 products, call aggregate(pipeline), and print the results.
solution_04.py 
from pymongo import MongoClient

client = MongoClient("mongodb://localhost:27017/")
orders = client["sales"]["orders"]

pipeline = [
    {"$match": {"region": "EMEA"}},         # Task 1: discard non-EMEA documents
    {"$group": {
        "_id": "$product",                  # Task 2: group key
        "total": {"$sum": "$amount"},       # sum the amount field
        "order_count": {"$sum": 1},         # increment by 1 per document
    }},
    {"$sort": {"total": -1}},              # Task 3: -1 = descending
    {"$limit": 3},                         # Task 4: keep top 3 only
]

results = list(orders.aggregate(pipeline))
for r in results:
    print(r["_id"], r["total"], r["order_count"])

Why this matters: The aggregation pipeline is MongoDB’s answer to SQL GROUP BY + WHERE + ORDER BY + LIMIT — every analytics query your application needs can be expressed with these four stages, and placing $match first is critical because it reduces the number of documents all later stages must process.

05

NoSQL or SQL?

Covers: decision framework for real-world scenarios

Tasks

For each scenario below: (i) name the most appropriate database type, (ii) name one real-world tool, and (iii) in one sentence explain why SQL would be a worse choice.

  1. AA hospital records system where every patient has the same 20 structured fields, with records frequently joined to appointment and billing tables.
  2. BA real-time leaderboard for a mobile game with 10 million active players where each score update must complete in under 5 milliseconds.
  3. CA content management system where blog posts have irregular custom metadata — news articles have a dateline, product reviews have a rating, video posts have a duration.
  4. DA fraud detection engine that must trace rings of connected accounts (A → B → C → D) up to 4 relationship hops within 100 ms.

Answers

(A) SQL — PostgreSQL or MySQL. Every record is identical and the data has natural relationships to other tables, so relational constraints and JOINs are a perfect fit — a document database would add complexity with zero benefit when the schema never varies.

(B) Key-Value — Redis. A single-key lookup takes under a millisecond and Redis sorted sets (ZADD / ZRANGE) are purpose-built for leaderboards — SQL could not sustain 10 million concurrent score writes at that latency.

(C) Document — MongoDB or Firestore. The irregular per-post-type metadata maps naturally to flexible JSON documents — an SQL table would require either dozens of nullable columns or a complex entity-attribute-value pattern to accommodate the variation.

(D) Graph — Neo4j or Amazon Neptune. Tracing four-hop chains in SQL requires chaining four self-JOINs on a massive table, which grows exponentially slower as the network expands — a graph database traverses each hop in constant time regardless of graph size.

Why this matters: Senior engineers are judged not just on writing correct code but on selecting the right tool for the job — being able to articulate the trade-offs between SQL and each NoSQL category is a common topic in data engineering interviews and directly influences architecture decisions.

Common Mistakes

Pitfalls beginners hit when working with NoSQL databases

Replicating SQL JOINs with Manual Python Lookups

In-Python joins across two collections make two network round-trips and lose all query optimization.

Pitfall

MongoDB has no native JOIN — fetching a customer then looping over a separate orders collection makes two round-trips and bypasses the query optimizer, making the code slower and harder to maintain than a SQL JOIN. If your query always needs data from two collections, either embed the related data in one document or move to a relational database.

Wrong — two round-trips, no optimizer 

# Two network calls + Python loop = slow + fragile
customer = customers.find_one({"_id": customer_id})
order_list = list(orders_col.find({"customer_id": customer_id}))

Correct — embed related data in one document 

# recent_orders is embedded — one round-trip, no join needed
customer = customers.find_one({"_id": customer_id})
order_list = customer["recent_orders"]  # already inside the document

Think of each MongoDB document as a self-contained record — if you find yourself writing a second find() call to “complete” the first, that data probably belongs embedded in the original document. When data has many complex relationships with strict integrity rules, PostgreSQL will serve you better.

Querying Without an Index Causes a Full Collection Scan

Without an explicit index, every find() reads every document in the collection regardless of size.

Pitfall

MongoDB does not create indexes on arbitrary fields automatically — without one, find() performs a COLLSCAN (collection scan), reading every document even when only one matches. Call create_index() once at application startup for every field you filter on regularly.

Wrong — COLLSCAN on every query 

# Reads all 500,000 documents to find one product
product = products.find_one({"sku": "LAPTOP-001"})

Correct — index created at startup 

# Create once at startup — find() now uses O(log n) IXSCAN
products.create_index([("sku", 1)], unique=True)
product = products.find_one({"sku": "LAPTOP-001"})  # instant lookup

Run products.find({"sku": "LAPTOP-001"}).explain() before and after adding the index. Look for "stage": "COLLSCAN" vs "IXSCAN" to confirm the query plan changed. Only bulk export queries should ever show COLLSCAN.

Choosing MongoDB When Every Record Has Identical Fields

Uniform-schema data belongs in SQL, which provides type constraints, JOIN support, and aggregation optimization for free.

Pitfall

When every record has the same fixed fields — like a payroll table with identical columns for all 5,000 employees — MongoDB adds no flexibility benefit and removes the type enforcement, foreign-key constraints, and aggregation optimizer that SQL provides automatically. A quick test: if fewer than 20% of your sampled records have unique field combinations, use SQL.

Wrong — fixed schema in MongoDB 

# Same 4 fields on every document — zero flexibility benefit
employees.insert_one({"emp_id": 42, "name": "Alice",
    "dept": "Engineering", "salary": 92000})

Correct — use SQL for uniform records 

# SQL enforces types, constraints, and enables JOINs automatically
# INSERT INTO employees (emp_id, name, dept, salary)
# VALUES (42, 'Alice', 'Engineering', 92000);

Ask: “Would this data fit neatly in a spreadsheet with consistent columns?” If yes, it is relational data — use SQL. NoSQL's flexibility benefit only applies when records genuinely have different shapes, such as a product catalog where electronics have a wattage field that furniture items simply omit.

Calling update_one() Without $set Silently Replaces the Entire Document

A bare update dictionary deletes every field not included in the update, causing irreversible silent data loss.

Pitfall

Unlike SQL UPDATE which patches named columns, update_one() treats a bare dictionary as a full replacement document — it overwrites the matched document with your dict and silently deletes every field you did not include. Always wrap the fields you want to change inside {"$set": {...}}.

Wrong — replaces entire document 

# DANGER: deletes name, role, languages — only price survives
products.update_one({"sku": "LAPTOP-001"}, {"price": 899.99})

Correct — patches price field only 

# $set updates only price; name, role, languages are untouched
products.update_one(
    {"sku": "LAPTOP-001"},
    {"$set": {"price": 899.99}}   # only this field changes
)

This mistake is silent — MongoDB confirms “1 document modified” in both cases, so you will not notice until you read the document back and find only one field. Always read the document after your first update_one() in a new application to confirm only the intended field changed.

Using Redis as the Sole Data Store Instead of a Cache Layer

Redis stores data in RAM; all data is lost on server restart unless persistence is configured, making it unsuitable as a primary database.

Pitfall

Redis is designed for temporary, fast lookups in RAM — it is not a durable database. Storing order records or user profiles in Redis without a TTL and without a persistent backing store means all data is permanently lost on every server restart or crash. Save durable records in PostgreSQL or MongoDB first, then use Redis as a short-lived cache layer with a TTL.

Wrong — orders in Redis only 

# No TTL, no persistent backing store — lost on restart
r.set(f"order:{order_id}", json.dumps(order_data))

Correct — MongoDB primary, Redis cache 

orders_col.insert_one(order_data)        # persist permanently in MongoDB
r.setex(f"order:{order_id}", 300,      # cache in Redis, expires in 5 min
        json.dumps(order_data))

The rule: if losing this data for 5 minutes would cause a business problem, it must live in a durable database. Redis fits perfectly as a cache layer in front of MongoDB or PostgreSQL — write to the durable store first, then populate Redis for fast reads with a TTL.
