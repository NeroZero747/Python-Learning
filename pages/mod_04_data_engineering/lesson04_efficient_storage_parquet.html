Overview

A high-level summary of the topic

Py

A CSV stores data the way a spreadsheet does — one row at a time. Parquet stores it the way an analytics engine thinks — one column at a time, compressed per column. That single difference makes Parquet 5–20× faster and 3–10× smaller.

Think of row vs column storage like two ways of organizing employee records. A row store (CSV) is like a binder where each page holds one employee’s complete record. A column store (Parquet) is like a set of indexed folders — one folder for all names, one for all salaries. When you ask “average salary?” you only open the salary folder.

CSV (Row Store)

One page per employee — complete record per row

Reads every column even when you only need one. Good for exports, bad for analytics at scale.

Parquet (Column Store)

Indexed folders by field — open only what you need

Stores each column contiguously with per-column compression. Reads only the columns your query touches.

Compression

Vacuum-sealed bags — same contents, fraction of the space

Parquet compresses each column independently. Columns with repetitive data (like region) compress extremely well.

Predicate Pushdown

Only open the folders that match your search

The reader checks row group statistics (min/max) and skips entire chunks that can’t match your filter. This means filtering happens at disk level, before data enters memory.

For analytics workloads — aggregations, filters, selecting specific columns — columnar storage is almost always faster and smaller than CSV.

Key Takeaways

The most important ideas to remember

Parquet Is the Industry Standard

Every major data platform — BigQuery, Snowflake, Databricks, AWS Athena — uses columnar storage internally or accepts Parquet as the native import format. When you save pipeline output as Parquet, it is immediately compatible with every major cloud analytics tool without any conversion step.

Parquet Columnar Industry standard

Requesting 4 of 200 Columns Reads Over 98% Less Data From Disk

Unlike CSV, Parquet lets you request specific columns and the reader skips everything else. On a 200-column dataset, reading 4 columns means 98% less data read.

Column pruning Selective reads I/O savings

Every pd.read_parquet() Call Silently Depends on PyArrow Being Installed

When you call pd.read_parquet(), pandas delegates all the actual work to PyArrow — which handles compression, encoding, and zero-copy reads. If PyArrow is not installed, the call fails with ModuleNotFoundError: No module named 'pyarrow', so always add it to your project dependencies alongside pandas.

PyArrow Apache Arrow Zero-copy

Use CSV to Share Data With Colleagues; Use Parquet to Analyze It Repeatedly

Parquet is a binary format that non-technical stakeholders cannot open in Excel, so CSV is still the right choice for any one-time export or external handoff. For every pipeline that reads the same dataset more than once — aggregations, joins, model training — Parquet pays back its overhead many times over in speed and storage savings.

Format choice Analytics use case Data portability

Key Concepts

Row vs column storage, writing and reading Parquet, column selection, and when to use each format.

Row vs Column Storage

How CSV and Parquet physically store your data

Storage model

A row store saves all fields for one row next to each other on disk. Reading one column means reading every column anyway. A columnar store like Parquet saves each column separately. You can request only salary and nothing else is ever read from disk.

CSV vs Parquet

CSV Parquet
Layout Row-oriented Column-oriented
Read 1 column ✗ Reads all ✓ Skips rest
Human-readable ✓ Plain text ✗ Binary only
Use for Sharing, exports Analytics, pipelines
Python 
import pandas as pd

# CSV: ALL columns are read even though we only need salary
df = pd.read_csv("employees.csv")           # reads every column
avg = df["salary"].mean()                   # extract after full load

# Parquet: ONLY salary is read — other columns never leave disk
avg = pd.read_parquet(
    "employees.parquet",
    columns=["salary"],                     # skips name, dept, start_date
)["salary"].mean()

Skipping a column costs nothing. Parquet stores each column in a separate section. A column you didn't request is never decompressed or moved into memory at all.

Write Parquet

to_parquet() and read_parquet() essential parameters

Method

to_parquet() saves a pandas DataFrame as a Parquet file on disk. read_parquet() loads it back. One line each — the file is ready to use immediately. Parquet preserves each column's data type automatically, so you never need to re-cast columns after loading.

Key Parameters

index=False Omit the row index. Use this almost always — without it, a mystery __index_level_0__ column appears in your file.
compression= "snappy" is the default — fast with good compression. Use "gzip" for smaller files at the cost of speed.
engine= Leave as default "pyarrow". Only change this if a specific tool requires "fastparquet".
Python 
import pandas as pd

df = pd.read_csv("sales.csv")               # load from CSV

# Write: index=False keeps your schema clean
df.to_parquet("sales.parquet", index=False) # compression="snappy" by default

# Read back: dtypes are restored exactly — no casting needed
df_back = pd.read_parquet("sales.parquet")

Always pass index=False when writing. The default saves the DataFrame's integer index as an extra column. That column adds file size and confuses any tool that reads the file later.

Column Selection & Filtering

Read only the columns and rows you actually need

Optimization

The columns= argument tells Parquet which columns to load. Every other column is skipped at the file level — never read from disk. The filters= argument skips entire row groups using stored statistics. Use both together for the biggest speed gains.

Read Parameters

Param What it skips Example value
columns= Unused columns (never read from disk) ["order_id", "amount"]
filters= Row groups that can't match (predicate pushdown) [("region", "==", "EMEA")]
Python 
import pandas as pd

# columns= skips unused columns at the file level
df = pd.read_parquet(
    "sales.parquet",
    columns=["order_id", "amount", "region"], # only 3 of 50 columns loaded
)

# filters= skips row groups that can't match — predicate pushdown
df_emea = pd.read_parquet(
    "sales.parquet",
    filters=[("region", "==", "EMEA")],       # skips non-EMEA row groups
    columns=["order_id", "amount"],           # combine both for max speed
)

Use columns= and filters= together. On a 50-column dataset, requesting 3 columns and filtering by region can reduce the data read from disk by over 95% compared to a plain pd.read_parquet() call.

When to Convert

Pick the right format for each job

Decision

CSV is plain text that opens in Excel or any text editor. Parquet is binary and requires Python or a query engine to read. Use CSV to share data with colleagues. Use Parquet for every analytical step that runs repeatedly — loading it five times costs the same as loading it once.

CSV vs Parquet

Situation CSV Parquet
Sharing with colleagues ✓ Excel-friendly ✗ Can't open in Excel
Millions of rows ✗ Slow to load ✓ Fast + compressed
Repeated analysis ✗ Re-parses text each time ✓ Binary — instant load
Cloud tools (BigQuery, Spark) Accepted but slow ✓ Native format
Python 
import pandas as pd

# INGEST: receive data as CSV from an external source
df = pd.read_csv("raw_sales.csv")

# OPTIMIZE: clean data and fix dtypes once
df["amount"] = df["amount"].astype("float32")  # shrink numeric footprint

# STORE: save as Parquet for all future analysis
df.to_parquet("sales.parquet", index=False)    # do this once

# SHARE: export a subset as CSV for a colleague
df[df["region"] == "EMEA"].to_csv(            # human-readable for Excel
    "emea_sales.csv", index=False
)

Convert once, analyze many times. A common pipeline pattern is: ingest CSV → clean → save as Parquet → run all analysis against Parquet. You only pay the CSV parsing cost once.

Partitioning

Split files by column values for faster queries

Performance

Partitioning splits a large Parquet dataset into separate folders based on the values of one or more columns — typically a date column like year or month. When you query only one month, the engine skips every other folder entirely. This is called partition pruning, and it can reduce read time by orders of magnitude on large datasets.

Common Partitioning Strategies

Strategy How it works Best for
Date-based Folder per year/month/day Time-series data, logs
Categorical Folder per region or category Sales by region, products
Hash-based Distribute rows evenly across N buckets Even load distribution (Spark)
Python 
import pandas as pd

df = pd.read_csv("sales_2024.csv")         # 12 months of data
df["month"] = pd.to_datetime(df["date"]).dt.to_period("M")

# Write partitioned Parquet — one folder per month
df.to_parquet(
    "sales_partitioned/",
    partition_cols=["month"],               # creates month=2024-01/ etc.
    index=False
)

# Read only January — engine skips 11 other folders
jan = pd.read_parquet("sales_partitioned/", filters=[("month", "==", "2024-01")])

Pick a column your queries filter on most. Partitioning by a column nobody filters on adds folder overhead with no speed benefit. Date columns are the safest default because most analytics queries filter by time range.

CSV vs Parquet — Size & Speed

Beginner

What This Does

Compare file sizes and read speed between CSV and Parquet.

csv_vs_parquet.py 
import pandas as pd, time, os             # import required modules

df.to_csv("sales.csv", index=False)        # write the full DataFrame as CSV
df.to_parquet("sales.parquet", index=False) # write the same data as Parquet

csv_mb = os.path.getsize("sales.csv") / 1024**2     # CSV file size in MB
pq_mb  = os.path.getsize("sales.parquet") / 1024**2  # Parquet file size in MB
print(f"CSV: {csv_mb:.1f} MB  Parquet: {pq_mb:.1f} MB  ({csv_mb/pq_mb:.1f}x smaller)")

t0 = time.perf_counter()                    # start timing CSV read
pd.read_csv("sales.csv")["amount"].sum()   # loads ALL columns, then extracts one
csv_time = time.perf_counter() - t0         # elapsed seconds for CSV

t0 = time.perf_counter()                    # start timing Parquet read
pd.read_parquet("sales.parquet", columns=["amount"])["amount"].sum()  # skips all other columns
pq_time = time.perf_counter() - t0          # elapsed seconds for Parquet
print(f"CSV: {csv_time:.2f}s  Parquet: {pq_time:.2f}s  ({csv_time/pq_time:.1f}x faster)")
Terminal $ python csv_vs_parquet.py
CSV: 412.3 MB Parquet: 58.7 MB (7.0x smaller) CSV: 8.41s Parquet: 0.73s (11.5x faster)

Notice that Parquet cuts both file size (7×) and read time (11.5×) simultaneously — the columnar layout reduces what is physically read from disk, and built-in compression reduces the bytes transferred from storage. You get both benefits with no extra effort.

03

Column Selection & Predicate Pushdown

Beginner

What This Does

Use columns= and filters= with pandas to read only the data you need, skipping the rest at the file level.

column_selection.py 
import pandas as pd

# ── columns= : skip columns at the file level ───────────────────────────────
# Only 3 of 50 columns are decompressed — the rest are never read from disk
df = pd.read_parquet(
    "sales.parquet",
    columns=["order_id", "amount", "region"],  # request only what you need
)
print(f"Shape: {df.shape}")   # (500000, 3) — all rows, 3 columns

# ── filters= : skip row groups whose data cannot match ─────────────────────
# Parquet checks min/max statistics per row group and skips non-matching ones
df_emea = pd.read_parquet(
    "sales.parquet",
    filters=[("region", "==", "EMEA"), ("amount", ">", 500)],
    columns=["order_id", "amount"],           # combine both for maximum speed
)
print(f"EMEA >500: {df_emea.shape}")          # rows reduced; only 2 columns loaded
Terminal $ python column_selection.py
Shape: (500000, 3) EMEA >500: (87234, 2)

Notice the row count dropped from 500,000 to 87,234 — that reduction happened before any data entered Python memory. filters= does not filter a DataFrame; it tells Parquet which chunks of the file to even open. Combine it with columns= to skip both rows and columns at the disk level simultaneously.

04

PyArrow for Finer Control

Intermediate

What This Does

Use PyArrow directly for explicit compression and predicate pushdown.

pyarrow_control.py 
import pyarrow as pa                                    # Apache Arrow core library
import pyarrow.parquet as pq                           # Parquet read/write module

table = pa.Table.from_pandas(df)                       # convert pandas DataFrame to Arrow Table
pq.write_table(table, "sales.parquet",
    compression="snappy",                              # snappy: fast, good ratio (default)
    row_group_size=100_000)                            # smaller groups = finer pushdown granularity

dataset = pq.read_table("sales.parquet",
    columns=["order_id", "amount"],                    # load only these 2 columns
    filters=[("amount", ">", 1000)])                   # predicate pushdown: skip rows ≤ 1000
df = dataset.to_pandas()                               # convert Arrow Table back to DataFrame
Terminal $ python pyarrow_control.py
Table written: sales.parquet (snappy, row_group_size=100,000) Filter applied: 1,483 rows where amount > 1000 Shape: (1483, 2)

Use PyArrow directly when you need fine-grained control: custom compression, explicit row-group tuning, or pushing down complex multi-column filters. For everyday tasks pd.read_parquet() is simpler, but both ultimately call the same PyArrow engine under the hood.

05

Best Practice — CSV In, Parquet Out

Beginner

What This Does

A pipeline step that reads CSV, optimizes dtypes, and saves as Parquet.

csv_to_parquet.py 
import pandas as pd, os

def csv_to_parquet(filename):
    df = pd.read_csv(os.path.join("raw/", filename))    # load raw CSV from input folder
    for col in df.select_dtypes("int64"):               # find integer columns
        df[col] = pd.to_numeric(df[col], downcast="integer")  # shrink to smallest int type
    for col in df.select_dtypes("float64"):             # find float columns
        df[col] = pd.to_numeric(df[col], downcast="float")    # shrink to float32
    for col in df.select_dtypes("object"):              # find string columns
        if df[col].nunique() / len(df) < 0.1:           # <10% unique = low cardinality
            df[col] = df[col].astype("category")        # convert to category for better compression
    out = os.path.join("clean/", filename.replace(".csv", ".parquet"))  # output path
    df.to_parquet(out, index=False)                     # write Parquet without row index
    print(f"Saved {filename} -> {out} ({len(df):,} rows)")
Terminal $ python csv_to_parquet.py
Saved jan_2025.csv -> clean/jan_2025.parquet (482,156 rows) Saved feb_2025.csv -> clean/feb_2025.parquet (391,047 rows)

This pattern — ingest CSV, optimize dtypes, save as Parquet — is one of the most common data engineering pipeline steps. Running it once on raw data can shrink storage costs by 70–90% and speed up every downstream query for the lifetime of the project.

04

Predicate Pushdown Filters

Covers: Objective 3 — filters= pushdown

Tasks

  1. 1Read sales.parquet with columns=["region", "amount", "product"] and print the shape.
  2. 2Re-read the same file adding filters=[("region", "==", "EMEA"), ("amount", ">", 500)] and print the new shape.
  3. 3Calculate and print how many rows were skipped and what percentage of the total that represents.
  4. 4Add an inline comment on the filters= line explaining at which stage the row reduction happened.
solution_04.py 
import pandas as pd

# ── Step 1: Load only the columns you need ────────────────────────────────────
df_all = pd.read_parquet(
    "sales.parquet",
    columns=["region", "amount", "product"],  # 3 of N columns; all rows loaded
)
print(f"All rows, 3 columns: {df_all.shape}")         # e.g. (500000, 3)

# ── Step 2: Add predicate pushdown — non-matching row groups are never read ───
df_emea = pd.read_parquet(
    "sales.parquet",
    columns=["region", "amount", "product"],
    filters=[("region", "==", "EMEA"), ("amount", ">", 500)],  # applied at file level
)
print(f"EMEA + amount >500: {df_emea.shape}")         # e.g. (87234, 3)

# ── Step 3: Measure how much was skipped before Python saw it ─────────────────
total_rows = df_all.shape[0]
skipped    = total_rows - df_emea.shape[0]
pct        = skipped / total_rows * 100
print(f"Rows skipped at file level: {skipped:,} ({pct:.1f}%)")

Why this matters: Predicate pushdown means the Parquet reader evaluates the filters= conditions against stored row-group statistics before decompressing any data — so the rows that don't match never enter Python memory at all, which is the key difference from a DataFrame .query() that filters after loading.

Common Mistakes

Pitfalls beginners hit when working with Parquet files

Saving Parquet with the Row Index Included

The default to_parquet() call silently adds an __index_level_0__ column that was not in your original data.

Pitfall

df.to_parquet() writes the pandas integer row index as an extra column by default. Pass index=False to write only the columns that were in your original DataFrame.

Wrong — extra column written 

df.to_parquet("data.parquet")
# adds __index_level_0__ silently

Correct — index excluded 

df.to_parquet("data.parquet", index=False)
# writes only your data columns

Think of index=False as "save the spreadsheet, not the row numbers." Every Parquet file written to a data warehouse should use it to keep the schema clean and avoid a phantom column appearing in downstream queries.

Reading Every Column When You Need Just One

pd.read_parquet() without columns= loads all columns into memory, even ones your analysis never uses.

Pitfall

Parquet supports column-level loading at read time. Calling read_parquet() without columns= loads every column into memory first, then your pandas expression filters down — the opposite of efficient. Pass columns= so unwanted columns are never decompressed.

Wrong — all 30 columns loaded 

df = pd.read_parquet("wide.parquet")
# all 30 columns enter memory
profit_total = df["profit"].sum()

Correct — one column loaded 

df = pd.read_parquet("wide.parquet",
    columns=["profit"])  # 1 of 30 loaded
profit_total = df["profit"].sum()

Column pruning is Parquet's biggest advantage over CSV — a CSV reader must scan every byte in every row to locate the columns you want, while Parquet skips entire column chunks at the storage layer. Name exactly the columns you need and Parquet does the rest.

Partitioning by a High-Cardinality Column

Using order_id as a partition key creates one file per order — millions of tiny files that are slower to read than no partitioning at all.

Pitfall

Parquet creates one sub-directory per unique value in the partition column. A column like order_id with millions of unique values produces millions of tiny files — each with its own metadata overhead — which completely cancels any read-time benefit. Partition by low-cardinality columns like year or region that split your data into a small number of meaningful folders.

Wrong — millions of files 

df.to_parquet("orders/",
    partition_cols=["order_id"])
# 500 k unique IDs → 500 k files

Correct — small number of folders 

df.to_parquet("orders/",
    partition_cols=["year", "region"])
# ~8 folders — fast and manageable

A good partition column has between 2 and roughly 200 unique values — enough to give queries a meaningful shortcut, small enough that each resulting file holds a reasonably sized block of data. When in doubt, partition by month or region first and only add more levels if query patterns justify it.

Filtering Rows in pandas After Reading the Full File

A pandas boolean mask loads every row into memory first — Parquet's filters= skips non-matching row groups before any data is decompressed.

Pitfall

A pandas boolean mask filters rows after they are already in memory — all rows were loaded and most are then discarded. Passing filters= directly to read_parquet() enables predicate pushdown: the Parquet reader skips entire row groups whose min/max statistics prove they cannot match, so those bytes are never decompressed.

Wrong — all rows loaded first 

df = pd.read_parquet("sales.parquet")
# all 10 M rows enter memory
emea = df[df["region"] == "EMEA"]

Correct — unmatched rows never read 

emea = pd.read_parquet("sales.parquet",
    filters=[("region", "==", "EMEA")])
# row groups skipped at file level

The key difference from a pandas .query() is where the filtering happens — filters= runs inside the Parquet file format before Python sees any data, whereas .query() runs after every row is already in a DataFrame.

Forgetting to Install PyArrow in the Project Dependencies

pd.read_parquet() raises ModuleNotFoundError at runtime because pandas does not bundle a Parquet engine — it delegates all I/O to PyArrow or fastparquet.

Pitfall

Pandas does not include a Parquet engine — it calls PyArrow (or fastparquet) as a separate installed library. If neither is present, every read_parquet() and to_parquet() call raises ModuleNotFoundError: No module named 'pyarrow' the first time it runs in a new environment.

Wrong — engine not declared 

# requirements.txt
pandas==2.2.0
# pyarrow absent → crashes at runtime

Correct — engine declared 

# requirements.txt
pandas==2.2.0
pyarrow>=14.0  # required Parquet engine

The error only surfaces when you deploy or create a fresh environment — everything works locally if PyArrow was already installed for another project. Always declare pyarrow explicitly in requirements.txt so it is never accidentally omitted.
