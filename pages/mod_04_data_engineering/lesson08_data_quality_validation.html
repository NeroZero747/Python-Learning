Overview

A high-level summary of the topic

Py

A data validation layer checks every incoming record against your quality rules and stops any that fail before they reach your warehouse.

Think of data validation like an airport security checkpoint. Each data record passes through a series of scanners before entering the departure lounge. Records that fail go to a holding area rather than disappearing silently. A validation layer is that checkpoint, and Python is the language you use to write it.

Schema Check

The ID scanner — flags unrecognized formats

A schema check verifies that every column has the expected data type and name — for example, that order_id is an integer and order_date is a date, not a plain string. Run it immediately after loading raw data so a supplier's field renaming or type change doesn't silently break every transform that follows.

Null Check

The X-ray — spots a completely blank pass

A null check counts how many rows in a required field contain no value at all, and raises an alert when that count exceeds the tolerance you set. Mandatory fields such as customer_id or amount must never be empty, because even one null can produce incorrect totals or join failures further down the pipeline.

Range Check

The bag scales — rejects overweight luggage

A range check confirms that numeric values stay within the expected bounds for that field — for example, that quantity is always between 1 and 9,999, or that discount_rate never exceeds 1.0. It catches data entry errors and feed anomalies before they inflate revenue figures or corrupt aggregate calculations.

Uniqueness Check

The seat manifest — same ID can’t board twice

A uniqueness check verifies that key fields such as order_id carry only distinct values within each batch, catching duplicate extracts or fan-out joins before they enter the warehouse. A single duplicated primary key can cause every downstream model that references this table to double-count rows.

Running Checks

The security protocol — runs all scans in one sequence

A validation check is a pandas expression that tests one quality rule against every row in a batch, returning a Boolean Series you count to measure how many records violated that rule. You chain multiple checks inside a single validate() function and call it after every extract so a quality report is generated automatically.

Quarantine Pattern

The holding area — failed records wait, clean ones board

The quarantine pattern separates each incoming batch into two DataFrames: clean rows that pass all checks and load into the warehouse, and invalid rows saved to a separate quarantine table for manual review. This prevents one bad record from crashing the entire load and creates an audit trail of every rejected row.

Schema Drift Detection

The route change alert — warns when the departure gate moves

Schema drift occurs when a data source silently adds, renames, or drops a column, which breaks any downstream transform that references the old column name without raising an obvious error. You detect it by comparing the current DataFrame's column list against an expected list using set operations and alerting before any transforms run.

If you have ever used Excel’s data validation to restrict cell values, this is the same idea applied to an entire pipeline.

Key Takeaways

The most important ideas to remember

Each Dimension Catches Failures the Others Miss

Each of the five dimensions — Completeness, Validity, Consistency, Uniqueness, and Timeliness — targets a different class of defect that the other four cannot detect on their own. A pipeline that only checks for nulls will still pass rows with an order_date of 1900-01-01, a quantity of −999, or a customer_id that appears 8,000 times — Validity, Range, and Uniqueness failures respectively.

Completeness Validity Uniqueness Timeliness

Assert Before Transforms, Not After

Running validation after a transform has already modified the data means you cannot tell whether the source supplied bad values or whether your own code introduced the problem. Log the row count, null counts, and violation count at the end of every check function so that when a downstream dashboard breaks, you can pinpoint which extract was the first to go wrong.

early validation logging fail-fast

One Bad Row Crashes the Whole Batch Without Quarantine

Without a quarantine pattern, a single invalid row causes to_sql() or to_parquet() to raise an exception and roll back the entire batch, discarding the other 9,999 clean rows alongside it. Saving failed rows to a separate table with a reason column lets analysts fix and resubmit records individually instead of re-running the full pipeline.

quarantine table partial load resubmit

Schema Drift Breaks Transforms Silently, Not Loudly

When a supplier renames customer_id to client_id, a join or column lookup referencing the old name does not raise an immediate error — it produces an all-null column that propagates silently until your reports show blank values. Comparing the incoming column set against an expected list with set(df.columns) - set(expected) catches the rename before any transforms run.

Key Concepts

Quality dimensions, check functions, quarantine pattern, and schema drift detection.

Quality Dimensions

The five categories of data defect your pipeline must check

5 Dimensions

A data quality dimension is a named category of data defect. Use all five — Completeness, Validity, Uniqueness, Consistency, and Timeliness — because each one catches a different class of error that the others cannot detect.

5 Quality Dimensions

Dimension What it catches pandas check
Completeness Null or missing values .notna().all()
Validity Out-of-range or wrong-type values .between(lo, hi).all()
Uniqueness Duplicate primary keys .is_unique
Consistency Contradictions between fields (df[A] >= df[B]).all()
Timeliness Stale or future-dated records df[col] >= yesterday
Python 
# COMPLETENESS — no required field should be null
assert df["order_id"].notna().all(), "Nulls found in order_id"

# VALIDITY — amount must sit within a sensible range
assert df["amount"].between(0, 1_000_000).all(), "Amount out of range"

# UNIQUENESS — each order must appear exactly once
assert df["order_id"].is_unique, "Duplicate order IDs found"

# CONSISTENCY — end date must never precede start date
assert (df["end_date"] >= df["start_date"]).all()

Wrap all asserts in one function. A single validate(df) call checks all five dimensions at once and makes the quality step impossible to skip.

Writing Checks

A reusable validate() function using pandas assertions

pandas

A validation check is a boolean pandas expression that tests one quality rule across every row in a DataFrame. Group multiple checks inside a single validate() function and call it right after loading raw data, before any transform runs.

Check Patterns

.notna().all() Completeness — no nulls in a required column
.between(lo, hi).all() Validity — every value is within the allowed range
.is_unique Uniqueness — no value appears more than once
assert cond, "msg" Raises ValueError with a clear message on failure
Python 
def validate(df):
    # completeness: order_id must never be null
    assert df["order_id"].notna().all(), "Nulls in order_id"
    # validity: amount must be a positive number below 1 million
    assert df["amount"].between(0, 1_000_000).all(), "Amount out of range"
    # uniqueness: no order can appear twice in the same batch
    assert df["order_id"].is_unique, "Duplicate order IDs"
    # consistency: end date must never precede start date
    assert (df["end_date"] >= df["start_date"]).all()
    print(f"Validation passed — {len(df):,} rows checked")
    return True

Call validate() before transforms, not after. Once you transform the data, a failure could come from your own code — you cannot tell whether the source or your transform introduced the problem.

Quarantine Pattern

Keep the pipeline alive — split clean rows from invalid ones

split

The quarantine pattern keeps your pipeline running when some rows fail validation. Build a boolean mask of passing rows, filter the DataFrame into a clean subset and a quarantine subset, then load only the clean rows to your warehouse.

clean vs quarantine

clean quarantine
Filter df[is_valid] df[~is_valid]
Destination Warehouse table Quarantine table
Next step Load normally Fix and resubmit
Python 
def split_quarantine(df):
    # build a Boolean mask — True means all rules pass for that row
    is_valid = (
        df["amount"].notna() &              # not null
        df["amount"].between(0, 1_000_000) &  # in valid range
        df["order_id"].notna()              # has an order ID
    )
    clean = df[is_valid].copy()             # rows that passed — load to warehouse
    quarantine = df[~is_valid].copy()       # rows that failed — save for review
    quarantine["reason"] = "failed_check"  # add a reason column for analysts
    return clean, quarantine

Always call .copy() on both subsets. Without .copy(), adding the reason column raises a SettingWithCopyWarning because the subset is a view, not an independent DataFrame.

Schema Drift

Detect added, renamed, or dropped columns using set operations

set ops

Schema drift happens when a data source silently changes its column structure — adding a new field, renaming one, or removing one. You detect it by comparing the incoming column set to a known-good expected list: EXPECTED - actual finds missing columns, and actual - EXPECTED finds unexpected new ones.

missing vs unexpected

missing unexpected
Set operation EXPECTED - actual actual - EXPECTED
Response raise ValueError print warning
Root cause Source removed a column Source added a new column
Python 
EXPECTED = {"order_id", "amount", "order_date", "customer_id"}

def validate_schema(df):
    actual  = set(df.columns)       # columns the source sent today
    missing = EXPECTED - actual     # columns we expected but did not get
    extras  = actual - EXPECTED     # columns we got but did not expect
    if missing:
        raise ValueError(f"Missing columns: {missing}")
    if extras:
        print(f"Warning — new columns detected: {extras}")
    return True

Store EXPECTED in a config file, not in the code. When a new column is added intentionally, update the config rather than editing the function — this keeps every schema change auditable and reviewable.

Data Governance

Policies, compliance, and data ownership

Policy

Data governance is the set of policies and procedures an organization uses to manage data quality, security, ownership, and lifecycle. Good governance means every dataset has an owner, every transformation is documented, and every access request follows a defined approval process.

Key Regulations

Regulation Scope Key requirement
GDPR EU residents' personal data Right to deletion, consent tracking, breach notification within 72 hours
CCPA California consumers Right to know what data is collected, right to opt out of data sales
HIPAA US healthcare data Encryption at rest and in transit, audit logs, minimum necessary access

Four pillars of data governance

Ownership — every table has an assigned team or individual responsible for its accuracy. Lineage — you can trace any row back to the pipeline and source that created it. Access control — role-based permissions ensure only authorized users can read or modify sensitive data. Retention — data has a defined lifecycle, and expired records are archived or deleted per policy.

Governance is not just a legal checkbox. When every dataset has clear ownership and lineage, debugging a broken dashboard takes minutes instead of hours because you know exactly which pipeline produced the data and who to contact.

01

Check for Nulls

Beginner Products assert, .isna()

What This Does

This script loads a products table and counts any rows where the name column is empty. It then uses assert — a one-line check that stops the script with an error message if the condition fails.

check_nulls.py 
import pandas as pd

products = pd.read_csv("products.csv")              # load the products table as a DataFrame
null_count = products["name"].isna().sum()          # count how many name values are missing
assert null_count == 0, "Some products have no name"  # stop immediately if any nulls are found
print("Check passed — all product names are present")  # only prints if the assert above passed
Terminal $ python check_nulls.py
Check passed — all product names are present

You can swap products["name"] for any column name — the same pattern checks completeness for any required field in your data.

02

Validate Amounts

Beginner Orders .between(), boolean mask

What This Does

This script tests that every amount value sits between 0 and 10,000. It builds a boolean mask — a True or False value per row — then counts the failures and stops if any are found.

validate_amounts.py 
import pandas as pd

orders = pd.read_csv("orders.csv")                            # load today's orders table
is_valid = orders["amount"].between(0, 10000)                 # True for rows where amount is in range
invalid_count = (~is_valid).sum()                             # count rows outside the allowed range
assert invalid_count == 0, f"{invalid_count} orders have an invalid amount"  # stop if any fail
print(f"Check passed — all {len(orders)} amounts are valid")  # report success
Terminal $ python validate_amounts.py
Check passed — all 1,200 amounts are valid

If the assert fires, run orders[~is_valid] in a notebook to see the exact rows that contain invalid amounts.

03

Split and Quarantine

Intermediate Sales quarantine, .copy()

What This Does

This script builds a boolean mask to test two rules at once — the amount must exist and be greater than zero. It then splits the DataFrame into a clean copy for loading and a quarantine copy for later review.

quarantine_rows.py 
import pandas as pd

sales = pd.read_csv("sales.csv")                             # load sales records as a DataFrame
is_valid = sales["amount"].notna() & (sales["amount"] > 0)  # True when amount exists and is positive
clean = sales[is_valid].copy()                               # rows that passed — safe to load
quarantine = sales[~is_valid].copy()                         # rows that failed — set aside for review
print(f"Clean: {len(clean)} rows | Quarantine: {len(quarantine)} rows")  # show the split
Terminal $ python quarantine_rows.py
Clean: 1,180 rows | Quarantine: 20 rows

Always call .copy() on both subsets — without it, adding a column to the quarantine rows raises a pandas warning about modifying a view.

Your Task

sales = pd.read_csv("sales.csv")                              # load the sales table as a DataFrame
is_bad = sales["amount"].isna() | (sales["amount"] < 0)     # True when amount is missing or negative
clean = sales[~is_bad].copy()                                 # rows that passed — safe to load

Common Mistakes

Pitfalls beginners hit when working with data quality validation

Using assert as a Production Safety Check

Python silently disables assert when running with the -O flag, so the check disappears without any warning.

Pitfall

assert is removed entirely when Python runs with the -O (optimize) flag, meaning your validation check silently disappears. Replace assert with an explicit if block that raises a ValueError — it runs every time regardless of how Python is started.

Wrong — bypassed with -O 

assert df["product_id"].notna().all()    # silently disabled with python -O

Correct — always raises 

if not df["product_id"].notna().all():          # check if any product IDs are missing
    raise ValueError("product_id has null values")  # always runs, cannot be disabled

assert is designed for development-time checks and automated tests — not production pipelines. Use raise ValueError for any check that must never be skipped.

Running Validation After Transforming the Data

Transforming corrupted data before validating it hides whether the pipeline or the source introduced the problem.

Pitfall

When you rename, filter, or calculate before validating, any problems visible in the raw source get overwritten — you can no longer tell whether the issue came from the source system or your own pipeline code. Always validate immediately after loading raw data, before any transforms run.

Wrong — transforms run first 

clean_df = raw_df.rename(columns={"amt": "amount"})  # transforms first
validate(clean_df)                                     # checking altered data

Correct — validate first 

validate(raw_df)                                       # check source before touching it
clean_df = raw_df.rename(columns={"amt": "amount"})  # safe to transform after checks

Think of validation as a quality gate on an assembly line — it only works at the very start, before any work has been done on the item. Moving it one step later makes the whole gate pointless.

Dropping Invalid Rows Instead of Quarantining Them

Calling dropna() permanently removes bad rows and makes it impossible to investigate the root cause later.

Pitfall

dropna() and boolean-filter assignment permanently delete rows from the DataFrame, so you lose all visibility into what went wrong and how many records were affected. Splitting into a clean copy and a quarantine copy keeps every row auditable.

Wrong — rows permanently lost 

clean_df = df.dropna(subset=["amount"])   # bad rows permanently deleted

Correct — quarantine for review 

is_valid = df["amount"].notna()            # True where amount exists
clean_df = df[is_valid].copy()             # good rows continue through the pipeline
quarantine_df = df[~is_valid].copy()       # bad rows saved for review

A quarantine copy is like a holding area at a warehouse — suspect items sit to the side for inspection rather than being thrown away, so every problem can be traced back to its source.
