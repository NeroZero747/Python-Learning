Overview

A high-level summary of the topic

Py

Writing a pipeline that works on your laptop is step one. Making it run reliably every night — without you touching it — is the real engineering challenge.

Think of a production pipeline like running a daily newspaper. The press runs automatically at the same time every night, and an alarm sounds if a machine jams. The scheduler is the clock that starts the press. CI/CD is the quality control process. Monitoring is the night editor’s phone.

Scheduler (cron)

The clock that starts the press at 2 AM

A cron expression is a five-field string — minute, hour, day-of-month, month, day-of-week — that tells a scheduler exactly when to fire. Every cloud platform (Airflow, Kubernetes CronJob, GitHub Actions) uses this same syntax, so learning cron once works everywhere.

CI/CD Pipeline

Quality control before print

Every time you push code, the CI server automatically runs your full test suite. If all tests pass, the CD step deploys the new version to the right environment — no manual copy-pasting, no forgotten steps, and a full git-history audit trail.

Staging Environment

The proof copy checked before the full run

Staging uses the same code, database schema, and configuration as production but with separate data. You validate every change in staging before authorising it to go live — so a bad deploy never corrupts real records.

Monitoring

The night editor’s phone — rings when something breaks

Structured logs give you a timestamped, searchable record of every pipeline run, row count, and error. A health-check endpoint lets your monitoring system ask "did this job succeed?" on a schedule — and alert you the moment a run is late, empty, or failed.

Automation takes a pipeline from “I run it manually” to “it runs itself, and I hear about it only when something goes wrong.”

Key Concepts

Cron syntax, Python scheduling, CI/CD config, deployment workflow, and monitoring.

Cron Syntax

Five fields that define when a job fires

Syntax

A cron expression is a five-field string that tells the scheduler exactly when to run a job. The fields are, in order: minute hour day-of-month month day-of-week. Every cloud scheduler — Airflow, Kubernetes, GitHub Actions — reads this exact format.

Common Cron Patterns

0 2 * * * Every day at 02:00
0 */6 * * * Every 6 hours
0 2 * * 1 Every Monday at 02:00
*/15 * * * * Every 15 minutes
Bash 
# Cron format: minute hour day-of-month month day-of-week
0  2  *  *  *   # Run at 02:00 every night
0  2  *  *  1   # Run at 02:00 on Monday only
*/30 * * * *    # Run every 30 minutes

Always validate your cron expression before deploying. Use an online cron validator — it translates the expression into plain English so you can confirm it fires at the right time. 0 2 * * 1 means Monday at 02:00, not every minute on Monday.

Python Scheduler

Schedule Python functions with APScheduler

Library

APScheduler is a Python library that lets you schedule functions inside your application. It uses the same cron syntax you already know. The BlockingScheduler keeps the process running and fires your function at the scheduled time every day.

APScheduler vs OS Cron

APScheduler OS Cron
Lives in Your Python app The server OS
Separate daemon? ✗ Not needed ✓ Yes, crond
Best for Bundled scheduling logic Server-level tasks
Python 
from apscheduler.schedulers.blocking import BlockingScheduler
import orders_pipeline                       # your pipeline module

scheduler = BlockingScheduler()

@scheduler.scheduled_job("cron", hour=2, minute=0)
def run_nightly():
    orders_pipeline.run()                    # runs every night at 02:00

scheduler.start()                            # keeps the process alive

APScheduler is great for short pipelines on a single server. For longer-running or multi-step workflows, consider a dedicated orchestrator like Apache Airflow. Both use cron syntax, so everything you learned here still applies.

CI/CD YAML

Pipeline-as-code using GitLab CI

Config

A CI/CD YAML file describes your pipeline stages in code stored alongside your source. Every git push triggers the file automatically. The when: manual flag adds a human approval step before production changes go live.

GitLab CI vs GitHub Actions

.gitlab-ci.yml GitLab CI
=
.github/workflows/*.yml GitHub Actions
→
Same concept
Bash 
# .gitlab-ci.yml — stored at the root of your repo
stages: [test, deploy]

test-pipeline:
  stage: test                     # runs on every push
  script:
    - pip install -r requirements.txt
    - pytest tests/ -v            # blocks deploy if any test fails

deploy-production:
  stage: deploy
  when: manual                    # requires a click to promote to prod
  script:
    - python orders_pipeline.py
  variables:
    DB_URL: $PROD_DB_URL          # secret injected by CI/CD settings

Never put secrets in your YAML file. Store database passwords and API keys as CI/CD variables in your platform's settings panel. Reference them in your YAML as $VARIABLE_NAME. The CI runner injects them at runtime — they never appear in your code history.

Deployment Workflow

develop → staging → production

Pattern

A deployment workflow is a fixed sequence of steps that every code change must pass through before reaching production. You push to develop, CI tests run, then the change promotes to staging, and only after sign-off does it go to production. This keeps bad code away from live data.

Deployment Stages

develop Push here — triggers CI tests automatically
staging Runs full pipeline on copy of production data
production Promoted manually after staging sign-off
git revert Rolls back production in under 30 seconds
Bash 
# Step 1: push to develop — CI runs tests automatically
git add . && git commit -m "fix: handle null region"
git push origin develop

# Step 2: after CI passes, merge to main (deploys to staging)
# Step 3: validate on staging, then approve production deploy

# If production breaks — rollback in 30 seconds:
git revert HEAD --no-edit && git push origin main

Fault tolerance patterns for production pipelines

Exponential backoff — when a step fails (e.g., an API returns 503), wait 1 second, then 2, then 4, doubling each time before retrying. This prevents overwhelming a struggling service. Dead letter queue — rows or events that fail processing are routed to a separate table or topic instead of crashing the entire batch. You can inspect and reprocess them later. Circuit breaker — after N consecutive failures, the pipeline stops retrying and raises an alert immediately rather than burning through all retries on a service that is clearly down.

Redundancy and failover

Critical pipelines run on at least two machines or cloud regions so that a hardware failure does not halt data delivery. Data replication keeps copies of processed output in a second storage location. Automatic failover switches traffic to the backup if the primary goes down — cloud schedulers and container orchestrators (like Kubernetes) handle this for you.

Let staging run at least one full scheduled cycle before you promote to production. A silent data error caught in staging is far cheaper to fix than one discovered after a production run has written wrong data to your dashboards. One day of delay saves hours of cleanup.

Monitoring & Alerting

Log runs, check health, alert on failure

Pattern

Monitoring is how you know your pipeline ran, succeeded, and loaded the expected number of rows. You write a status file after every run. A /health endpoint reads that file and returns 200 OK or 503 depending on whether the last run was recent and successful.

Monitoring Checklist

Structured logging timestamp + level + message
Row-count check alert if rows == 0
Exit code sys.exit(0) success   sys.exit(1) failure
Stale check alert if last_run > 25 h ago
Python 
import json
from datetime import datetime

def write_status(status: str, rows: int) -> None:
    with open("pipeline_status.json", "w") as f:
        json.dump({
            "status":      status,                   # "success" or "failure"
            "rows_loaded": rows,                     # alert if this is 0
            "last_run":    datetime.now().isoformat() # used by /health endpoint
        }, f)

Row-count monitoring is just as important as error monitoring. A pipeline that loads zero rows and exits with sys.exit(0) looks healthy to the scheduler but is silently broken. Always alert on unexpectedly low row counts, not only on exceptions.

Code Examples

Hands-on code snippets to explore the concepts

01

Log a Pipeline Step

Beginner Orders datetime print()

What This Does

This script records when a pipeline step starts and finishes using variables and print(). You can see the step name, the row count, and the elapsed time — all in two lines of output.

log_step.py 
from datetime import datetime              # import datetime to track timing
step_name = "load_orders"                 # the name of the pipeline step we are running
start_time = datetime.now()               # record the exact time this step began
print("START:", step_name, start_time.strftime("%H:%M:%S"))  # log the start message
rows_loaded = 320                          # number of rows this step processed
end_time = datetime.now()                 # record when the step finished
elapsed = (end_time - start_time).seconds # calculate how many seconds it took
print("DONE:", step_name, rows_loaded, "rows in", elapsed, "seconds")  # log result
Terminal $ python log_step.py
START: load_orders 09:14:22
DONE: load_orders 320 rows in 0 seconds

Every real pipeline logs the same information — step name, row count, and timestamp — so you can trace exactly what ran and when.

02

Write a Run Log

Beginner Sales datetime File I/O

What This Does

This script appends a pipeline result to a text file each time it runs. Because it uses append mode, every run adds a new line without overwriting the previous ones.

write_run_log.py 
from datetime import datetime                   # import datetime for timestamps
step_name = "load_sales"                       # the name of the pipeline step
rows_loaded = 1420                              # how many rows this step processed
run_time = datetime.now().strftime("%H:%M:%S") # format the current time as text
log_line = f"{run_time} | {step_name} | {rows_loaded} rows | success"  # build the log entry
with open("run_log.txt", "a") as f:           # open the log file in append mode
    f.write(log_line + "\n")                   # write this entry as a new line
print("Logged:", log_line)                     # confirm the entry was saved
Terminal $ python write_run_log.py
Logged: 09:14:25 | load_sales | 1420 rows | success

Append mode keeps a full history of every pipeline run in a single file — check the last few lines to see what ran most recently.

03

Chain Two Steps

Beginner Products Functions Sequential Flow

What This Does

This script defines two functions and calls them in order — the output of the first step goes straight into the second. This is the core pattern of every data pipeline.

pipeline_steps.py 
def extract():                        # step 1: return a list of product names
    return ["Widget A", "Widget B", "Widget C"]
def summarize(products):              # step 2: count how many products loaded
    return len(products)              # return the total as a single number

raw_data = extract()                  # run step 1 - get the raw product list
total = summarize(raw_data)           # run step 2 - count the products
print("Pipeline complete:", total, "products loaded")  # show the final count
Terminal $ python pipeline_steps.py
Pipeline complete: 3 products loaded

Add more functions between extract() and summarize() to add more processing steps to your pipeline.

Chain Two Steps

Your Task

Write two functions: one called get_products that returns a list of three product names, and one called count_items that takes a list and returns how many items are in it. Call get_products first, pass its result straight into count_items, and print the total. Your output should be: Total products loaded: 3.

chain_steps.py 
def get_products():                              # step 1: return a list of product names
    return ["Widget A", "Widget B", "Widget C"]  # three products to load

def count_items(product_list):                   # step 2: count how many items are in the list
    return len(product_list)                     # return the total as a number

product_data = get_products()                    # run step 1 and store the result

print("Total products loaded:", total_items)     # print the final count

Total products loaded: 3

Every real pipeline works exactly like this — the output of one function becomes the input of the next. Add a transform function between the two and you already have a three-step extract-transform-load pipeline.

Common Mistakes

Pitfalls to avoid

Printing the Error Without Saving It to a Log

A print() call inside except disappears in a scheduled job — the failure leaves no trace.

Pitfall

When a cron job catches an exception and only calls print(), the message goes to stdout and is immediately discarded — nothing is written to disk. Write the error to a file using open("pipeline.log", "a") so there is a permanent record every time the job fails.

Wrong — error silently discarded 

try:
    run_pipeline()
except Exception as e:
    print("Error:", e)

Correct — write error to a log file 

try:
    run_pipeline()                          # run the pipeline step
except Exception as error_message:
    with open("pipeline.log", "a") as f:   # open the log in append mode
        f.write("ERROR: " + str(error_message) + "
")  # save the error

Think of this like a doctor keeping patient notes — a conversation that isn't written down never happened. Always write errors to a file so you can look back at what went wrong the next morning.

Catching an Error but Not Exiting with a Failure Code

Without sys.exit(1), the scheduler sees exit code 0 and marks the job as successful even when it failed.

Pitfall

Every program exits with a number — 0 means success, anything else means failure. Catching an exception and not calling sys.exit(1) leaves the exit code at 0, so cron, Airflow, or any other scheduler reports the job as passed. Add sys.exit(1) inside the except block to signal the failure.

Wrong — scheduler reports success 

try:
    run_pipeline()
except Exception as error_message:
    print("Failed:", error_message)
    # exit code stays at 0 — job appears to succeed

Correct — signal failure with exit code 1 

import sys                                  # needed to call sys.exit()
try:
    run_pipeline()                          # run the pipeline step
except Exception as error_message:
    print("Failed:", error_message)         # log the reason
    sys.exit(1)                             # tell the scheduler it failed

Exit codes are how programs talk to their environment — 0 is a green light, 1 is a red light. Without the red light, no alert fires and the broken job runs again tomorrow.

Using a Relative Path That Works Locally but Fails in a Scheduled Job

Cron starts from the system root — a path like "data/orders.csv" resolves to the wrong folder and raises a FileNotFoundError.

Pitfall

Python resolves a relative path like "data/orders.csv" from the current working directory — which is your project folder in a terminal but the system root in cron. Use os.path.dirname(os.path.abspath(__file__)) to get the folder the script lives in, then build every path from there.

Wrong — path breaks under cron 

import pandas as pd
# relative path works in your terminal, fails in cron
orders = pd.read_csv("data/orders.csv")

Correct — path built from script location 

import os, pandas as pd
script_folder = os.path.dirname(os.path.abspath(__file__))  # folder this script lives in
data_path = os.path.join(script_folder, "data", "orders.csv")  # build the full path
orders = pd.read_csv(data_path)                              # always resolves correctly

Think of __file__ as the script's own home address — building every path from it means the script finds its files no matter which directory it is launched from.
