Overview

A high-level summary of the topic

Py

Almost every modern data source — Salesforce, Stripe, Google Analytics, Jira — exposes its data through an API. If you can call APIs from Python, you can build pipelines that pull fresh data from any of them.

Think of an API like a restaurant’s ordering system. The menu (API documentation) lists everything available. You place an order (HTTP request) by telling the waiter what you want. The kitchen processes it and returns your dish (the response) in a standard container (JSON).

Endpoint

A specific menu item — you point to it and get exactly what it describes

A URL that identifies a specific resource — for example, GET /orders returns a list of all orders. In Python, you pass the endpoint URL as a string to requests.get(), optionally followed by params={} for query-string filters like ?page=2&limit=100.

Request

Placing your order — you specify what you want

An HTTP call that specifies a method (GET to read, POST to create), optional query parameters, and authentication headers. In practice you build a request by passing headers={...} for tokens and params={...} for filters, then calling requests.get(url, headers=headers, params=params).

Response

The dish that arrives — structured, predictable format

JSON data returned by the API together with a status code — 200 means success, 404 means not found, 429 means you sent too many requests. Call response.raise_for_status() to turn any error code into a Python exception, then response.json() to get a plain Python dict you can index and iterate.

API Key

Your membership card — proves you are allowed to order

A secret token sent in the Authorization header on every request to prove your identity — for example, {"Authorization": "Bearer YOUR_KEY"}. Never place the key value directly in source code; store it in an environment variable and read it with os.environ.get("API_KEY").

Resilience

The patient diner — waits when the kitchen is busy, never storms out

Real APIs throttle high-volume callers with a 429 Rate Limit Exceeded response and split large datasets across multiple pages. Your pipeline must follow pagination with a while loop until no next page remains, and handle 429 by reading the Retry-After header and waiting before retrying.

An API is just a formalized way to request data from a remote system. Once you understand the pattern, every API works the same way.

Key Takeaways

The most important ideas to remember

APIs Replace Manual Exports

Every time your team downloads a Jira board export or Salesforce report by hand, a Python script calling the API can do the same job automatically on a schedule — no human involvement needed. Because the script runs on a timer, your dashboards always reflect the latest data; you eliminate the entire category of “I forgot to refresh the report” errors.

Automation Scheduling No manual steps

requests Handles HTTP for You

Without the requests library, making an HTTP call requires writing low-level socket code, manual header encoding, and error handling for timeouts. The requests library wraps all of that into a single function call so you focus entirely on parsing the data you received, not on how it traveled over the network.

requests HTTP Python library

JSON Becomes a DataFrame Directly

The list returned by response.json() passes straight into pd.DataFrame() in a single line — the same table structure as an Excel sheet, ready to filter, sort, and export as a CSV without any reformatting. If the API returns nested JSON where each record contains a sub-list, use pd.json_normalize() to automatically flatten those nested fields into extra columns.

response.json() DataFrame pandas

Leaked Keys Live in Git Forever

Deleting a Python file that contained a hard-coded API key does not remove it from your repository — every commit is stored permanently in git history, and anyone who clones the repo can run git log -p to read those deleted lines. The only safe fix is to rotate the key with the API provider immediately and load credentials with os.environ.get() or python-dotenv so they never appear in committed files.

git history os.environ.get() .env

Missing Pages Means Missing Data

If your script requests only the first page of a paginated API and stops, you get at most 100 rows back with no error and no warning — the response looks completely valid. You won’t discover the truncation until a downstream report produces totals that are far too low, by which point the root cause is very hard to trace.

silent failure pagination next_page

Key Concepts

HTTP cycle, status codes, authentication, and resilience patterns.

HTTP Cycle

Request, endpoint, response

Core concept

Every API call follows the same pattern. Your script sends an HTTP request to a URL endpoint, specifying a method like GET and optional parameters. The server processes it and returns a response containing a status code and a JSON body you can parse.

HTTP Methods

Method What it does Python call
GET Read / fetch data requests.get(url)
POST Create a new record requests.post(url, json=data)
PUT Replace a record requests.put(url, json=data)
DELETE Remove a record requests.delete(url)
Python 
import requests

url = "https://api.example.com/orders"    # the endpoint URL
params = {"status": "open", "limit": 50}  # query-string filters
headers = {"Authorization": "Bearer TOKEN"} # auth header

response = requests.get(url, params=params, headers=headers)
response.raise_for_status()  # raises exception for 4xx / 5xx codes
data = response.json()       # converts JSON body to a Python dict

Always call raise_for_status() before .json(). If the server returned a 401 or 500 code, .json() will parse the error body without warning — you'll process bad data silently.

Status Codes

2xx success, 4xx client error, 5xx server error

Reference

Every response includes a 3-digit status code. The first digit tells you where the problem is: 2xx is success, 4xx means your request was wrong, and 5xx means the server failed. Use raise_for_status() to automatically convert any non-2xx code into a Python exception.

Common Codes

Code Meaning Your action
200 OK — success Call .json()
401 Unauthorized Check your API key
404 Not found Verify URL path
429 Rate limited Read Retry-After and wait
500 Server error Retry with backoff
Python 
response = requests.get(url, headers=headers)

# Option A — let the library raise an exception automatically
response.raise_for_status()  # HTTPError raised if code is 4xx or 5xx

# Option B — inspect the code yourself for fine-grained handling
if response.status_code == 429:
    wait = int(response.headers.get("Retry-After", 5))
    time.sleep(wait)           # pause the instructed number of seconds
elif response.status_code == 200:
    data = response.json()     # only parse on confirmed success

Use raise_for_status() as your default. Only switch to manual code-checking when you need to treat a specific code differently — such as pausing on 429 before retrying instead of raising an error immediately.

Authentication

API keys and Bearer tokens

Security

Most APIs require you to prove your identity on every request. You pass an API key — a long secret string — inside the Authorization header as Bearer YOUR_KEY. Load the key from an environment variable so it never appears in committed code.

Auth Approaches

Bearer token in Authorization header preferred
API key as URL query parameter avoid — key visible in server logs
OAuth 2.0 access token enterprise APIs — short-lived, auto-refreshed
Python 
import os, requests

# Load from environment — never write the key directly in your script
api_key = os.environ.get("MY_API_KEY")

# Build the Authorization header in Bearer format
headers = {"Authorization": f"Bearer {api_key}"}

# Pass headers on every request — not just the first one
response = requests.get("https://api.example.com/data", headers=headers)
response.raise_for_status()
data = response.json()

Add .env to your .gitignore file immediately. Deleting a committed key from a file does not remove it from git history — anyone who clones your repo can still read every past commit and find the key.

Resilience

Pagination and rate-limit retries

Patterns

Real APIs restrict two things: how many requests you can send per minute (rate limiting, signalled by a 429 code) and how many records they return per response (pagination, signalled by a next_page field in the JSON). You must handle both in every production pipeline.

Resilience Patterns

Situation Signal Pattern
Rate limited 429 Read Retry-After, sleep, retry
More pages exist next_page while loop until key is None
Network timeout Timeout Set timeout=10, retry with backoff
Python 
import time, requests

all_records = []   # accumulate every page here
page = 1           # start at page 1

while True:
    response = requests.get(url, headers=headers,
                            params={"page": page}, timeout=10)
    if response.status_code == 429:                     # rate limited
        wait = int(response.headers.get("Retry-After", 5))
        time.sleep(wait)                                # wait then retry
        continue
    response.raise_for_status()                         # error on other failures
    payload = response.json()
    all_records.extend(payload["records"])              # add page to results
    print(f"Fetched page {page} ({len(payload['records'])} records)")
    if payload.get("next_page") is None:                # no more pages
        break
    page += 1                                           # advance to next page

print(f"Collected {len(all_records)} total records")

Log each page number inside the while loop. If the script fails midway through 200 pages, you'll know exactly which page to resume from instead of restarting from page 1 and duplicating data.

Secrets Management

Keep credentials out of your code

Security

Secrets management is the practice of storing API keys, database passwords, and tokens outside your source code so they are never accidentally committed to Git or exposed in logs. Environment variables are the simplest approach, but production teams use dedicated secrets managers that handle encryption, rotation, and access control automatically.

Secrets Maturity Ladder

✗ Hardcoded Key written directly in Python file — leaked the moment you push to GitHub.
○ .env file Stored in a local .env file and loaded with python-dotenv. Add .env to .gitignore.
● Env variable Set in CI/CD secrets or server config. Read with os.environ["KEY"].
✓ Secrets manager AWS Secrets Manager, Azure Key Vault, or HashiCorp Vault — encrypted, audited, auto-rotated.
Python 
import os
from dotenv import load_dotenv   # pip install python-dotenv

load_dotenv()                    # reads .env file into os.environ

api_key = os.environ["API_KEY"]  # crashes if missing — fail fast
db_pass = os.environ["DB_PASS"]  # never print these in logs

# Use the key in a request header
headers = {"Authorization": f"Bearer {api_key}"}

Rotate keys regularly and use the narrowest permissions possible. If an API key only needs read access, do not grant write access. If a credential leaks, a narrow scope limits the damage.

HTTP Request/Response Cycle

Beginner

What This Does

Send a GET request to a public test API, check the status code, parse the JSON body, and preview the returned records.

http_cycle.py 
import requests

url    = "https://jsonplaceholder.typicode.com/posts"  # free public test API — no key needed
params = {"userId": 1}                                # query filter: only return user 1's posts

response.raise_for_status()                           # raise HTTPError for any 4xx or 5xx code
posts = response.json()                               # decode JSON body → Python list of dicts

print(f"Status : {response.status_code}")             # 200 = success
print(f"Records: {len(posts)}")                       # how many items came back
print(f"First  : {posts[0]['title'][:45]}")           # preview the title of the first post

02

Authenticating With an API Key

Beginner

What This Does

Load an API key from an environment variable, attach it as a Bearer token in the request header, and call a protected endpoint safely.

auth_request.py 
import os, requests

api_key = os.environ.get("MY_API_KEY")               # read from environment — never hard-code
if not api_key:                                       # fail loudly before making a network call
    raise ValueError("MY_API_KEY is not set in your environment")

headers = {
    "Authorization": f"Bearer {api_key}",            # standard Bearer token format
    "Accept": "application/json",                    # request JSON as the response format
}
response = requests.get(
    "https://api.example.com/reports",                # protected endpoint — rejects requests without auth
    headers=headers,
    timeout=10,                                       # abort if the server doesn't respond in time
)
response.raise_for_status()                           # 401 = bad key  |  403 = key lacks permission

data = response.json()
print(f"Reports available : {len(data['reports'])}")
print(f"Key ends with     : ...{api_key[-4:]}")       # log last 4 chars only — never log the full key
Terminal $ python auth_request.py
Reports available : 42
Key ends with : ...7f3a

Never print or log the full API key — only log the last 4 characters as a sanity check that the correct key loaded. If a key is accidentally written to a log file or stdout, rotate it with the API provider immediately; you cannot "un-leak" it from logs that already exist.

03

Collecting All Pages of API Results

Intermediate

What This Does

Loop page-by-page through a paginated API, accumulate every record, and stop only when the API signals that no further pages exist.

pagination.py 
import os, requests

def collect_all_posts(api_key):
    all_records = []                                      # grow this list across all pages
    page = 1                                              # every paginated API starts at page 1
    while True:                                           # keep looping until the break signal
        response = requests.get(
            "https://api.example.com/posts",
            params={"page": page, "per_page": 50},       # request 50 records per page
            headers={"Authorization": f"Bearer {api_key}"},
            timeout=10,
        )
        response.raise_for_status()                       # stop immediately on any 4xx / 5xx
        payload = response.json()
        all_records.extend(payload["posts"])              # append this page's items to the list
        print(f"  Fetched page {page} — {len(payload['posts'])} records")
        if payload.get("next_page") is None:              # None = no further pages exist
            break
        page += 1                                         # advance to the next page number

    print(f"\nTotal collected: {len(all_records)} records")
    return all_records

posts = collect_all_posts(os.environ.get("MY_API_KEY"))
Terminal $ python pagination.py
Fetched page 1 — 50 records
Fetched page 2 — 50 records
Fetched page 3 — 50 records
Fetched page 4 — 23 records

Total collected: 173 records

Notice that page 4 returned only 23 records. The only safe stopping condition is a None signal from the API itself — never stop because a page returned fewer records than per_page. A partial page in the middle of a dataset would cause you to silently discard everything after it.

04

Retrying After a Rate Limit (429)

Intermediate

What This Does

Detect a 429 rate-limit response, read the Retry-After header to know exactly how long to wait, then automatically retry the same request.

rate_limit.py 
import os, requests, time

def get_with_retry(url, headers, max_retries=5):
    for attempt in range(max_retries):                    # try up to max_retries times
        response = requests.get(url, headers=headers, timeout=10)
        if response.status_code == 429:                   # server is telling us to slow down
            wait = int(response.headers.get(
                "Retry-After", 10))                       # default 10 s if header is absent
            print(f"Rate limited. Waiting {wait}s (attempt {attempt + 1}/{max_retries})...")
            time.sleep(wait)                              # pause for exactly as long as the server asks
            continue                                      # go back to the top of the loop
        response.raise_for_status()                       # raise for any other non-2xx code
        return response.json()                            # success — return the decoded response
    raise RuntimeError(f"Failed after {max_retries} retries")  # all attempts exhausted

data = get_with_retry(
    "https://api.example.com/events",
    headers={"Authorization": f"Bearer {os.environ.get('MY_API_KEY')}"},
)
print(f"Events received: {len(data['events'])}")
Terminal $ python rate_limit.py
Rate limited. Waiting 10s (attempt 1/5)...
Rate limited. Waiting 10s (attempt 2/5)...
Events received: 88

Always read Retry-After instead of using a fixed sleep. A hard-coded 1-second pause gets another 429 if the real limit is 60 seconds — and a fixed 60-second pause wastes a full minute when the actual limit is just 5 seconds.

05

Full Pipeline — API Data into a DataFrame

Advanced

What This Does

Combine authentication, pagination, and rate-limit handling into a single production-ready function that returns a clean, typed pandas DataFrame.

api_to_df.py 
import os, requests, pandas as pd, time

def fetch_all_orders(api_key):
    all_records, page = [], 1
    while True:
        resp = requests.get(
            "https://api.example.com/orders",
            params={"page": page, "per_page": 100},       # 100 records per request
            headers={"Authorization": f"Bearer {api_key}"},
            timeout=10,
        )
        if resp.status_code == 429:                        # rate limited mid-pagination
            time.sleep(int(resp.headers.get("Retry-After", 5)))
            continue                                       # retry the same page — do not increment
        resp.raise_for_status()                            # raise for any other errors
        payload = resp.json()
        all_records.extend(payload["orders"])             # accumulate across all pages
        if payload.get("next_page") is None:               # API says no further pages exist
            break
        page += 1                                          # advance to the next page

    df = pd.json_normalize(all_records)                    # flatten nested JSON keys to columns
    df["created_at"] = pd.to_datetime(df["created_at"])   # parse ISO strings to datetime
    df["amount"]     = df["amount"].astype("float64")     # ensure numeric type for aggregation
    return df

df = fetch_all_orders(os.environ.get("ORDERS_API_KEY"))
print(df.shape)                                            # (rows, columns)
print(df.dtypes)                                           # confirm every column has the right type
print(df.head(2))                                          # preview the first two rows
Terminal $ python api_to_df.py
(347, 8)
order_id int64
customer object
amount float64
created_at datetime64[ns]
status object
dtype: object

order_id customer amount created_at status
0 1001 Acme Corp 254.99 2024-01-03 09:12:00 shipped

pd.json_normalize() is essential here. Without it, nested JSON keys like customer.address.city land as raw Python dict objects in a single column instead of being split into separate, queryable columns that you can filter and aggregate.

Common Mistakes

Pitfalls to avoid

Not Checking Status Before Parsing

Calling response.json() when the API returned an error causes confusing exceptions.

Pitfall

Every API can return error responses — 401, 403, 404, 500, and more. If you call response.json() on an error page, Python raises a JSONDecodeError or returns an unexpected dict. Calling raise_for_status() first converts HTTP errors into clear HTTPError exceptions so you know exactly what went wrong.

Wrong — crashes on error pages 

data = response.json()["orders"]  # crashes on error page

Correct — raises clear HTTP error first 

response.raise_for_status()
data = response.json()["orders"]

Add raise_for_status() as the first line after every API call. It costs one line and saves hours of debugging confusing JSON errors that are actually HTTP errors in disguise.

Stopping at Page 1 of a Paginated API

Treating the first response as the complete dataset silently loses all subsequent records.

Pitfall

Most production APIs return 100–1,000 records per request. If the endpoint gives you 100 orders but there are actually 10,000, a single requests.get() call returns only the first page and no error is raised — you just silently miss 99% of the data. Follow the pagination field in the response until it signals no more pages.

Wrong — only 100 of 10,000 records 

response = requests.get(url, params={"page": 1})
orders = response.json()["orders"]  # only first page

Correct — follow all pages 

page, all_orders = 1, []
while True:
    r = requests.get(url, params={"page": page})
    r.raise_for_status()
    data = r.json()
    all_orders.extend(data["orders"])
    if not data.get("has_next_page"):
        break
    page += 1

Always inspect the API documentation for the pagination field name — common names are has_next_page, next_cursor, and next. When in doubt, print the full response structure on the first request to see all pagination fields.

Hard-Coding API Keys in Source Code

Credentials embedded in scripts are visible in git history indefinitely — even after deletion.

Pitfall

When you push an API key to git, it becomes part of the repository history permanently. Even if you delete the line in the next commit, anyone with access to the repository can recover it. Always read credentials from environment variables using os.environ or os.getenv(), or use a .env file with python-dotenv that is added to .gitignore.

Wrong — key visible in git forever 

API_KEY = "sk_live_abc123..."

Correct — read from environment 

import os
API_KEY = os.environ["MY_API_KEY"]

Create a .env file with your keys and load it with python-dotenv. Add .env to your .gitignore immediately. This pattern keeps credentials off the network entirely — they never leave your local machine.
