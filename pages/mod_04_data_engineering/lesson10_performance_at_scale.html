Overview

A high-level summary of the topic

Py

Optimizing a pipeline without profiling it first is guesswork. Data engineers who profile before optimizing consistently find that 80% of time is spent in one or two steps — and those steps are almost never what they expected.

Think of performance optimization like a Formula 1 pit stop analysis. Before improving anything, the team reviews telemetry data to see exactly where time was lost. Profiling is your telemetry. Once you know where the time goes, you fix the exact bottleneck.

Profiling

Reviewing lap telemetry data

Profiling means measuring how long each step in your pipeline takes using time.perf_counter(), Jupyter's %timeit, or memory_profiler to track RAM. Without it you optimize the wrong step — engineers consistently discover the bottleneck is a database write, not the transform logic they suspected.

Identifying Bottlenecks

Finding which wheel change was slowest

Sorting your profiling results reveals the one or two steps that consume the largest share of total runtime — in a typical ETL pipeline, the database load step is 5–10× slower than the extract or transform steps. Fixing only the biggest bottleneck first gives a far better return than spreading effort evenly across all steps.

Parallel Processing

Multiple mechanics working different wheels simultaneously

Parallel processing spreads CPU-intensive tasks across all available CPU cores using ProcessPoolExecutor from Python's concurrent.futures library. If you have 12 monthly files and 8 cores, you can cut runtime from roughly 12 units to 2 units by processing 8 files at the same time.

Best Practices

Standard procedures that keep every stop under 3 seconds

The best practices checklist is a set of five proven, independent optimizations you apply after profiling: use Parquet instead of CSV, select only the columns you need, downcast float64 to float32, replace iterrows() with vectorized ops, and batch database writes. Applying just the first two typically cuts RAM usage and I/O time by 50–80% on large datasets.

A 10× speedup from profiling the real bottleneck beats a 2× speedup from blindly parallelizing code that wasn’t the problem.

Key Takeaways

The most important ideas to remember

Profiling Reveals What You Cannot Guess

You cannot improve pipeline performance by guessing — 80% of runtime is almost always concentrated in one or two steps, and those steps are rarely the transform logic you suspect. If you skip profiling and optimize the wrong step, you may spend hours saving 0.3 seconds while the real 12-second bottleneck (usually a database write) goes untouched.

%timeit perf_counter memory_profiler

ProcessPoolExecutor Bypasses Python's GIL

Python's Global Interpreter Lock (GIL) prevents threads from executing Python bytecode on multiple cores simultaneously, which means ThreadPoolExecutor delivers no speedup for CPU-bound data work. ProcessPoolExecutor sidesteps the GIL entirely by spawning separate OS processes — each gets its own Python interpreter and a dedicated CPU core.

ProcessPoolExecutor GIL CPU-bound

Dask Is Lazy — Nothing Runs Until .compute()

Dask builds a task graph of your operations but defers all computation until you explicitly call .compute() — this means you can chain .filter(), .groupby(), and .sum() across 200 GB of data and only load the partitions that survive each filter. If you forget to call .compute(), you hold a task graph object, not actual numbers.

Dask DataFrame .compute() Out-of-core

The Five Checklist Items Compound Each Other

Switching from CSV to Parquet with column selection can cut I/O time by 70% and RAM by 80% before your code even runs a single calculation. Downcasting float64 to float32 and replacing iterrows() with vectorized ops then compound on top of that reduced dataset, delivering a combined 5–10× overall speedup from items that each take only one line to apply.

Parquet float32 vectorized chunksize

Key Concepts

Profiling, parallelism, Dask, and performance best practices.

Profiling Tools

Measure before you fix

Definition

Profiling means measuring how long each step in your pipeline takes and how much memory it uses, so you can fix the real bottleneck instead of guessing. Python provides three tools: time.perf_counter() for wall-clock timing, Jupyter's %timeit for repeated-run statistics, and memory_profiler for line-by-line RAM tracking.

Three Profiling Tools

Tool Best For Example
perf_counter() Timing steps in scripts time.perf_counter()
%timeit Comparing two approaches in a notebook %timeit df["col"].sum()
memory_profiler Finding which line spikes RAM memory_usage((fn, (df,)))
Python 
import time                                # built-in — no install needed
t0 = time.perf_counter()                  # record start time in fractional seconds
result = some_function()                   # run the step you want to time
elapsed = time.perf_counter() - t0        # calculate elapsed wall-clock time
print(f"Extract: {elapsed:.3f}s")         # prints e.g. "Extract: 1.234s"

Profile on production-size data. Tiny test files hide I/O bottlenecks — your to_sql() step may take 12s on real data and 0.1s on a 100-row sample, making it invisible to the profiler.

Parallel Processing

concurrent.futures

Definition

Parallel processing runs multiple independent tasks at the same time across all CPU cores using Python's concurrent.futures module. You choose between ProcessPoolExecutor for CPU-bound work (data transforms, file processing) and ThreadPoolExecutor for I/O-bound work (API calls, database queries).

Python 
from concurrent.futures import ProcessPoolExecutor  # CPU-bound parallel executor
def transform(path):                               # each worker runs this function
    df = pd.read_parquet(path)                     # worker reads its own file
    return df.groupby("region")["amount"].sum()    # CPU-heavy aggregation per file
with ProcessPoolExecutor() as ex:                  # auto-scales to all CPU cores
    parts = list(ex.map(transform, files))         # maps each file to a worker

ProcessPool vs ThreadPool

ProcessPool ThreadPool
Use for CPU-bound work I/O-bound work
Bypasses GIL ✓ Yes ✗ No
Startup cost ~100 ms per process Near-zero

Don't use ProcessPool for API calls. Each process costs ~100 ms to start. For I/O tasks that wait on the network, use ThreadPoolExecutor instead — threads share memory and have near-zero startup overhead.

Dask in Brief

Pandas for data larger than RAM

Awareness

Dask is a parallel computing library that gives you a pandas-compatible API for data that is too large to fit in RAM. It splits your dataset into partitions that each fit in memory, builds a task graph of your operations, and only executes when you call .compute(). This lazy evaluation means Dask can optimize the entire pipeline before touching disk.

Key Dask API

Call What It Does Runs Now?
dd.read_parquet() Reads a glob of files as one lazy DataFrame No — lazy
.groupby().sum() Builds task graph — no data moved yet No — lazy
.compute() Triggers execution; returns a pandas object Yes — runs now
Python 
import dask.dataframe as dd                        # pandas-compatible API for large data
ddf = dd.read_parquet("data/*.parquet")            # lazy — nothing loaded into RAM yet
total = (ddf[ddf["amount"] > 100]                  # filter: adds step to task graph
           .groupby("region")["amount"].sum())     # aggregate: still no data moved
result = total.compute()                           # NOW runs; returns a pandas Series

Only call .compute() once. Each call reruns the whole pipeline from disk. Build your full filter, groupby, and select chain first, then trigger execution once at the end — Dask will optimize the whole graph in a single pass.

Best Practices Checklist

Five independent optimizations

Checklist

The performance checklist is a set of five proven, independent optimizations you apply after profiling identifies the bottleneck. Each targets a different source of waste — storage format, memory allocation, compute strategy, or database I/O — and they compound when applied together.

Performance Checklist

Use Parquet + columns= biggest gain cuts I/O and RAM before anything else runs
Downcast float64 → float32 Up to 50% less memory per numeric column
Vectorized ops, not iterrows 50–100× faster on large DataFrames
del large intermediates when done Frees RAM immediately, not on GC
Set chunksize= on DB writes Prevents timeouts on large loads
Python 
df = pd.read_parquet("data.parquet", columns=["id", "amount", "region"])  # 1. only needed columns
df["amount"] = df["amount"].astype("float32")      # 2. half the RAM of float64
df["region"] = df["region"].astype("category")     # 2. ~10x smaller than object dtype
df["tax"] = df["amount"] * 0.1                     # 3. vectorized — 100x faster than iterrows
del raw_df                                         # 4. free RAM immediately; don't wait for GC
df.to_sql("orders", engine, chunksize=10_000)      # 5. batch writes prevent DB timeouts

Start with item one, not the easiest. Parquet + columns= reduce data volume first, so every step below them — dtype casts, transforms, and DB loads — automatically handles a smaller dataset and benefits too.

Database Indexing

Speed up queries by reducing rows scanned

SQL

An index is a data structure the database maintains alongside your table that lets it find matching rows without scanning every single row. Think of it like a book's index — instead of reading every page to find "pandas," you flip to the index, find the page number, and jump straight there. Without an index, a query on a 100-million-row table performs a full table scan; with one, it may touch only a few hundred rows.

Common Index Types

Type How it works Best for
Single-column (B-tree) Sorted tree on one column — database walks the tree to find matching values WHERE clauses on a single column (e.g., WHERE user_id = 42)
Composite B-tree on multiple columns — column order matters (leftmost column used first) Queries that filter on two or more columns together (e.g., WHERE region = 'US' AND date > '2024-01-01')
Bitmap Bit array per distinct value — fast OR/AND operations across low-cardinality columns Columns with few distinct values (e.g., status, region, boolean flags)
SQL 
-- Single-column index: speeds up lookups by user_id
CREATE INDEX idx_orders_user ON orders (user_id);

-- Composite index: covers queries filtering region + date together
CREATE INDEX idx_orders_region_date ON orders (region, order_date);

-- Check how the database will execute a query (execution plan)
EXPLAIN ANALYZE
SELECT * FROM orders WHERE region = 'US' AND order_date > '2024-01-01';
-- Look for "Index Scan" (good) vs "Seq Scan" (full table scan — slow)

When indexes hurt performance

Every index must be updated on every INSERT, UPDATE, and DELETE. A table with ten indexes makes writes ten times more expensive. Only index columns that appear frequently in WHERE, JOIN, and ORDER BY clauses — and drop indexes you no longer use.

Always read the execution plan before adding an index. Run EXPLAIN ANALYZE on your slow query first — the plan shows whether the database is doing a sequential scan. If it is, adding an index on the filtered column usually fixes it. If it already uses an index, the bottleneck is elsewhere.

Code Examples

Hands-on code snippets to explore the concepts

01

Time a Pipeline

Intermediate Sales perf_counter()

What This Does

This script measures how long your load and summary steps take using Python's built-in perf_counter(). Run it to see which stage is the bottleneck before making any changes.

time_pipeline.py 
import time, pandas as pd                       # import the tools we need
t0 = time.perf_counter()                        # record start time before loading
df = pd.read_csv("sales.csv")                   # load all rows from disk
load_time = time.perf_counter() - t0            # seconds taken to load
t0 = time.perf_counter()                        # reset for the next step
totals = df.groupby("region")["amount"].sum()   # summarize sales per region
group_time = time.perf_counter() - t0           # seconds taken to summarize
print(f"Load: {load_time:.2f}s  Group: {group_time:.2f}s")  # show which was slower
Terminal $ python time_pipeline.py
Load: 2.14s Group: 0.08s

If the load step dominates, switching from CSV to Parquet typically cuts read time by 5–10x — no other code changes needed.

02

Run Files in Parallel

Intermediate Reports ProcessPoolExecutor

What This Does

This script uses ProcessPoolExecutor to process three Parquet files at the same time — one per CPU core. Without it, Python handles each file one after another.

parallel_reports.py 
from concurrent.futures import ProcessPoolExecutor  # parallel processing tool
import pandas as pd                                  # for reading Parquet files
def summarize(path):                                 # each worker calls this function
    return pd.read_parquet(path)["amount"].sum()     # read one file, return its total
files = ["jan.parquet", "feb.parquet", "mar.parquet"]  # three monthly report files
with ProcessPoolExecutor() as ex:                    # open one worker per CPU core
    totals = list(ex.map(summarize, files))          # all three files run at once
print(totals)                                        # list of per-file totals
Terminal $ python parallel_reports.py
[12450.75, 9832.50, 14201.00]

Use ProcessPoolExecutor for CPU-heavy work like groupbys — not for API calls, where threads are cheaper and start faster.

03

Measure RAM per Line

Intermediate Inventory memory_profiler

What This Does

This script wraps your load function with memory_usage() to record how much RAM each step consumes. Use it to find exactly which line causes a memory spike.

track_memory.py 
from memory_profiler import memory_usage       # install: pip install memory-profiler
import pandas as pd                            # for loading the data
def load_inventory():                          # wrap your pipeline code in a function
    df = pd.read_csv("inventory.csv")          # load from disk
    df["price"] = df["price"].astype("float32")  # downcast to use half the RAM
    return df["category"].value_counts()       # count items per category
mem = memory_usage((load_inventory, ()))       # run and record RAM at each step
print(f"Peak RAM: {max(mem):.1f} MB")         # print the highest reading
Terminal $ python track_memory.py
Peak RAM: 147.3 MB

If the load step alone spikes RAM, add columns= to your read call — it loads only the fields your pipeline actually uses.

04

Apply the Checklist

Intermediate Orders dtype, chunksize

What This Does

This script applies four optimizations in sequence: column-selective reads, dtype downcasting, vectorized math, and batched database writes. Each line targets a different source of wasted time or memory.

optimized_pipeline.py 
import pandas as pd                                  # pandas for data work
raw = pd.read_parquet("orders.parquet",              # Parquet reads faster than CSV
                      columns=["id","amount","region"])  # only load needed columns
raw["amount"] = raw["amount"].astype("float32")      # halves RAM for this column
raw["region"] = raw["region"].astype("category")     # stores strings as compact codes
raw["tax"] = raw["amount"] * 0.1                     # vectorized — no for-loop needed
raw.to_sql("orders", engine, chunksize=10_000)       # write to DB in batches
print(f"{len(raw):,} rows written")                  # confirm the row count
Terminal $ python optimized_pipeline.py
125,000 rows written

Apply column selection first — it shrinks the data before it enters RAM, so every step below it automatically handles a smaller dataset.

Load Fewer Columns

Beginner Inventory columns=

Your Task

You have a warehouse Parquet file with ten columns, but your report only needs "sku", "quantity", and "category". Write a script that reads only those three columns from warehouse.parquet using pd.read_parquet(). Print the column names and row count to confirm what loaded.

Parquet stores each column separately on disk, so columns you skip with columns= are never read at all — this is why the speed improvement is much larger than with CSV, where the whole file is always scanned.

04

Run Files in Parallel

Intermediate Reports ProcessPoolExecutor

Your Task

You have three monthly Parquet files — jan.parquet, feb.parquet, and mar.parquet — each with a revenue column. Write a function called get_total that reads one file and returns the sum of its revenue column. Use ProcessPoolExecutor to run your function on all three files at the same time, then print the list of results.

parallel_totals.py 
from concurrent.futures import ProcessPoolExecutor  # CPU-parallel executor
import pandas as pd                                  # for reading Parquet files
def get_total(path):                                 # top-level function each worker calls
    return pd.read_parquet(path)["revenue"].sum()    # read one file, return its total
files = ["jan.parquet", "feb.parquet", "mar.parquet"]  # three monthly report files
with ProcessPoolExecutor() as ex:                    # opens one process per CPU core
    totals = list(ex.map(get_total, files))          # all three files processed at once
print(totals)                                        # list of three monthly totals
Terminal $ python parallel_totals.py
[18432.50, 21005.75, 19887.00]

The function you pass to executor.map() must be defined at the top level of the file — not nested inside another function — so that each worker process can import it independently.

Common Mistakes

Pitfalls to avoid

Optimizing Without Profiling First

Fixing code that isn't the bottleneck wastes hours while the real slow step stays untouched.

Pitfall

Guessing which step is slow before measuring it leads to wasted effort — you might spend two hours on a transform that saves 0.3 seconds, while a 12-second load step runs unchanged. Wrap every step with time.perf_counter() first, then fix the step with the highest time.

Wrong — optimize what you assume is slow 

# Rewrote the transform step after assuming it was slow
fast_transform = sales_df.apply(vectorized_fn)
# Load step still takes 12 s — was never measured

Correct — profile first, then fix the bottleneck 

t = time.perf_counter()                       # start the clock
clean_df.to_sql("sales", engine)              # run the load step
load_time = time.perf_counter() - t           # 12.1 s — fix this first

Profiling is like reading a time-sheet before deciding which employee to reassign — without the data you're just guessing. Sort your timings from slowest to fastest and start with the top row every time.

Using Threads for CPU-Bound Processing

Python's Global Interpreter Lock (GIL) prevents threads from running on multiple CPU cores at the same time.

Pitfall

When you run CPU-heavy work with ThreadPoolExecutor, the GIL allows only one thread to execute Python at a time — so you get rapid switching between threads, not true parallel execution. Switch to ProcessPoolExecutor to give each task its own CPU core.

Wrong — threads for CPU work 

from concurrent.futures import ThreadPoolExecutor
with ThreadPoolExecutor(8) as ex:
    results = list(ex.map(crunch_numbers, chunks))
# GIL limits to 1 core — no real speedup

Correct — processes for CPU work 

from concurrent.futures import ProcessPoolExecutor  # bypasses GIL
with ProcessPoolExecutor(8) as ex:                  # one process per core
    results = list(ex.map(crunch_numbers, chunks))  # true parallel execution

Think of the GIL as a single telephone — all threads have to wait for the same line, so adding more threads doesn't speed up the call. Processes each get their own phone, so they truly work at the same time.

Using Processes for I/O-Bound Tasks Like API Calls

Each new process takes around 100 ms to start up, which makes I/O-bound parallel code slower than sequential.

Pitfall

For I/O-bound work the bottleneck is waiting on the network — not using the CPU — so spawning a new process for each API call adds ~100 ms startup overhead per item with no benefit. ThreadPoolExecutor is the right tool here: threads share memory and start in microseconds.

Wrong — processes for I/O work 

from concurrent.futures import ProcessPoolExecutor
with ProcessPoolExecutor() as ex:
    results = list(ex.map(fetch_report, api_urls))
# ~100 ms process startup per URL — often slower

Correct — threads for I/O work 

from concurrent.futures import ThreadPoolExecutor   # lightweight startup
with ThreadPoolExecutor() as ex:                    # threads share memory
    results = list(ex.map(fetch_report, api_urls))  # all network waits overlap

Processes are like hiring a new contractor for each small errand — great for heavy construction, wasteful for quick phone calls. Use ThreadPool whenever the work is mostly waiting, and save ProcessPool for tasks that keep the CPU busy the whole time.
