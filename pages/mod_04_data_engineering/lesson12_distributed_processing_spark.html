This lesson introduces Apache Spark — the industry-standard engine for processing datasets too large for a single machine.

Overview

The big picture before we dive in

Py

Apache Spark is a distributed computing engine that splits a large dataset across many machines so they can all process their slice at the same time.

Think of Spark like a restaurant kitchen during a busy dinner rush: one head chef (the driver) reads the order ticket and decides how to divide the work, then hands each station (the executors) a specific task — one grills the protein, another preps the salad, a third plates the dessert. Every station works in parallel, and the head chef assembles the final plate.

Driver Program

The head chef — reads the ticket and delegates

The driver is the process that runs your main Python script. It builds the execution plan, breaks the work into tasks, and sends those tasks to executors across the cluster.

Executors

The kitchen stations — each processes a portion

Executors are JVM processes running on worker machines. Each executor receives a partition of data and performs the transformations the driver assigned to it.

Lazy Evaluation

Writing the prep list — nothing cooks yet

When you chain transformations like filter() and groupBy(), Spark records them in a plan but does not execute anything until you call an action like show() or write.

In-Memory Processing

A hot stove — keeps pans warm between steps

Unlike Hadoop MapReduce, which writes intermediate results to disk after every step, Spark keeps data in RAM between transformations. This makes iterative algorithms and multi-step pipelines dramatically faster.

Spark is the tool you reach for when a pandas DataFrame no longer fits in memory on a single machine. If your data is under a few gigabytes, pandas or Polars will be simpler and faster — Spark's advantage appears at tens of gigabytes and beyond.

Key Takeaways

The most important ideas from this lesson

Parallelism Turns Hours Into Minutes

Spark splits your dataset into partitions and sends each one to a different executor, so hundreds of machines crunch data at the same time. Without this architecture, a 500 GB file would take hours on a single laptop — with Spark it finishes in minutes.

Driver Executors Partitions

Lazy Evaluation Lets the Optimizer Rewrite Your Plan

Because transformations are recorded but not executed immediately, the Catalyst optimizer can rearrange steps, eliminate redundant work, and push filters earlier — all before a single row is touched. If Spark ran each step eagerly, it could never apply these whole-pipeline optimizations.

Lazy Catalyst DAG

In-Memory Processing Eliminated the Disk Bottleneck

Hadoop MapReduce wrote intermediate data to disk after every step, making iterative workloads agonizingly slow. Spark keeps data in RAM between stages, which is why an ML training loop that took an hour on MapReduce can finish in under a minute on Spark.

In-Memory MapReduce 100× Faster

spark-submit Decouples Code From Infrastructure

You write and test PySpark scripts locally with --master local[*], then deploy the same script to a 100-node YARN or Kubernetes cluster by changing one flag. Without spark-submit, you would need to rewrite deployment logic every time the cluster changes.

spark-submit local[*] YARN / K8s

Key Concepts

Spark architecture, DataFrames, transformations, actions, and spark-submit.

Spark Cluster Architecture

Driver + Cluster Manager + Executors

Core

A Spark application runs as a set of independent processes coordinated by a SparkSession in the driver program. The cluster manager (YARN, Mesos, Kubernetes, or Spark standalone) allocates resources, and executors on worker nodes do the actual data processing.

Component Roles

Component Role
DriverRuns your main() function, creates the execution plan (DAG), and distributes tasks
Cluster ManagerAllocates CPU and RAM to the application (YARN, Kubernetes, Standalone)
ExecutorJVM process on a worker node — runs tasks and stores cached partitions
PartitionA chunk of the dataset — each executor processes one or more partitions in parallel

You write code as if it runs on one machine. Spark handles splitting the data, sending tasks to executors, and collecting results — the parallelism is invisible to your Python script.

Spark DataFrames

Distributed tables with named columns

API

A Spark DataFrame looks like a pandas DataFrame — rows and named columns — but it is distributed across the cluster. You interact with it using the same kinds of operations (select, filter, groupBy), and Spark handles spreading the work across machines.

Python 
from pyspark.sql import SparkSession

spark = SparkSession.builder.appName("demo").getOrCreate()

# Read a CSV into a Spark DataFrame
df = spark.read.csv("sales.csv", header=True, inferSchema=True)

# DataFrame operations look like pandas
df.select("product", "amount").filter(df.amount > 100).show(5)

Prefer DataFrames over RDDs. DataFrames give the Catalyst optimizer enough information to rearrange and speed up your query plan. Raw RDDs skip this optimization and are almost always slower.

Transformations vs Actions

Lazy plans vs immediate execution

Core

Every Spark operation is either a transformation (returns a new DataFrame but does nothing yet) or an action (triggers execution and returns a result). Understanding the difference is essential because it explains why Spark code "does nothing" until you call .show(), .count(), or .write.

Common Examples

Transformations (lazy) Actions (trigger execution)
select(), filter()show(), collect()
groupBy(), join()count(), first()
withColumn(), drop()write.parquet(), toPandas()

Chain transformations freely. Since nothing executes until an action, you can build complex pipelines step by step. Spark's optimizer will collapse redundant steps and push filters as early as possible.

spark-submit

Deploy a PySpark job to a cluster

CLI

spark-submit is the command-line tool that packages your Python script and ships it to the cluster manager. You use flags to control how many executors to request, how much memory each one gets, and which cluster manager to target.

Bash 
# Submit a PySpark job to a YARN cluster
spark-submit \
  --master yarn \
  --deploy-mode cluster \
  --num-executors 4 \
  --executor-memory 8g \
  --executor-cores 4 \
  etl_pipeline.py

Key Flags

--masterCluster manager URL (yarn, k8s://…, local[*])
--num-executorsHow many executor processes to launch
--executor-memoryRAM per executor (e.g. 8g)
--deploy-modeclient (driver on your machine) or cluster (driver on a worker)

Test locally first with --master local[*]. This runs Spark on your laptop using all CPU cores. Once the logic works, switch the flag to yarn or k8s:// for the real cluster.

read_transform.py 
from pyspark.sql import SparkSession
from pyspark.sql.functions import col, upper, year

spark = SparkSession.builder.appName("read-transform").getOrCreate()

# Read CSV with schema inference
df = spark.read.csv("sales.csv", header=True, inferSchema=True)

# Chain transformations — nothing executes yet (lazy)
cleaned = (
    df
    .filter(col("amount") > 0)              # keep only positive sales
    .withColumn("region", upper(col("region")))  # normalize region names
    .withColumn("year", year(col("date")))   # extract year from date
    .select("year", "region", "product", "amount")
)

# Action: triggers the full plan
cleaned.show(5)

Practice Exercises

Reinforce what you learned with hands-on tasks

01

Filter & Select

Beginner Orders filter() select()

Your Task

You have a CSV file called orders.csv with columns order_id, customer, status, and total. Read it into a Spark DataFrame and keep only rows where status equals "completed". Select the three columns order_id, customer, and total, then print the count to confirm.

filter_orders.py 
from pyspark.sql import SparkSession                                # import Spark
spark = SparkSession.builder.appName("filter-orders").getOrCreate()  # start a session
df = spark.read.csv("orders.csv", header=True, inferSchema=True)     # load the CSV file
completed = df.filter(df.status == "completed")                      # keep completed rows only
result = completed.select("order_id", "customer", "total")           # pick three columns
print("Completed orders:", result.count())                           # print the row count
spark.stop()                                                         # shut down Spark
Terminal $ python filter_orders.py
Completed orders: 847

Filtering before selecting columns is good practice. Spark drops unwanted rows first, so less data flows through the rest of your pipeline.

02

Group & Aggregate

Beginner Revenue groupBy() agg()

Your Task

Your manager needs revenue totals per product category. Read orders.csv, group by category, and compute the total revenue and order count for each. Print how many categories the summary contains.

category_revenue.py 
from pyspark.sql import SparkSession                                 # import Spark
from pyspark.sql.functions import sum as spark_sum, count            # import agg functions
spark = SparkSession.builder.appName("category-rev").getOrCreate()   # start a session
df = spark.read.csv("orders.csv", header=True, inferSchema=True)     # load the CSV file
grouped = df.groupBy("category").agg(spark_sum("total").alias("revenue"), count("*").alias("orders"))  # aggregate
print("Categories found:", grouped.count())                          # print the category count
spark.stop()                                                         # shut down Spark
Terminal $ python category_revenue.py
Categories found: 5

Aliasing aggregation results with .alias() gives columns clear names. Without it, Spark generates labels like sum(total) that are hard to use in downstream code.

03

Partitioned Write

Beginner Events partitionBy() write

Your Task

You have a Parquet file called events.parquet with an event_date column. Extract the month number from each date and write the result as Parquet files partitioned by month. Print a confirmation message when the write finishes.

partition_events.py 
from pyspark.sql import SparkSession                                    # import Spark
from pyspark.sql.functions import col, month                             # import helper functions
spark = SparkSession.builder.appName("partition-events").getOrCreate()   # start a session
df = spark.read.parquet("events.parquet")                                # load the Parquet file
df = df.withColumn("month", month(col("event_date")))                    # extract month number
df.write.mode("overwrite").partitionBy("month").parquet("output/events") # write partitioned output
print("Partitioned write complete.")                                     # confirm success
spark.stop()                                                             # shut down Spark
Terminal $ python partition_events.py
Partitioned write complete.

Partitioning on disk mirrors how you query the data. If reports always filter by month, a month-partitioned layout lets Spark skip irrelevant folders entirely.

04

Join & Enrich

Intermediate Customers join() withColumn()

Your Task

Your team stores orders in orders.csv and customer details in customers.csv. Join the two files on customer_id and add a tax column equal to the total times 0.08. Save the enriched dataset as Parquet.

enrich_orders.py 
from pyspark.sql import SparkSession                                         # import Spark
from pyspark.sql.functions import col                                        # import col function
spark = SparkSession.builder.appName("join-enrich").getOrCreate()            # start a session
orders = spark.read.csv("orders.csv", header=True, inferSchema=True)         # load orders
customers = spark.read.csv("customers.csv", header=True, inferSchema=True)   # load customers
enriched = orders.join(customers, on="customer_id").withColumn("tax", col("total") * 0.08)  # join + add tax
enriched.write.mode("overwrite").parquet("output/enriched_orders")           # save to Parquet
spark.stop()                                                                 # shut down Spark
Terminal $ python enrich_orders.py
Enriched rows written: 1024

Joining early and saving the enriched result means downstream jobs read one file instead of two. This avoids repeating the same join in every report.

spark-submit

Intermediate Analytics spark-submit CLI

Your Task

Write a PySpark script that reads events.parquet, keeps only the last 30 days, and counts occurrences by event_type. Save the result as Parquet. Then run the script locally with spark-submit using 4 CPU cores.

recent_events.py 
from pyspark.sql import SparkSession                                      # import Spark
from pyspark.sql.functions import col, current_date, datediff, count       # import helpers
spark = SparkSession.builder.appName("recent-events").getOrCreate()        # start a session
df = spark.read.parquet("events.parquet")                                  # load the events file
recent = df.filter(datediff(current_date(), col("event_date")) <= 30)  # last 30 days only
result = recent.groupBy("event_type").agg(count("*").alias("total"))       # count per type
result.write.mode("overwrite").parquet("output/recent_events")             # save the result
spark.stop()                                                               # shut down Spark
Terminal $ spark-submit --master local[4] recent_events.py
Job completed. Output saved to output/recent_events

The --master local[4] flag is perfect for testing. Once your script works, swap it for yarn or k8s:// to run on a real cluster with no code changes.

Common Mistakes

Pitfalls to watch out for when using Spark

Calling collect() on a Large DataFrame

The driver crashes with an OutOfMemoryError because every row is pulled into one machine.

Pitfall

collect() transfers every row from the cluster into the driver’s RAM. On a 50 GB dataset, the driver runs out of memory and the job dies. Use .show() or .limit() to preview data safely.

Wrong — pulls all rows 

all_rows = df.collect()
for row in all_rows:
    print(row)

Correct — preview only 

df.show(20)                          # display 20 rows
sample = df.limit(100).toPandas()    # small safe subset

Think of collect() like printing every page of a 10,000-page book at once — your printer jams. Preview a few pages first, then decide what you need.

Using Python for-Loops Instead of DataFrame Operations

The job runs on a single core and ignores the entire cluster.

Pitfall

A Python for loop processes rows one at a time on the driver. Spark DataFrame methods run in parallel across all executors. Replace loops with filter(), select(), and withColumn().

Wrong — driver-only loop 

for row in df.collect():
    if row["amount"] > 100:
        process(row)

Correct — parallel filter 

high_value = df.filter(col("amount") > 100)  # runs on all executors
high_value.write.parquet("high_value")        # saves directly

If you catch yourself writing for row in, stop and look for the equivalent DataFrame method. Spark can only parallelize work it knows about.

Forgetting to cache() a Reused DataFrame

Spark re-reads and re-transforms the source data every time an action runs.

Pitfall

Each action triggers a full recomputation from the source. Calling .cache() stores the result in executor memory so it is only computed once. Add it after your last transformation and before the first action.

Wrong — recomputed twice 

cleaned = df.filter(col("valid") == True)
print(cleaned.count())
cleaned.write.parquet("output")

Correct — cached once 

cleaned = df.filter(col("valid") == True)  # filter rows
cleaned.cache()                            # store in memory
print(cleaned.count())                     # computed once
cleaned.write.parquet("output")            # reuses cache

Caching is like saving a spreadsheet after a big edit. Without it, every formula recalculates from scratch each time you open the file.

Using Plain Strings Where col() Is Required

Python compares the string itself, not the column, producing a silent always-true condition or a TypeError.

Pitfall

Writing "amount" > 100 compares the string "amount" to the integer 100 — Python says True and every row passes the filter. Wrap column names with col() so Spark evaluates the actual data.

Wrong — string, not column 

df.filter("amount" > 100)
df.withColumn("tax", "price" * 0.1)

Correct — use col() 

df.filter(col("amount") > 100)          # Column object
df.withColumn("tax", col("price") * 0.1) # arithmetic works

A quick rule: if you are comparing or doing math, always use col("name"). Bare strings only work in methods like select() and groupBy() that expect column names, not expressions.

Writing Thousands of Tiny Output Files

HDFS and downstream tools slow to a crawl when reading thousands of small files.

Pitfall

Spark writes one file per partition. A DataFrame with 200 partitions produces 200 small files. Use .coalesce(N) to merge partitions into fewer, larger files before writing.

Wrong — 200 tiny files 

df.write.parquet("output/events")

Correct — fewer files 

df.coalesce(4).write.parquet("output/events")  # 4 manageable files

Aim for output files between 128 MB and 1 GB each. That balances read speed with parallelism for the next job in your pipeline.

PySpark DataFrame API vs Spark SQL

The same operations written in both styles

Spark lets you express the same logic two ways: with the PySpark DataFrame API (method chaining in Python) or with Spark SQL (standard SQL strings). Both compile to the same execution plan, so performance is identical — pick whichever style your team reads most easily.

PySpark DataFrame API
Spark SQL

Filter rows

Python 
from pyspark.sql.functions import col

result = df.filter(col("status") == "completed")
SQL 
df.createOrReplaceTempView("orders")

result = spark.sql("""
  SELECT * FROM orders
  WHERE status = 'completed'
""")

Group and aggregate

Python 
from pyspark.sql.functions import sum as spark_sum, count

result = (
    df
    .groupBy("category")
    .agg(
        spark_sum("total").alias("revenue"),
        count("*").alias("orders"),
    )
    .orderBy(col("revenue").desc())
)
SQL 
result = spark.sql("""
  SELECT category,
         SUM(total)  AS revenue,
         COUNT(*)    AS orders
  FROM orders
  GROUP BY category
  ORDER BY revenue DESC
""")

Join two DataFrames

Python 
joined = orders_df.join(
    customers_df,
    on="customer_id",
    how="inner",
)
SQL 
result = spark.sql("""
  SELECT o.*, c.name
  FROM orders AS o
  INNER JOIN customers AS c
    ON o.customer_id = c.customer_id
""")

Add a computed column

Python 
result = df.withColumn(
    "tax", col("total") * 0.08
)
SQL 
result = spark.sql("""
  SELECT *, total * 0.08 AS tax
  FROM orders
""")

Write results to Parquet

Python 
result.write.mode("overwrite").parquet(
    "output/clean_orders"
)
SQL 
-- SQL result is a DataFrame — write it
result.write.mode("overwrite").parquet(
    "output/clean_orders"
)

Both styles compile to the same Catalyst execution plan, so there is no performance difference. Use the DataFrame API when you need complex Python logic in the pipeline, and Spark SQL when your team already knows SQL or when you want to share queries with analysts.
