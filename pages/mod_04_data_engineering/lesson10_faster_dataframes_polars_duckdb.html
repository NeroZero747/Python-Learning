Overview

A high-level summary of the topic

Py

Pandas is the analyst’s trusted workhorse — but it was designed for single-threaded operation. When datasets grow to tens of millions of rows, pandas gets slow. Polars and DuckDB were built to use every core your machine has.

Think of processing data like checking out at a supermarket. Pandas is like a single fast cashier. Polars is like a bank of self-checkout machines running simultaneously. DuckDB is like the back-office system that already knows every price — you hand it a query and get the answer without moving stock.

Pandas

One cashier — reliable, familiar, handles the job

Pandas processes every operation on a single CPU core, which means adding extra memory or cores to your machine does not make it faster. It is the right tool for datasets under roughly one million rows, where its large ecosystem of integrations and familiar syntax outweigh the speed difference.

Polars

Bank of self-checkouts — all lanes running at once

Polars is a DataFrame library written in Rust that automatically uses every CPU core on your machine for operations like filtering, grouping, and joining. Its lazy API also builds a query plan and optimizes it before touching any data, which means it often reads far less of your file than you might expect.

DuckDB

Back-office system — answers SQL queries without moving stock

DuckDB is an embedded analytical database that runs inside your Python process — no installation of a database server is required. You can use standard SQL to query Parquet files, CSV files, or even live pandas DataFrames, and the results come back as a DataFrame without loading the entire file into memory first.

Choose the Right Tool

The express lane — pick the checkout for your basket size

No single library wins every situation, and the right choice depends on dataset size and whether your team thinks in code or SQL. A quick rule of thumb: use pandas for everyday analysis on smaller files, Polars when something in pandas feels slow, and DuckDB when you want to explore a large Parquet file without writing a single line of pandas.

You don’t need to replace pandas everywhere. Use Polars for DataFrames with millions of rows. Use DuckDB when you want to query large files with SQL.

Key Takeaways

The most important ideas to remember

Switching to Polars Instantly Speeds Up Any Large Query

Pandas uses a single CPU core at a time, so adding more cores to your machine gives it zero benefit. Polars automatically fans out work across every available core, so on a 4-core laptop a 10-million-row groupby that takes 18 seconds in pandas finishes in under 2 seconds in Polars.

Multi-threaded Single-core pandas 5–20× faster

Polars Never Reads Data Until You Call .collect()

If you call pl.read_csv() instead of pl.scan_csv(), the entire file loads into memory immediately — you lose all query optimization benefits. Switching to the lazy API means Polars can drop unused columns and push your filters to the file scan, sometimes cutting memory use by 80 % or more.

scan_csv() .collect() Query plan

DuckDB Lets You Write SQL Against Files Without a Server

Traditional databases like PostgreSQL or SQL Server require a running server process and a connection string to query your data. DuckDB is embedded inside your Python script like a library — you can run a full SQL SELECT against a Parquet or CSV file with just two lines of code and get back a pandas DataFrame.

Embedded DB SQL on files No server needed

Tool Choice Depends on Data Size and Whether You Think in SQL

Using DuckDB on a 5,000-row file adds unnecessary overhead compared to a simple pd.read_csv(), and converting a team’s pandas scripts to Polars for a 20,000-row report gives no measurable gain. The practical rule: use pandas under ~1 million rows where team familiarity matters, Polars when DataFrames are large and speed counts, and DuckDB when you want SQL syntax directly against files without loading them fully into memory first.

1M+ rows → Polars SQL mindset → DuckDB Small data → pandas

Key Concepts

Polars syntax, lazy evaluation, DuckDB SQL, and choosing the right tool.

Polars Syntax

Method chains and pl.col() expressions

Python API

Polars uses a method-chain style where you connect steps with dots. Column references always use pl.col("name") — never the df["name"] subscript style from pandas. The lazy API reads no data until you call .collect().

Pandas vs Polars Syntax

pandas Polars
Read CSV pd.read_csv(f) pl.scan_csv(f)
Filter rows df[df["x"]>5] .filter(pl.col("x")>5)
Column ref df["col"] pl.col("col")
Run query Runs automatically .collect()
Python 
import polars as pl

result = (
    pl.scan_csv("sales.csv")          # lazy read â no data loaded yet
      .filter(pl.col("amount") > 100) # keep rows where amount > 100
      .group_by("region")             # group rows by the region column
      .agg(pl.col("amount").sum())    # sum the amount column per group
      .collect()                      # run the plan and return a DataFrame
)

Switch df["col"] to pl.col("col") when migrating from pandas. Inside .filter(), .select(), and .agg(), column names must always be wrapped in pl.col().

Lazy vs Eager

How Polars decides when to read data

Execution Model

Polars has two execution modes. Eager mode (pl.read_csv) loads all data immediately. Lazy mode (pl.scan_csv) builds a query plan and waits. Polars then optimizes the plan and reads only the rows and columns your query actually needs.

Eager vs Lazy

Eager Lazy
Load with pl.read_csv() pl.scan_csv()
Reads data Immediately Only at .collect()
Optimized ✗ No ✓ Yes
Best for Small files, exploration Large files, production
Python 
import polars as pl

# Eager â loads the entire file right away
df = pl.read_csv("sales.csv")            # all data in memory immediately
filtered = df.filter(pl.col("x") > 100) # filters the full loaded file

# Lazy â builds a plan, reads only what is needed
result = (
    pl.scan_csv("sales.csv")             # no data read yet
      .filter(pl.col("x") > 100)         # added to the query plan
      .collect()                         # Polars reads and filters in one pass
)

Call .collect() only once, at the very end of your chain. Calling it inside a loop reruns the full plan every iteration and erases all the performance benefits of lazy evaluation.

DuckDB Basics

SQL directly on files and DataFrames

SQL Engine

DuckDB runs SQL queries inside your Python process â no server required. duckdb.connect() opens a session. You pass a SQL string to .execute() and call .fetchdf() to get a pandas DataFrame back.

DuckDB FROM Targets

CSV file on disk FROM 'file.csv' ✓
Parquet file on disk FROM 'data.parquet' ✓
Folder of Parquet files FROM 'folder/*.parquet' ✓
Live pandas variable FROM df ✓ (variable name)
Python 
import duckdb

con = duckdb.connect()               # no URL or password needed
result = con.execute(
    """
    SELECT region, SUM(amount) AS total
    FROM 'sales.csv'                 -- DuckDB reads the file directly
    WHERE amount > 100
    GROUP BY region
    ORDER BY total DESC
    """
).fetchdf()                          # returns a pandas DataFrame

Use FROM df to query a live pandas variable. Write the Python variable name directly in your SQL â no registration step is needed. DuckDB finds all variables in scope automatically.

When to Use Each

Match your task to the right tool

Decision Guide

All three tools can load and transform the same data. Use pandas for small datasets and quick scripts. Use Polars when a pandas query runs slowly. Use DuckDB when your team prefers writing SQL over Python method chains.

Polars vs DuckDB

Polars DuckDB
API style pl.col() chains SQL strings
Returns Polars DataFrame pandas via .fetchdf()
Multi-core ✓ Yes ✓ Yes
Best when Building Python pipelines Team knows SQL
Python 
import polars as pl, duckdb

# Step 1 â Polars handles the heavy aggregation
summary = (
    pl.scan_csv("sales.csv")          # lazy read â no data loaded yet
      .group_by("region")             # group by the region column
      .agg(pl.col("amount").sum())    # sum per group
      .collect()                      # run the plan
      .to_pandas()                    # convert to pandas for DuckDB
)

# Step 2 â DuckDB queries the Polars result with SQL
top5 = duckdb.sql(
    "SELECT * FROM summary ORDER BY amount DESC LIMIT 5"
).fetchdf()

Start with pandas. If a query is slow, try Polars. If your team prefers SQL, use DuckDB. Convert between them with .to_pandas() (Polars → pandas) and .fetchdf() (DuckDB → pandas).

Polars Lazy Filter

Beginner Orders scan_csv() .filter()

What This Does

This script reads a CSV file lazily using Polars, then filters rows where the order total exceeds 500. No data is loaded into memory until you call .collect().

polars_filter.py 
import polars as pl                               # import the Polars library

lf = pl.scan_csv("orders.csv")                   # lazy read — no data loaded yet
result = (
    lf.filter(pl.col("total") > 500)             # keep rows where total exceeds 500
      .collect()                                 # run the plan and load matching rows
)
print(result.shape)                             # print (rows, columns) of the result
Terminal $ python polars_filter.py
(89, 4)

Laziness means Polars only reads the rows that pass your filter — a 1 GB file where 5% of rows match loads just 50 MB of data.

02

Polars Group & Sum

Beginner Sales .group_by() .agg()

What This Does

This script groups sales rows by category and sums the revenue for each group. Polars builds the full plan lazily and runs everything in one optimized pass.

polars_groupby.py 
import polars as pl                               # import the Polars library
result = (
    pl.scan_csv("sales.csv")                     # lazy read — no data yet
      .group_by("category")                      # group rows by their category
      .agg(pl.col("revenue").sum())              # sum the revenue column per group
      .collect()                                 # execute the plan and return a DataFrame

print(result.shape)                             # print (rows, columns) of the result

(5, 2)

Call .collect() once at the end of your chain — calling it inside a loop replays the full file scan every iteration.

DuckDB SQL on CSV

Beginner Customers duckdb.connect() .fetchdf()

This script runs a SQL query on a CSV file using DuckDB — no database server needed. It counts customers per region and returns the result as a pandas DataFrame.

duckdb_query.py

con = duckdb.connect()                           # open an in-memory DuckDB session

Choose the Right Tool

Beginner Sales .to_pandas() .fetchdf()

What This Does

This script uses Polars to do the heavy filtering on a large file, then converts the result to a pandas DataFrame for charting or export — combining the best of both libraries in one workflow.

tool_handoff.py 
import polars as pl                              # Polars for large-file filtering

result = (
    pl.scan_csv("sales.csv")                    # lazy read — file not loaded yet
      .filter(pl.col("region") == "Europe")     # push filter to the file scan
      .collect()                                # only European rows enter memory

pandas_df = result.to_pandas()                  # hand the result off to pandas

Terminal $ python tool_handoff.py

Use Polars for the heavy lifting (filtering, grouping, joining), then call .to_pandas() at the very end when you need a library — such as matplotlib or openpyxl — that only accepts a pandas DataFrame.

Polars vs DuckDB — Side by Side

The same data operations written in both Polars and DuckDB SQL

Polars and DuckDB can perform the same operations on your data — they just express those operations differently. Use this reference to translate between the two styles whenever you need to switch tools or work with a team that prefers SQL.

Polars (Python)
DuckDB (SQL)

Load data from a file

Python 
import polars as pl

# Lazy — file is NOT loaded into memory yet
lf = pl.scan_csv("orders.csv")
# Execute when ready
df = lf.collect()
SQL 
-- DuckDB reads the file lazily too
SELECT *
FROM 'orders.csv'
LIMIT 10;

Filter rows

Python 
result = (
    pl.scan_csv("orders.csv")
    .filter(pl.col("status") == "shipped")
    .collect()
)
SQL 
SELECT *
FROM 'orders.csv'
WHERE status = 'shipped';

Group and aggregate

Python 
result = (
    pl.scan_csv("orders.csv")
    .group_by("region")
    .agg(pl.col("revenue").sum().alias("total"))
    .sort("total", descending=True)
    .collect()
)
SQL 
SELECT region, SUM(revenue) AS total
FROM 'orders.csv'
GROUP BY region
ORDER BY total DESC;

Select specific columns

Python 
result = (
    pl.scan_csv("orders.csv")
    .select(["order_id", "region", "revenue"])
    .collect()
)
SQL 
SELECT order_id, region, revenue
FROM 'orders.csv';

Join two tables

Python 
orders = pl.scan_csv("orders.csv")
customers = pl.scan_csv("customers.csv")

result = (
    orders.join(customers, on="customer_id")
    .collect()
)
SQL 
SELECT o.*, c.name, c.email
FROM 'orders.csv' AS o
JOIN 'customers.csv' AS c
  ON o.customer_id = c.customer_id;

Both tools achieve the same final result — the choice is about your team's preference (Python vs SQL) and your data size. Polars is better when you need to build complex Python pipelines; DuckDB is better when your whole team speaks SQL fluently.

Practice Exercises

Hands-on tasks to reinforce the concepts

01

Polars Filter & Aggregate — Lazy Mode

Tasks

  1. Import polars as pl at the top of your script.
  2. Call pl.scan_csv("orders.csv") — do not use pl.read_csv() here.
  3. Chain .filter(pl.col("status") == "shipped") to keep only shipped orders.
  4. Chain .group_by("product_category").agg(...) to sum revenue and count order IDs.
  5. Sort by revenue descending, call .collect(), and print the result.
exercise_01.py 
import polars as pl

result = (
    pl.scan_csv("orders.csv")             # lazy — no data read yet
      .filter(pl.col("status") == "shipped")  # keep only shipped rows
      .group_by("product_category")       # group by category
      .agg([
          pl.col("revenue").sum().alias("total_revenue"),
          pl.col("order_id").count().alias("order_count")
      ])
      .sort("total_revenue", descending=True)  # highest revenue first
      .collect()                           # now execute the full plan
)
print(result)

Why this matters: This pattern — scan, filter, aggregate, sort, collect — is the Polars equivalent of a filtered SQL GROUP BY query, and it is the most common pipeline you will write on large sales or log datasets.

Benchmark pandas vs Polars

Tasks

  1. Import pandas as pd, polars as pl, and time.
  2. Write a pandas version: read large_data.csv, filter score > 75, mean of value grouped by category. Wrap it in time.perf_counter() calls.
  3. Write the same query in Polars lazy mode: scan_csv() → .filter() → .group_by().agg() → .collect(). Time that too.
  4. Compute the speedup as pd_time / pl_time and print both times plus the ratio.
exercise_03.py 
import pandas as pd, polars as pl, time

# ----- pandas version -----
t0 = time.perf_counter()
df = pd.read_csv("large_data.csv")
result_pd = (
    df[df["score"] > 75]
      .groupby("category")["value"]
      .mean()
)
pd_time = time.perf_counter() - t0

# ----- polars version -----
t0 = time.perf_counter()
result_pl = (
    pl.scan_csv("large_data.csv")
      .filter(pl.col("score") > 75)
      .group_by("category")
      .agg(pl.col("value").mean().alias("mean_value"))
      .collect()
)
pl_time = time.perf_counter() - t0

print(f"Pandas : {pd_time:.2f}s")
print(f"Polars : {pl_time:.2f}s")
print(f"Speedup: {pd_time / pl_time:.1f}x")

Why this matters: Running this benchmark on your own machine makes the performance difference tangible and gives you a data-backed argument for adopting Polars the next time a pandas pipeline starts slowing down a daily report.

Common Mistakes

Pitfalls to avoid

Mistake 1: Loading a Large File Eagerly with pl.read_csv()

When you call pl.read_csv() on a large file, Polars loads the entire file into memory immediately before applying any filters — you pay the full memory cost even if your query only needs 5 % of the rows. Switching to pl.scan_csv() gives Polars the chance to push your filter conditions down to the file read, which is dramatically faster and uses far less memory.

✗ Wrong — reads entire file

df = pl.read_csv("huge.csv")  # all rows loaded
result = df.filter(pl.col("x") > 100)

✓ Correct — lazy, push-down filter

result = (
    pl.scan_csv("huge.csv")   # no data read yet
      .filter(pl.col("x") > 100)
      .collect()               # read only matching rows
)

Use scan_csv() as your default for any file that could grow beyond a few hundred thousand rows — you can always drop back to read_csv() for quick interactive exploration of a small file.

Mistake 2: Calling .collect() Multiple Times on the Same LazyFrame

Every call to .collect() re-executes the entire query plan from scratch — if you call it three times, Polars reads and processes the source data three times. The correct pattern is to call .collect() exactly once, assign the result to a variable, and then work with that DataFrame for all subsequent operations.

✗ Wrong — query runs 2×

lf = pl.scan_csv("data.csv")
print(lf.collect().shape)  # run 1
print(lf.collect().head())  # run 2

✓ Correct — collect once, reuse

df = lf.collect()   # single execution
print(df.shape)
print(df.head())

Treat .collect() like a database query — run it once at the very end of your chain, assign the result to a variable, and reuse that variable for all subsequent steps.

Mistake 3: Treating a LazyFrame Like a DataFrame Before .collect()

A LazyFrame is a query plan, not a table of data — it has no rows to display, no length, and no values until you call .collect(). Trying to print it, index into it with df[0], or call len() on it either raises an error or returns the schema rather than the data you expected.

✗ Wrong — printing a plan

lf = pl.scan_csv("data.csv")
print(lf)   # shows schema / plan, not rows
print(len(lf))  # AttributeError

✓ Correct — materialize first

df = lf.collect()  # LazyFrame → DataFrame
print(df)          # shows actual rows
print(len(df))     # correct row count

Call .collect() as soon as you need to see, count, or index into your data — every step you add before it is just extending the plan, not producing values you can work with.

Mistake 4: Using Pandas Bracket Indexing df["col"] Inside Polars Expressions

This is the most common mistake for analysts coming from pandas: inside Polars method chains such as .filter() and .select(), column references must be written as pl.col("name") — using string names or bracket indexing either raises a TypeError or silently applies the wrong operation.

✗ Wrong — pandas syntax in Polars

df = pl.read_csv("sales.csv")
# TypeError — "amount" is not a valid Polars expr
result = df.filter(df["amount"] > 100)

✓ Correct — pl.col() expression

df = pl.read_csv("sales.csv")
result = df.filter(pl.col("amount") > 100)

When you see a TypeError inside a Polars method chain, look for any column reference not wrapped in pl.col() — that is almost always the cause.

Mistake 5: Reaching for DuckDB When pandas Would Do Fine

DuckDB adds real value when you need to run SQL on large Parquet files or bridge a SQL-familiar team into Python workflows, but for a 10,000-row CSV that you process once a day, the extra setup cost and unfamiliarity in code reviews is pure overhead. Using pandas for small, routine files keeps the code simpler and the script easier for the whole team to maintain.

✗ Overly complex for tiny data

con = duckdb.connect()
# unnecessary overhead for 10k rows
result = con.execute(
    "SELECT SUM(x) FROM 'tiny.csv'"
).fetchdf()

✓ Simple pandas is better here

result = pd.read_csv("tiny.csv")["x"].sum()
# one line, zero setup overhead

Reach for DuckDB when your file has millions of rows, when your team thinks in SQL, or when you need to join multiple Parquet files without loading any of them into memory first.
