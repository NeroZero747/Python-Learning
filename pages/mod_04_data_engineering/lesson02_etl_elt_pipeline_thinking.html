Overview

A high-level summary of the topic

Py

A data pipeline is the automated replacement for the copy-paste-clean routine that every analyst does manually — except it runs on a schedule, handles any data volume, and never forgets a step.

Think of an ETL pipeline like a car assembly line. Each station performs one specific job in a fixed sequence: the frame arrives, then the engine is fitted, then the bodywork, then the inspection. A data pipeline works the same way — raw data arrives, passes through each defined stage, and emerges as a clean, structured result ready to use.

ETL vs ELT

Two assembly strategies — build-to-spec vs warehouse-first

ETL is the build-to-spec line — every part is inspected and shaped before it moves on, so only approved components reach the end of the line. ELT is the ship-first strategy — raw parts are delivered straight to the factory floor, and specialist workers configure them to order using the full power of the machinery already there.

Design Principles

The quality manual — rules every station follows every shift

A production pipeline must be idempotent — re-running it with the same input always produces the same output — so automated overnight jobs can safely recover from crashes. It must also be modular and observable so you can identify exactly which stage failed without reading thousands of log lines.

Batch vs Streaming

Shift schedule — nightly run vs 24-hour continuous floor

Batch pipelines run on a fixed schedule, processing all records since the last run in one large job — the right choice for most analytics workloads. Streaming pipelines process events continuously as they arrive, which is only necessary for real-time use cases like fraud detection or live dashboards.

Modular Functions

Interchangeable stations — each tested and swapped alone

Splitting the extract, transform, and load steps into separate Python functions means each can be tested, debugged, and replaced without touching the others. When a pipeline fails at 2 AM, modular code points you to the broken function in seconds rather than requiring you to trace through one long monolithic script.

Whether you choose ETL or ELT depends on one question: is it cheaper to clean data before or after you store it? Modern cloud warehouses have made “after” increasingly practical.

Key Takeaways

The most important ideas to remember

ETL Locks in Your Schema Before You Know What Analysts Will Query

When you transform data before loading it, every downstream query depends on the decisions you made at load time — so if an analyst later needs a column you dropped, the pipeline must be redesigned and re-run from scratch. ELT avoids this by keeping raw data in the warehouse, letting analysts reshape it themselves with SQL or dbt without waiting for an engineer.

Schema Re-run Risk Flexibility

Non-Idempotent Pipelines Double Records on Every Re-run

A pipeline that uses if_exists='append' without deduplication doubles every row each time it runs — a nightly job that crashes and auto-restarts will silently fill your table with duplicates by morning. Switching to if_exists='replace' or an upsert keyed on a unique column makes the pipeline safe to run as many times as needed.

Idempotency Upsert Safety

Batch Is the Right Starting Point — Streaming Only When You Need Sub-Minute Freshness

Most reporting and analytics needs only last night's data, so adding a streaming architecture before you need sub-minute freshness multiplies infrastructure cost and operational complexity for no business gain. Start with a scheduled batch job and upgrade to streaming only when a stakeholder's use case genuinely requires data in under a minute.

Batch Scheduling Streaming

One Big Function Means You Must Re-run Everything to Fix One Step

When extract, transform, and load share the same function, a failure at step 3 forces you to re-extract all data even if the source data didn't change — wasting time, network bandwidth, and sometimes API rate limits. Separate functions let you resume from the failed step and run unit tests against each stage independently.

Key Concepts

ETL vs ELT, design principles, batch vs streaming, and modular code.

ETL vs ELT

Two pipeline architectures

Core

ETL (Extract-Transform-Load) cleans data before storing it. ELT (Extract-Load-Transform) stores raw data first, then transforms it inside the warehouse. ETL keeps your destination clean. ELT lets you re-run transforms when rules change.

ETL vs ELT

ETL ELT
Transform step Before load (Python) After load (SQL / dbt)
Raw data kept? ✗ No ✓ Yes
Re-run transforms Must re-extract Just re-query
Best for Strict schemas, PII Cloud warehouses
Python 
# ETL — transform BEFORE load
raw   = extract("sales.csv")       # pull from source
clean = transform(raw)             # Python cleans the data
load(clean, "warehouse_table")     # only clean rows arrive

# ELT — transform AFTER load
raw = extract("sales.csv")         # pull from source
load_raw(raw, "raw_table")         # raw rows go straight in
transform_in_warehouse()           # SQL cleans inside the warehouse

ELT is the default for cloud warehouses. Snowflake and BigQuery transform billions of rows faster than Python can. Keeping raw data lets you re-run transforms when business rules change.

Design Principles

Five rules for production pipelines

Principles

Production-ready pipelines follow five design rules. Break any one and your pipeline may work locally but fail silently in production.

Production Checklist

Idempotency Safe to re-run after a crash
Modularity Each function is testable alone
Observability You know exactly where a failure occurred
Fail-fast Bad data is caught before it spreads
Backfill-ready Can re-process old data on demand
Python 
def load(df, table, engine):
    # "replace" makes this idempotent — re-runs won't duplicate rows
    df.to_sql(table, engine, if_exists="replace", index=False)
    # observability — always log how many rows you wrote
    print(f"Loaded {len(df):,} rows into '{table}'")

Log the row count at every step. Print counts after extract, transform, and load. A sudden drop between steps is the fastest way to spot silent data loss.

Batch vs Streaming

Two execution models

Patterns

Batch pipelines run on a schedule and process all new records in one job. Streaming pipelines process events the instant they arrive. Most teams start with batch because it is simpler to build and debug.

Batch vs Streaming

Batch Streaming
Trigger Schedule (cron) New event arrives
Latency Minutes to hours Milliseconds to seconds
Complexity Low High (Kafka, Flink)

Python 
import schedule, time

def run_batch():
    raw   = extract("sales.csv")        # pull latest records
    clean = transform(raw)              # clean and reshape
    load(clean, "clean_sales", engine)  # write to warehouse

# run the pipeline every day at 2 AM
schedule.every().day.at("02:00").do(run_batch)

Start with batch. If stakeholders say "daily is fine," a nightly job is far simpler than a 24/7 streaming pipeline. Add streaming only when latency matters.

Modular Code

One function per pipeline stage

Best Practice

A modular pipeline splits each stage into its own function. You write one function for extract, one for transform, and one for load. Each function can be tested and debugged on its own.

Function Roles

extract() Reads data from the source
transform() Cleans and reshapes the data
load() Writes results to the destination
run() Calls all three in order
Python 
def extract(path):
    return pd.read_csv(path)            # reads the source file

def transform(df):
    df = df.dropna(subset=["order_id"]) # drops incomplete rows
    df["total"] = df["qty"] * df["price"]
    return df                           # returns clean data

def load(df, engine):
    df.to_sql("orders", engine, if_exists="replace", index=False)
    print(f"Loaded {len(df):,} rows")   # confirms the write

def run():
    engine = create_engine("sqlite:///dw.db")
    load(transform(extract("raw.csv")), engine)

Test each function on its own. Call transform(sample_df) in a notebook first. Verify your logic before running the full pipeline on millions of rows.

Example 1 — Complete ETL Pipeline

Beginner

What This Does

A modular ETL pipeline with separate extract, transform, and load functions.

etl_pipeline.py 
import pandas as pd
import sqlalchemy

def extract(filepath: str) -> pd.DataFrame:
    return pd.read_csv(filepath)          # reads a CSV and wraps every row in a DataFrame

def transform(df: pd.DataFrame) -> pd.DataFrame:
    df = df.dropna(subset=["sale_date", "amount"])    # remove rows missing key fields
    df["sale_date"] = pd.to_datetime(df["sale_date"])  # parse string → datetime object
    df["amount"]    = df["amount"].abs()               # force all amounts to positive
    df = df[df["amount"] > 0]                          # filter out zero-value rows
    df["year_month"] = df["sale_date"].dt.to_period("M")  # add a YYYY-MM grouping column
    return df

def load(df: pd.DataFrame, table: str, engine) -> None:
    # if_exists="replace" makes this idempotent — re-runs never produce duplicate rows
    df.to_sql(table, con=engine, if_exists="replace", index=False)
    print(f"Loaded {len(df):,} rows into '{table}'.")  # confirm the write succeeded

def run():
    engine = sqlalchemy.create_engine("sqlite:///warehouse.db")  # connect to local warehouse
    raw    = extract("sales_raw.csv")   # step 1: pull raw data from source
    clean  = transform(raw)             # step 2: clean and reshape
    load(clean, "clean_sales", engine)  # step 3: write to warehouse

if __name__ == "__main__":
    run()
Terminal $ python etl_pipeline.py
Loaded 14,832 rows into 'clean_sales'.

Notice the output prints the exact row count after loading — you can see at a glance that 14,832 rows survived the transform. Every production pipeline should print the count at extract, after transform, and after load; a sudden drop between steps is the fastest way to catch silent data loss before it reaches your analysts.

Example 2 — Idempotent Upsert

Intermediate

What This Does

Prove idempotency by calling the same load function twice and observing that the row count stays the same on the second run — not doubled.

idempotent_load.py 
import pandas as pd
from sqlalchemy import create_engine, text

def count_rows(table: str, engine) -> int:
    """Return current row count — 0 if the table doesn't exist yet."""
    try:
        with engine.connect() as conn:
            return conn.execute(text(f"SELECT COUNT(*) FROM {table}")).scalar()
    except Exception:
        return 0                           # table hasn't been created yet

def load_idempotent(df: pd.DataFrame, table: str, engine) -> None:
    before = count_rows(table, engine)     # snapshot row count before writing
    # if_exists="replace" wipes the old table and writes fresh — never appends
    df.to_sql(table, con=engine, if_exists="replace", index=False)
    after = count_rows(table, engine)      # snapshot after writing to confirm
    print(f"'{table}': {before:,} → {after:,} rows")  # log the before/after change

engine = create_engine("sqlite:///warehouse.db")
load_idempotent(clean_df, "orders", engine)  # run 1 — inserts all rows
load_idempotent(clean_df, "orders", engine)  # run 2 — same result, not doubled
Terminal $ python idempotent_load.py
'orders': 0 → 14,832 rows 'orders': 14,832 → 14,832 rows

The key observation is the second output line: 14,832 → 14,832. The row count does not increase, which proves the pipeline is safe to re-run — a nightly job that crashes and restarts will never double your data when you use if_exists="replace".

Example 3 — ELT Pattern

Beginner

What This Does

Load raw data first, then transform inside the database with SQL.

elt_pattern.py 
# Step 1 (Python): load raw data directly — no cleaning yet
raw_df.to_sql("raw_orders", con=engine,
              if_exists="replace",   # overwrite raw table on every run (idempotent)
              index=False)
print(f"Raw orders loaded: {len(raw_df):,} rows")  # log how many rows went in

# Step 2 (SQL): clean and reshape inside the database using SQL
TRANSFORM_SQL = """
CREATE OR REPLACE TABLE clean_orders AS
SELECT
    order_id,                           /* keep the unique row key */
    CAST(order_date AS DATE)   AS order_date,  /* parse string → proper date type */
    ABS(amount)                AS amount,       /* force all amounts to positive */
    UPPER(TRIM(region))        AS region        /* normalise region casing/spacing */
FROM raw_orders
WHERE amount   IS NOT NULL    /* drop rows with no amount */
  AND order_id IS NOT NULL    /* drop rows with no key */
"""
with engine.connect() as conn:
    conn.execute(text(TRANSFORM_SQL))   # run the SQL transform inside the warehouse
print("SQL transform applied — clean_orders table ready")
Terminal $ python elt_pattern.py
Raw orders loaded: 15,241 rows SQL transform applied — clean_orders table ready

Notice that 15,241 raw rows went in but the SQL transform in the warehouse decides what comes out — analysts can change the WHERE clause or add columns to the SELECT without the Python script changing at all. That flexibility is the core advantage of ELT over ETL.

Example 4 — Pipeline Logging

Beginner

What This Does

Add structured logging so you know exactly what happened at each step.

logged_pipeline.py 
import logging, time

logging.basicConfig(level=logging.INFO,
                    format="%(asctime)s %(levelname)s %(message)s")
log = logging.getLogger(__name__)

def run():
    log.info("Pipeline started")
    t0 = time.perf_counter()

    raw = extract("sales_raw.csv")
    log.info(f"Extracted {len(raw):,} rows")

    clean = transform(raw)
    log.info(f"Transformed: {len(clean):,} rows retained")

    load(clean, "clean_sales", engine)
    log.info(f"Pipeline complete in {time.perf_counter()-t0:.1f}s")
Terminal $ python logged_pipeline.py

Example 5 — Batch Scheduling

Intermediate

What This Does

Schedule the batch pipeline to run automatically at the same time every day using Python's schedule library — no cron syntax required.

batch_scheduler.py 
import schedule
import time
import logging

logging.basicConfig(level=logging.INFO,
                    format="%(asctime)s %(levelname)s %(message)s")

def run_batch_pipeline():
    """Execute the full ETL job — called once per scheduled tick."""
    logging.info("Batch run started")
    engine = create_engine("sqlite:///warehouse.db")  # connect to warehouse
    raw    = extract("sales_raw.csv")                 # step 1: pull source data
    clean  = transform(raw)                            # step 2: clean and reshape
    load(clean, "clean_sales", engine)                # step 3: write to warehouse
    logging.info(f"Batch complete — {len(clean):,} rows loaded")

# Register the job: run run_batch_pipeline() every day at 2:00 AM
schedule.every().day.at("02:00").do(run_batch_pipeline)

logging.info("Scheduler started — waiting for 02:00 run")

while True:
    schedule.run_pending()   # fire any jobs whose time has come
    time.sleep(60)           # wait 60 s then check again
Terminal $ python batch_scheduler.py
2026-04-16 00:00:01,042 INFO Scheduler started — waiting for 02:00 run 2026-04-16 02:00:00,318 INFO Batch run started 2026-04-16 02:00:05,203 INFO Batch complete — 14,832 rows loaded

The while True loop keeps the scheduler alive so it can fire the job night after night. In production you would run this script as a background service (Linux systemd or Windows Task Scheduler) so it restarts automatically after a server reboot — the schedule library itself is purely in-memory and does not persist state between restarts.

Exercise 1 — Write a Modular ETL Pipeline

Your Task

  2. Complete the transform(df) function: drop nulls in customer_id and total, parse order_date as datetime, remove rows where total < 0, and add a tax column equal to total * 0.1.

exercise_01_solution.py 
import pandas as pd
from sqlalchemy import create_engine

def extract(filepath):
    df = pd.read_csv(filepath)              # read the source CSV from disk
    print(f"Extracted {len(df):,} rows")   # log the raw row count immediately
    return df

def transform(df):
    df = df.dropna(subset=["customer_id", "total"])   # remove rows missing critical fields
    df["order_date"] = pd.to_datetime(df["order_date"])  # parse date strings to Timestamp
    df = df[df["total"] >= 0]              # discard refund or data-error rows
    df["tax"] = df["total"] * 0.1          # derive a 10% tax column from total
    print(f"Transformed: {len(df):,} rows remain")
    return df

def load(df, engine, table):
    df.to_sql(table, engine, if_exists="replace", index=False)  # full refresh — idempotent
    print(f"Loaded {len(df):,} rows into '{table}'")

def run():
    engine = create_engine("sqlite:///orders.db")
    raw    = extract("orders.csv")           # step 1 — read source file
    clean  = transform(raw)                   # step 2 — clean and enrich
    load(clean, engine, "orders")             # step 3 — write to warehouse
    print("Pipeline complete")

run()

Splitting extract, transform, and load into separate functions means that if the load step fails at midnight, you fix and rerun only that one function without re-downloading or re-cleaning all the data — critical when your source is a slow API or a multi-gigabyte CSV file.

Exercise 3 — Fix the Idempotency Bug

Objective: Apply pipeline design principles — idempotency and validation

Your Task

  1. Open the solution and read the buggy pipeline. Trace what happens to the sales table if you call run() three times in a row, then write one comment in the code explaining the problem.
  2. Fix the load() function so the pipeline is idempotent — running it any number of times must produce exactly the same result in the database.
  3. Add a row-count guard in transform(): if more than 20% of input rows are dropped, raise a ValueError that states how many rows were lost and what percentage that represents.
exercise_03_solution.py 
import pandas as pd
from sqlalchemy import create_engine

def extract(filepath):
    return pd.read_csv(filepath)

def transform(df):
    raw_count = len(df)                                  # record starting row count before drops
    df = df.dropna(subset=["order_id", "total"])      # remove incomplete rows
    df["total"] = df["total"].abs()                    # convert negatives to positive
    dropped = raw_count - len(df)                        # calculate how many rows were removed
    if dropped / raw_count > 0.20:                      # guard: >20% row loss is suspicious
        raise ValueError(
            f"Transform dropped {dropped:,} rows "
            f"({dropped / raw_count:.0%} of {raw_count:,}). Aborting load."
        )
    return df

def load(df, engine):
    df.to_sql("sales", engine, if_exists="replace", index=False)  # FIXED: replace not append
    print(f"Loaded {len(df):,} rows into 'sales'")

def run():
    engine = create_engine("sqlite:///sales.db")
    raw    = extract("orders.csv")
    clean  = transform(raw)
    print(f"Passing {len(clean):,} clean rows to load")  # confirm count before write
    load(clean, engine)
    print("Done — pipeline is idempotent")

run()

A pipeline that silently appends duplicate rows on every re-run raises no error and produces plausible-looking totals until someone audits the data days later; the 20% guard makes suspicious data loss loud and immediate, so failures are caught in seconds rather than discovered in a board report.

Exercise 4 — Simulate a Streaming Pipeline

Objective: Recognize batch vs streaming patterns and write a generator

Your Task

  1. Open the solution and read the batch pipeline. Identify the single line that makes it batch rather than streaming, and add a comment on that line explaining why.
  2. Write a simulate_stream(filepath) generator function that reads the CSV once but yields one row as a dict at a time, simulating a Kafka or Pub/Sub message stream.
  3. Write a process_stream(generator) function that iterates the generator, skips rows where total is None, adds a tax field equal to total * 0.1, and prints each processed record.
exercise_04_solution.py 
import pandas as pd

# ── Batch pipeline (unchanged reference) ────────────────────
def extract_batch(filepath):
    return pd.read_csv(filepath)  # BATCH: this line loads ALL rows into memory at once

def transform_batch(df):
    df = df.dropna(subset=["total"])   # remove rows with no total
    df["tax"] = df["total"] * 0.1     # add derived column
    return df

def run_batch():
    df = extract_batch("orders.csv")
    df = transform_batch(df)
    print(f"Batch processed {len(df):,} rows")

# ── Streaming simulation ─────────────────────────────────────
def simulate_stream(filepath):
    """Generator that yields one order dict at a time — simulates a Kafka message stream."""
    df = pd.read_csv(filepath)          # read source (in production: use a Kafka consumer)
    for _, row in df.iterrows():        # iterate one row at a time — not all in memory at once
        yield row.to_dict()             # yield a single record as a plain dict

def process_stream(generator):
    """Processes each incoming record individually as it arrives from the stream."""
    for record in generator:                    # iterate until the stream ends
        if record.get("total") is None:         # guard: skip records missing a total value
            continue
        record["tax"] = record["total"] * 0.1  # apply per-record transform in place
        print(
            f"order {record.get('order_id', '?')}: "
            f"total={record['total']:.2f}, tax={record['tax']:.2f}"
        )

process_stream(simulate_stream("orders.csv"))

Real streaming pipelines (Kafka, Pub/Sub, Kinesis) deliver one event at a time and never give you a complete batch to load into a DataFrame — understanding Python generators prepares you to handle live data sources where you must process each record immediately rather than accumulating everything before acting.

Common Mistakes

Pitfalls to avoid when working with pipelines

Appending Without Deduplication

Every re-run doubles your row count.

Pitfall

Beginners use if_exists='append' because they want incremental loading — adding new rows without wiping the table — but they don't realize that every pipeline re-run (from a crash, a cron retry, or a manual restart) loads all rows again, silently doubling the data each time.

Wrong — append re-runs duplicate rows 

df.to_sql("sales", con=engine,
          if_exists="append")
# Every re-run adds duplicate rows

Correct — idempotent full refresh 

df.to_sql("sales", con=engine,
          if_exists="replace")
# Full refresh — always idempotent

Use if_exists='replace' for full-refresh loads and an upsert keyed on a unique column for incremental loads where you want to add new rows without touching existing ones.

All Logic in One Giant Function

Impossible to test, debug, or resume.

Pitfall

Beginners combine all three stages into one function because writing a single block feels simpler, but they don't account for the fact that a failure at step 3 forces a complete restart from line 1 — including re-downloading data the source system may rate-limit or time out on.

Wrong — monolithic pipeline 

def run_everything():
    df = pd.read_csv("data.csv")
    df = df.dropna()
    df["date"] = pd.to_datetime(df["date"])
    df.to_sql("table", engine,
              if_exists="replace")

Correct — three separate functions 

raw   = extract("data.csv")
clean = transform(raw)
load(clean, "table", engine)
# Each step is independently testable

Separate functions are independently testable and make debugging far easier. When a 2 AM pipeline fails, modular code tells you exactly which function broke in seconds rather than minutes of tracing a monolithic script.

Choosing ETL or ELT Without Considering the Warehouse

Transforming in Python when the warehouse is faster.

Pitfall

Beginners default to transforming in Python because that is where they wrote their cleaning scripts, not realizing that cloud warehouses like BigQuery and Snowflake are purpose-built to run the same SQL aggregations across billions of rows faster and at a fraction of the cost of a Python process.

Wrong — heavy Python transform 

# Heavy Python transform before BigQuery
# — slower and harder to maintain
df = (df.groupby("region")
        .agg({"revenue": "sum"})
        .reset_index())
df.to_gbq("dataset.summary")

Correct — ELT with warehouse SQL 

# ELT: load raw, transform in BigQuery
df.to_gbq("dataset.raw_sales",
          if_exists="replace")
# Then run SQL/dbt inside BigQuery

If you are working with BigQuery or Snowflake, ELT is almost always faster and cheaper. Keep your Python pipeline to extract and load only, and write your business logic as SQL in dbt or a scheduled query.

Skipping Row-Count Validation at Each Step

Silent data loss with no error raised.

Pitfall

Beginners assume a Python script that exits with no exception has produced correct output, but dropna(), filter conditions, and join mismatches can silently discard the majority of rows without raising any error.

Wrong — no row count check 

raw   = extract("sales.csv")
clean = transform(raw)    # may drop rows silently
load(clean, "sales", engine)
# No row count check — data loss invisible

Correct — log counts at every step 

raw   = extract("sales.csv")
print(f"Extracted: {len(raw):,} rows")
clean = transform(raw)
print(f"Transformed: {len(clean):,} rows")
load(clean, "sales", engine)
print(f"Loaded: {len(clean):,} rows")

If the row count drops by more than a small threshold between extract and transform, raise an exception rather than loading partial data. A loud failure is always better than silent data loss reaching your warehouse.

No Logging in the Pipeline Script

No way to know what ran, when, or why it failed.

Pitfall

Beginners rely on print() for feedback during development, not realizing that when the script runs in a scheduler like cron or Airflow, print() output is silently discarded and no record of what happened survives.

Wrong — no logging 

def run():
    raw   = extract("sales.csv")
    clean = transform(raw)
    load(clean, "sales", engine)
# No record of what happened

Correct — Python logging module 

import logging
logging.basicConfig(level=logging.INFO,
    format="%(asctime)s %(levelname)s %(message)s")

def run():
    logging.info("Pipeline started")
    raw = extract("sales.csv")
    logging.info(f"Extracted {len(raw):,} rows")
    clean = transform(raw)
    logging.info(f"Transformed to {len(clean):,} rows")
    load(clean, "sales", engine)
    logging.info("Pipeline complete")

Use Python's built-in logging module rather than print(). It writes timestamps automatically, lets you filter by severity level, and can be redirected to a file so you have a record even when the job runs unattended overnight.

Summary of what you learned

01

ETL Pattern

You learned that ETL transforms data before loading it into the destination — the right choice when the target system has a rigid schema or cannot accept raw, unvalidated records.

02

ELT Pattern

You now know that ELT loads raw data into the warehouse first and runs transformations there using SQL — making it the faster and cheaper choice on modern cloud platforms like BigQuery and Snowflake.

03

Idempotency

You recognized that an idempotent pipeline always produces the same result no matter how many times it runs — a property you achieve by using if_exists='replace' or a deduplication step rather than blind appends.

04

Modular Pipelines

You learned to split a pipeline into separate extract(), transform(), and load() functions so each step can be tested in isolation and a failure at one stage does not force a full restart from the beginning.

05

Batch vs Streaming

You can now distinguish batch pipelines — which process data in scheduled chunks — from streaming pipelines, which handle each event the moment it arrives, and you know that the right choice depends on how quickly downstream decisions need the data.
