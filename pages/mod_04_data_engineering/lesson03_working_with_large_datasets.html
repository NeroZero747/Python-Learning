Overview

A high-level summary of the topic

Py

When a dataset outgrows your laptop’s memory, pandas throws a MemoryError and your pipeline dies. Data engineers solve this not by buying more RAM, but by working smarter with the data they already have.

Think of loading large data like moving house with a small truck. You can’t fit every box in one trip — so instead of trying to hire a bigger truck, you make multiple smaller trips. Chunk processing works exactly the same way.

Chunked Reading

Multiple smaller trips instead of one giant one

Read a manageable slice of the file, process it, save the result, then read the next slice.

Dtype Downcasting

Smaller boxes — why use a wardrobe box for one shirt?

A column with values 0–100 doesn’t need 8 bytes. Downcast int64 to int8 and save 87% memory.

Memory Diagnosis

Weighing each box first — know what's heavy before optimizing

Call df.memory_usage(deep=True) to see exactly how many bytes each column consumes. Diagnose before you optimize.

Vectorized Operations

Using a dolly instead of carrying each box by hand

Pandas vectorized operations run in compiled C and are 10–100× faster than Python loops.

The average analyst uses 5–10× more memory than needed because pandas defaults to the largest possible dtype for every column. Fixing this is free.

Key Takeaways

The most important ideas to remember

Pandas Defaults Waste Memory Before You Write a Single Line of Code

Pandas always loads integers as int64 (8 bytes each) and strings as Python objects (50+ bytes each), even when the data would fit in int8 or a category column. Calling df.memory_usage(deep=True) before writing any optimization code shows you exactly which columns are the biggest offenders and how much memory you can recover.

memory_usage() int64 object

Chunk Processing Turns a Memory Problem Into a Time Problem

Reading a 20 GB file with chunksize=100_000 means pandas only holds 100,000 rows in RAM at a time, so the machine's total memory never matters. If your script is taking too long you can tune the chunk size upward; if it's crashing with MemoryError you tune it down — but it will always finish.

chunksize Iterator pd.concat()

Converting Strings to category Dtype Eliminates Duplicate Storage

A column with the value "Approved" stored in 2 million rows holds 2 million separate Python string objects in memory — converting it to category stores that string exactly once and replaces every other row with a small integer pointer. The savings are largest when cardinality is low (fewer than 50 unique values in a column with millions of rows).

category astype() Low cardinality

A Python Loop Over a Million Rows Is 100× Slower Than a Vectorized Operation

When you write df["tax"] = df["amount"] * 0.1, pandas executes that multiplication in compiled C code across the entire column in one call. Writing the equivalent as a for loop forces Python to interpret every iteration individually, making it 10–100× slower and causing MemoryError risks on large DataFrames from repeated intermediate allocations.

Key Concepts

Memory profiling, dtype downcasting, chunked reading, and vectorized operations.

memory_usage()

Profile memory before you optimize

Diagnostic tool

Memory profiling tells you exactly how many bytes each column occupies in RAM. You call df.memory_usage(deep=True) to get a byte count per column. Divide the total by 1024² to convert to megabytes.

df.info() vs memory_usage(deep=True)

df.info() memory_usage()
Shows Dtypes & null counts Exact bytes per column
String cost ✗ Underestimates ✓ Accurate
Use for Quick schema check Optimization planning
Python 
df = pd.read_csv("large_sales.csv")
# See bytes used by each column
print(df.memory_usage(deep=True))
# Convert total bytes to megabytes
total_mb = df.memory_usage(deep=True).sum() / 1024**2
print(f"DataFrame uses {total_mb:.1f} MB")

Always profile first — run memory_usage(deep=True) before changing any dtypes. You need to see which columns are the biggest offenders. Guessing wastes time.

Dtype Rules

Shrink columns to their smallest safe size

Downcasting

Dtype downcasting replaces a column's default data type with a smaller one that still fits the data. You use pd.to_numeric(col, downcast="integer") for numbers and col.astype("category") for text. The key question is: what is the maximum value in this column?

Downcast Cheat Sheet

Current dtype Downcast to Memory saved
int64 (max ≤ 127) int8 87.5%
int64 (max ≤ 32,767) int16 75%
float64 float32 50%
object (<50 unique) category 70–95%
Python 
# Downcast integer columns automatically
df["quantity"] = pd.to_numeric(df["quantity"], downcast="integer")
# Downcast float columns
df["price"] = pd.to_numeric(df["price"], downcast="float")
# Convert low-cardinality text to category
df["status"] = df["status"].astype("category")
df["region"] = df["region"].astype("category")

Check the max value before downcasting — run df["col"].max() first. If the maximum exceeds the dtype's limit, you will get silent data corruption with no error message.

Chunked Reading

Process files larger than your RAM

Memory pattern

Chunked reading reads a large file in small batches instead of all at once. You pass chunksize=100_000 to pd.read_csv(). This returns an iterator — an object you step through to get one batch at a time. Your RAM only ever holds one batch, not the whole file.

Load All vs Chunked

read_csv() All at once
→
chunksize= Batches
→
No crash
Python 
results = []
# Read 100,000 rows at a time — never the whole file
for chunk in pd.read_csv("sales.csv", chunksize=100_000):
    # Filter and summarize each batch
    filtered = chunk[chunk["status"] == "Approved"]
    summary = filtered.groupby("region")["amount"].sum()
    results.append(summary)
# Combine all partial summaries into one result
final = pd.concat(results).groupby(level=0).sum()

Tune chunksize to your machine — start with 100_000. If you get MemoryError, halve it. If it runs slowly, double it. The right value depends on how many columns your file has.

Vectorized Ops

Replace loops with column operations

Performance

A vectorized operation applies a calculation to an entire column in one compiled C instruction. You write df["tax"] = df["amount"] * 0.1 and pandas processes every row at once. A Python for loop does the same job row by row — 10–100× slower.

Loop vs Vectorized

for loop Vectorized
Speed (1M rows) ~3 seconds ~3 milliseconds
Executed by Python interpreter Compiled C code
Use for Almost never Column-level operations
Python 
# Slow — loop processes one row at a time (~3s on 1M rows)
for i, row in df.iterrows():
    df.at[i, "tax"] = row["amount"] * 0.1
# Fast — vectorized operates on the whole column (~3ms on 1M rows)
df["tax"] = df["amount"] * 0.1
# Conditional vectorization — use np.where instead of if/else
import numpy as np
df["label"] = np.where(df["score"] >= 90, "Pass", "Fail")

If you find yourself writing a loop over rows, stop — there is almost always a vectorized alternative. Use df["col"] * value for math, np.where() for conditions, and .str.contains() for text filtering.

02

Auto-Downcast DataFrame

Beginner

What This Does

Automatically downcast integers, floats, and categorize low-cardinality strings.

auto_downcast.py 
def optimize_dtypes(df):
    for col in df.select_dtypes(include=["int64","int32"]).columns:
        df[col] = pd.to_numeric(df[col], downcast="integer")
    for col in df.select_dtypes(include=["float64"]).columns:
        df[col] = pd.to_numeric(df[col], downcast="float")
    for col in df.select_dtypes(include=["object"]).columns:
        if df[col].nunique() / len(df) < 0.05:
            df[col] = df[col].astype("category")
    return df

df = optimize_dtypes(df)
03

Chunked Aggregation

Intermediate

What This Does

Sum sales by region across a 10-million-row file without loading it all at once.

chunked_agg.py 
import pandas as pd

totals = {}
for i, chunk in enumerate(pd.read_csv("sales_10m.csv", chunksize=250_000)):
    chunk["amount"] = chunk["amount"].astype("float32")
    chunk["region"] = chunk["region"].astype("category")
    for region, total in chunk.groupby("region")["amount"].sum().items():
        totals[region] = totals.get(region, 0) + total
    print(f"Chunk {i+1} processed ({(i+1)*250_000:,} rows)")

for region, total in sorted(totals.items()):
    print(f"{region}: ${total:,.2f}")

Vectorized Operations vs. Loops

Intermediate

What This Does

Demonstrates the speed difference between a Python loop and a vectorized pandas operation on one million rows, and shows how to use np.where() for conditional column creation.

vectorized.py 
import pandas as pd, numpy as np, time

df = pd.DataFrame({"amount": np.random.uniform(0, 5000, 1_000_000)})

# --- Slow: Python loop ---
start = time.time()
for i, row in df.sample(10_000).iterrows():  # sample to avoid timeout
    df.at[i, "tax_loop"] = row["amount"] * 0.2
loop_s = time.time() - start

# --- Fast: vectorized (full 1M rows) ---
start = time.time()
df["tax_vec"] = df["amount"] * 0.2
vec_s = time.time() - start

print(f"Loop (10k rows):       {loop_s:.3f}s")
print(f"Vectorized (1M rows):  {vec_s:.4f}s")

# Conditional column — vectorized with np.where
df["tier"] = np.where(df["amount"] >= 1000, "Premium", "Standard")
Terminal $ python vectorized.py
Loop (10k rows): 2.847s
Vectorized (1M rows): 0.0031s

import pandas as pd

02

Downcast a DataFrame's Dtypes

Covers: Objective 3 — dtype optimization

Tasks

  1. 1Measure and print the DataFrame's starting size in MB using memory_usage(deep=True).
  2. 2Loop over all int64 columns and downcast each one using pd.to_numeric(col, downcast="integer").
  3. 3Do the same for float64 columns using downcast="float".
  4. 4Convert the "region" and "status" columns to category dtype, then print the final size and the MB saved.
exercise_02.py 
import pandas as pd, numpy as np

# Sample DataFrame with bloated dtypes (all defaults: int64, float64, object)
df = pd.DataFrame({
    "order_id":  np.arange(500_000, dtype="int64"),
    "quantity":  np.random.randint(1, 50, 500_000, dtype="int64"),
    "amount":    np.random.uniform(0.5, 999.9, 500_000),
    "region":    np.random.choice(["North", "South", "East", "West"], 500_000),
    "status":    np.random.choice(["Approved", "Pending", "Rejected"], 500_000),
})

# 1. Before size
before_mb = df.memory_usage(deep=True).sum() / 1024**2
print(f"Before: {before_mb:.1f} MB")

# 2. Downcast integer columns — pandas picks the smallest safe int type
for col in df.select_dtypes("int64").columns:
    df[col] = pd.to_numeric(df[col], downcast="integer")

# 3. Downcast float columns — float64 → float32 saves 50%
for col in df.select_dtypes("float64").columns:
    df[col] = pd.to_numeric(df[col], downcast="float")

# 4. Convert low-cardinality strings to category
df["region"] = df["region"].astype("category")  # 4 unique values
df["status"] = df["status"].astype("category")   # 3 unique values

after_mb = df.memory_usage(deep=True).sum() / 1024**2
print(f"After:  {after_mb:.1f} MB  |  Saved: {before_mb - after_mb:.1f} MB")
print(df.dtypes)

Why this matters: On a real 500 MB production file this same pattern routinely cuts memory to under 100 MB — often the difference between an analysis that crashes and one that runs. The fix takes under 10 lines and never changes the data values, only the internal storage format.

Count Rows Without Loading the Full File

Covers: Objective 2 — chunked reading

Tasks

  1. 1Create a chunk iterator by calling pd.read_csv("large_sales.csv", chunksize=100_000).
  2. 2For each chunk, filter the rows where amount > 1000 using a boolean condition.
  3. 3Accumulate the count of qualifying rows from each chunk into a running total.
  4. 4Print the final count formatted with a thousands separator — note that the full file was never held in RAM at once.
exercise_03.py 
import pandas as pd

total_count = 0  # running total across all chunks

# 1. Read 100,000 rows at a time — the full file is never in RAM
for chunk in pd.read_csv("large_sales.csv", chunksize=100_000):

    # 2. Filter this batch to rows where amount exceeds 1000
    qualifying = chunk[chunk["amount"] > 1000]

    # 3. Add the count from this batch to the running total
    total_count += len(qualifying)

# 4. Print the final total with a thousands separator
print(f"Rows with amount > 1000: {total_count:,}")
# Every row was visited exactly once — one chunk at a time

Why this matters: This is the standard pattern for summarizing a file that is larger than available RAM — the file can be 10 GB and the machine can have 4 GB of RAM and the script will still finish correctly because it only holds one batch of 100,000 rows at a time.

04

Replace a Slow Loop with a Vectorized Operation

Covers: Objective 4 — vectorized operations

Tasks

  1. 1Run the provided iterrows() loop on a DataFrame with 200,000 rows and record how long it takes.
  2. 2Rewrite the grade assignment using np.where(condition, "Pass", "Fail") and time that too.
  3. 3Verify both methods produce the same result by comparing the two "grade" columns using .equals().
  4. 4Calculate and print the speedup ratio: loop_time / vectorized_time.
exercise_04.py 
import pandas as pd, numpy as np, time

df = pd.DataFrame({"score": np.random.randint(0, 100, 200_000)})

# 1. SLOW — iterrows() processes one row at a time through the Python interpreter
df_slow = df.copy()
t0 = time.time()
for i, row in df_slow.iterrows():
    df_slow.at[i, "grade"] = "Pass" if row["score"] >= 60 else "Fail"
loop_time = time.time() - t0
print(f"Loop:       {loop_time:.2f}s")

# 2. FAST — np.where applies the condition to the whole column in compiled C code
df_fast = df.copy()
t1 = time.time()
df_fast["grade"] = np.where(df_fast["score"] >= 60, "Pass", "Fail")
vec_time = time.time() - t1
print(f"Vectorized: {vec_time:.4f}s")

# 3. Verify both methods produce identical results
print(f"Results match: {df_slow['grade'].equals(df_fast['grade'])}")

# 4. Speedup ratio
print(f"Speedup: {loop_time / vec_time:.0f}x faster")

Why this matters: At 200,000 rows you will typically see 30–100x speedups — and the ratio grows with row count. The vectorized version also uses less memory because pandas doesn't need to create a Python object for every row, so it wins on both time and space on large datasets.

05

Build a Chunked Aggregation Pipeline

Covers: Synthesis — chunked reading + groupby

Tasks

  1. 1Open "transactions.csv" using pd.read_csv(chunksize=100_000).
  2. 2For each chunk, filter for rows where status == "Approved", then group by "region" and sum "amount".
  3. 3Append each partial result to a list, then combine all of them using pd.concat().
  4. 4Run a second .groupby().sum() on the combined result to merge the partial sums, then print the final region totals sorted in descending order.
exercise_05.py 
import pandas as pd

partials = []  # will hold one partial sum per chunk

# 1. Read the file in batches — the full CSV is never in RAM
for chunk in pd.read_csv("transactions.csv", chunksize=100_000):

    # 2. Filter: keep only approved transactions
    approved = chunk[chunk["status"] == "Approved"]

    # Group this batch by region and sum the amounts
    partial = approved.groupby("region")["amount"].sum()
    partials.append(partial)

# 3. Stack all partial sums into one Series
combined = pd.concat(partials)

# 4. Group a second time — each region appears once per chunk, so we merge them
final = combined.groupby(level=0).sum()

# Print sorted by total amount, highest first
print(final.sort_values(ascending=False))

Why this matters: The two-pass pattern (partial aggregation per chunk, then a final merge) is the foundational idea behind distributed computing tools like Apache Spark — they spread the partial aggregation across many machines and merge at the end. Understanding this pattern in pandas means you already understand the logic at the heart of big-data frameworks.

Common Mistakes

Pitfalls to avoid when handling large data

Loading then filtering

Filtering after a full load wastes all the memory you were trying to save

✗ Wrong

df = pd.read_csv("huge.csv")
# Loads 10M rows just to keep 100k
df = df[df["region"] == "EMEA"]

✓ Correct

chunks = []
for chunk in pd.read_csv("huge.csv",
                          chunksize=100_000):
    chunks.append(chunk[chunk["region"]=="EMEA"])
df = pd.concat(chunks)

Loading 10 million rows then immediately discarding 9.9 million uses the same peak RAM as loading everything — you get none of the memory benefit. Filter inside the for chunk in pd.read_csv(..., chunksize=) loop so pandas never holds the full file in memory at once. Accumulate only the filtered rows and call pd.concat() at the end.

Using iterrows() on large DataFrames

Every row creates a new Python object — 100× slower than a vectorized operation

✗ Wrong

for idx, row in df.iterrows():
    df.at[idx, "tax"] = row["amount"] * 0.1

✓ Correct

# Single vectorized operation
df["tax"] = df["amount"] * 0.1

Vectorized operations execute in compiled C code and process the entire column in a single call. On a million-row DataFrame, the vectorized version runs in milliseconds; the iterrows() loop takes seconds. If you need row-level conditional logic, use np.where() or np.select() — both are vectorized and avoid the Python object overhead entirely.

Forgetting category dtype limitations

Category columns reject values not in their original vocabulary — you must extend it first

✗ Wrong

df["region"] = df["region"].astype("category")
# Raises ValueError — new value not in categories
df.loc[0, "region"] = "New Region"

✓ Correct

df["region"] = df["region"].astype("category")
# Extend vocabulary first, then assign
df["region"] = df["region"].cat.add_categories("New Region")
df.loc[0, "region"] = "New Region"

A category column stores a fixed vocabulary of unique values — it will reject anything outside that vocabulary with a ValueError. Call .cat.add_categories(["New Value"]) before assigning to extend the vocabulary, or temporarily cast back to object, make the assignment, then re-apply astype("category"). Remember: category dtype is optimized for reading, not for frequent writes.

Using category on a high-cardinality column

Category dtype only saves memory when the column has few unique values — on columns like URL or email it can use more memory

✗ Wrong

# user_id is unique for every row
df["user_id"] = df["user_id"].astype("category")
# Uses MORE memory — 1M categories + integer map

✓ Correct

# Check cardinality before converting
unique_ratio = df["user_id"].nunique() / len(df)
if unique_ratio < 0.05:  # less than 5% unique
    df["user_id"] = df["user_id"].astype("category")

The category dtype saves memory by replacing each repeated string with a small integer, but the lookup table must also be stored. When a column has nearly as many unique values as rows (like user IDs or emails), the table is as large as the original data and the integer map adds overhead on top. Before converting any text column, check its cardinality: df["col"].nunique() / len(df) < 0.05 is a reliable rule of thumb for safe category conversion.

Not specifying dtypes at read time

Loading with pandas defaults and then downcasting wastes time and uses peak memory twice

✗ Wrong

# Load all at default dtypes, then optimize
df = pd.read_csv("data.csv")
df["age"] = df["age"].astype("int8")
df["status"] = df["status"].astype("category")

✓ Correct

# Specify dtypes at read time — never allocates int64
df = pd.read_csv("data.csv", dtype={
    "age": "int8",
    "status": "category",
    "amount": "float32"
})

When you load a file with default dtypes and then cast, pandas allocates memory for the wide default type first and then creates a copy in the compact target type — so at peak, both versions exist simultaneously. Passing a dtype={} dictionary to pd.read_csv() avoids this entirely: pandas writes each column directly at the compact type on first read, so the large intermediate allocation never happens.
