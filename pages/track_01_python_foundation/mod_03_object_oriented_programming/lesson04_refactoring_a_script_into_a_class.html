Overview

What refactoring is and why it matters

Py

Refactoring is the process of reorganizing your code to make it cleaner and easier to manage, without changing what the code actually does.

Think of your Python script like a chef's stack of paper notes — each step written somewhere, but nothing labelled, nothing filed, and nothing a colleague could pick up and follow. A class is that recipe book, and Python is the language you use to write it.

The Loose Script

Loose notes — scattered, unlabelled, unfiled

Every step scattered across the countertop with no clear structure.

A Class

The recipe book — one cover, every step

Gathers all your loose steps under one named cover.

A Method

One recipe — filed in the right chapter

One named page in the book, ready to call when needed.

The run() Method

Cook the full meal — start to finish

The page that calls every other recipe in the right order.

If you have ever used an Excel macro or a SQL stored procedure that runs steps in a fixed order, you already understand what a class does — it is the same idea applied to your Python scripts.

Key Concepts

Refactoring, classes, self, and run() — the four ideas behind moving a script into a class.

Refactoring

Improving code shape without changing its output

structure

Refactoring means rewriting the structure of your code without changing what it does. You take the same logic that already works and reorganize it into a cleaner layout — usually by moving loose functions into a class.

When to refactor

Script has 3 or more related functions Group them into a single class
You catch yourself scrolling to find functions Methods are easier to navigate inside a class
Same script runs in multiple workflows A class lets you create a fresh object each time
before_refactor.py 
# Before — loose functions, no grouping
def load_data():
    print("Loading data...")

def clean_data():
    print("Cleaning data...")

def save_report():
    print("Report saved!")

# You must remember to call them in the right order every time
load_data()
clean_data()
save_report()

Nothing changes for the user: refactoring is invisible to anyone running your script — the output stays the same, only the internal organization improves.

class keyword

The container that holds all your methods

class Name:

The class keyword creates a named container for all your related functions. Once you define a class, every function you put inside it becomes a method — and all those methods share the same name, so you always know where they live.

class naming rules

Start with a capital letter ReportRunner ✓ reportRunner ✗
Use PascalCase (no underscores) OrderProcessor ✓ order_processor avoid
End the header line with a colon class ReportRunner: ✓
report_runner.py 
class ReportRunner:                  # capital R — the class header
    def load_data(self):             # method 1 — indented inside the class
        print("Loading data...")

    def clean_data(self):            # method 2 — same indent level
        print("Cleaning data...")

    def save_report(self):           # method 3 — every function is now a method
        print("Report saved!")

Indentation is the container: Python knows a method belongs to a class because it is indented one level inside the class body — if you forget to indent, the function stays outside the class and loses its connection.

self

Each method's link to its own object

self

When you move a function into a class, you must add self as the first parameter of every method. This tells Python "this method belongs to an object" — without it, Python cannot connect the method call to the right object.

self rules

Always first, inside the class def load_data(self): ✓ def load_data(): ✗
Never pass it when calling report.load_data() ✓ report.load_data(self) ✗
Python fills it in automatically Python passes the object as self for you
report_runner.py 
class ReportRunner:
    def load_data(self):             # self is required — do not skip it
        print("Loading data...")

    def clean_data(self):            # same pattern for every method
        print("Cleaning data...")

report = ReportRunner()              # create the object
report.load_data()                   # Python passes report as self — automatically

The most common refactoring mistake is copying a function into a class but forgetting to add self — Python will raise a TypeError the first time you call the method.

run() method

The single-line trigger that runs all the steps

run()

A run() method is a single method that calls all the other steps in the correct order. Instead of listing every step in your main script, you call just run() and the class handles the sequence for you.

run() patterns

Calls other methods via self self.load_data() ✓
Steps in the right order load → clean → save
Called once to trigger everything report.run() ✓
report_runner.py 
class ReportRunner:
    def load_data(self):
        print("Loading data...")

    def clean_data(self):
        print("Cleaning data...")

    def save_report(self):
        print("Report saved!")

    def run(self):                   # chains all steps in the right order
        self.load_data()             # step 1
        self.clean_data()            # step 2
        self.save_report()           # step 3

report = ReportRunner()
report.run()                         # all three steps run with one call

Your main script stays clean: by putting the sequence logic inside run(), you reduce the main script to two lines — creating the object and calling run — making it trivial to hand off to a colleague.

Refactoring Step by Step — the Default Pattern

You start by finding functions that work together — like load_data, clean_data, and save_report. You then write a class, indent those functions inside it, and add self as the first parameter to each. Finally, you add a run() method that calls all of them in order. Every time you follow this pattern, the script produces exactly the same output as before.

Here is the refactored ReportRunner — notice how each method has self as its first parameter and run() calls all three steps in order:

report_runner.py 
class ReportRunner:              # Step 1 — create the class container
    def load_data(self):         # Step 2 — move load_data in as a method
        print("Loading data...")

    def clean_data(self):        # Step 3 — move clean_data in as a method
        print("Cleaning data...")

    def save_report(self):       # Step 4 — move save_report in as a method
        print("Report saved!")

    def run(self):               # Step 5 — add run() to call all steps
        self.load_data()
        self.clean_data()
        self.save_report()

report = ReportRunner()          # create one object from the class
report.run()                     # one line triggers the full pipeline
Terminal $ python report_runner.py
Loading data...
Cleaning data...
Report saved!

The Three Concepts in Any Refactoring

Every refactoring you ever do in Python uses these three building blocks — master them once and you can restructure any script.

Refactoring

Same output — cleaner shape

Refactoring only changes how your code is organized, never what it returns or prints.

The Class

Your labelled container

Writing class ReportRunner: creates a named container that holds all related methods together.

The run() Method

One call, all steps

Adding a run() method means the rest of your program only ever needs to call one line to run the whole workflow.

Once you have refactored a script, you can reuse it anywhere in your project with just two lines — one to create the object and one to call run(). That same two-line pattern works whether your class has three methods or thirty.

03

Adding run()

Beginner Reports run() Methods

What This Does

This script adds a run() method that calls all three steps in order from inside the class. Your main script shrinks to two lines, and you never need to remember which steps exist or what order they run in.

report_runner.py 
class ReportRunner:                  # the container that holds all methods

    def load_data(self):             # step 1 — load the data
        print("Loading data...")

    def clean_data(self):            # step 2 — clean the data
        print("Cleaning data...")

    def save_report(self):           # step 3 — save the report
        print("Report saved!")

    def run(self):                   # calls all three steps in order
        self.load_data()             # runs step 1
        self.clean_data()            # runs step 2
        self.save_report()           # runs step 3

report = ReportRunner()              # create the object
report.run()                         # one line triggers everything
Terminal $ python report_runner.py
Loading data...
Cleaning data...
Report saved!

Once a class has a run() method, you can drop it into any other script and trigger the whole pipeline with just one line.

Build the Class

Beginner Reports class def self

Your Task

Your team runs a data pipeline every morning to load, clean, and save a report. Create a class called ReportRunner with three methods: load_data(self), clean_data(self), and save_report(self). You just turned three separate function calls into three organized methods on one object. If the team ever needs to run this pipeline again, you create another ReportRunner() object — the logic stays in one place and never needs to be rewritten.

Add a run() Method

Beginner Pipelines run() self.method()

Your Task

The ReportRunner class has three methods, but you still need to call them one by one. Add a fourth method called run(self) that calls each of the three steps in order using self.         print("Report saved!")

The run() method turns the class into a one-call pipeline — you no longer need to remember the order of three separate calls. This is the same idea as a SQL stored procedure: call one name and all the steps fire in the right sequence.

Common Mistakes

Three mistakes beginners make when refactoring scripts into classes

Defining a Method Without the self Parameter

Python passes the object automatically — writing def load_data(): raises a TypeError when the method is called.

Pitfall

Every method inside a class must have self as its first parameter — Python uses it to pass the object reference automatically when you call report.load_data(). Without it, Python raises TypeError: load_data() takes 0 positional arguments but 1 was given. Add self as the first parameter of every method definition.

Wrong — no self parameter 

class ReportRunner:
    def load_data():
        print("Loading data...")

Correct — self as first parameter 

class ReportRunner:
    def load_data(self):       # self receives the object
        print("Loading data...")

Think of self as the method's name tag — it tells Python which object the method belongs to. Every method in a class needs it, and Python always passes it silently for you when you call report.load_data().

Calling a Method on the Class Instead of an Object

ReportRunner.load_data() calls the class directly — Python raises a TypeError because there is no object to pass as self.

Pitfall

Writing ReportRunner.load_data() calls the method on the class blueprint rather than on a real object, so Python has nothing to pass as self and raises TypeError: load_data() missing 1 required positional argument: 'self'. Create an object first with report = ReportRunner(), then call report.load_data().

Wrong — calling on the class 

# no object created
ReportRunner.load_data()

Correct — calling on an object 

report = ReportRunner()  # build the object
report.load_data()       # call the method on it

A class is a blueprint, not a usable thing. You have to build something from the blueprint first — report = ReportRunner() is that building step. Only then can you call its methods.

Calling Another Method Inside run() Without self.

Writing load_data() inside run() raises a NameError — Python looks for a global function, not a method on the object.

Pitfall

Inside run(self), writing load_data() tells Python to search for a standalone global function — which does not exist — and raises NameError: name 'load_data' is not defined. Use self.load_data() so Python knows to look for the method on the same object.

Wrong — missing self. prefix 

def run(self):
    load_data()    # NameError
    clean_data()   # NameError
    save_report()  # NameError

Correct — self. on every call 

def run(self):
    self.load_data()    # call the method on this object
    self.clean_data()   # call the method on this object
    self.save_report()  # call the method on this object

Inside a class, self. is always the prefix when one method needs to call another — it scopes the call to the current object. Think of it as saying "run the version of this method that belongs to me, not someone else's".
