Overview

A high-level summary of the topic

Py

Merging combines two tables into one by matching rows that share the same key value — identical to a SQL JOIN.

Think of merge like matching receipts to invoices — both share an invoice number, so you staple the matching pairs together into one document.

merge()

The stapler — joins two tables on a shared key

You call pd.merge(left, right, on="key") and pandas walks through every row in both DataFrames, pairing rows whose key values are equal — exactly like a SQL JOIN. The result is a single, wider DataFrame that contains columns from both original tables, ready for analysis or export.

Join type (how=)

The matching rule — what to do with no match

The how= parameter tells pandas what to do when a key exists in one table but not the other — inner drops them, left keeps left-side rows, right keeps right-side rows, and outer keeps everything. Choosing the wrong join type is the most common reason analysts lose rows they expected to keep or end up with unexpected NaN values in their report.

Join key columns

The invoice number — the column used to match

You pass a column name (or a list of names) to the on= parameter, and pandas only pairs rows where those columns have equal values — just like the ON clause in a SQL JOIN. When the key columns have different names in each table, you use left_on= and right_on= instead.

Unmatched rows

The orphan receipt — no match in the other pile

After a left or outer join, any row from one table that had no matching key in the other table shows up with NaN in the columns that came from the unmatched side. Spotting these NaN values with merged.isna().sum() is the fastest way to verify data quality and catch key mismatches before they reach your final report.

The four join types — inner, left, right, outer — work identically to SQL; pick the same one you would use in a query.

Key Takeaways

The most important ideas to remember

Inner Join Silently Drops Rows

The default merge is an inner join, which discards every row whose key value has no match in the other table. Always compare row counts before and after the merge to catch unexpected data loss — a 10% drop often signals a data-quality issue, not a missing feature.

Inner join Row count Silent drops

Duplicate Keys Multiply Row Count

If the key column contains duplicate values in both tables, merge() produces a Cartesian product that can inflate a 100-row table to 10,000 rows. Run df["key"].duplicated().sum() on the lookup table before every merge to catch duplicates early.

Duplicates Cartesian validate=

Composite Keys Prevent False Matches

Merging on a single column is only safe when that column uniquely identifies each row; if it does not, pass a list of columns to on= so both must match. This is the same logic as a SQL composite key — without it, two different transactions that share an ID but fall on different dates get incorrectly paired.

on=[list] Composite key Uniqueness

NaN After a Merge Means Keys Did Not Match

When you see NaN in columns that came from the right table, those rows had no matching key — they are the data-quality red flags your stakeholders will ask about. Running merged.isna().sum() immediately after a left join tells you exactly how many rows failed to match.

isna() Left join NaN Data quality

Key Concepts

merge(), join types, and key columns.

merge()

Combine two DataFrames

method

merge() joins two DataFrames on a shared column — like a VLOOKUP across two tables. Set on to the column name they share.

Key Parameters

on Column name shared by both tables
how Join type — inner, left, right, outer
left_on / right_on When column names differ
suffixes Labels for duplicate column names
Python 
merged = pd.merge(orders, customers, on="CustomerID")  # inner join
print(merged.shape)              # combined table
print(merged.columns.tolist())   # columns from both tables

Check for duplicates before merging. If one table has repeated keys, merge() multiplies rows. Run df.duplicated() first.

Inner Join

Keep only matching rows

how

An inner join keeps only rows that appear in both tables. Rows with no match in the other table are dropped. This is the default how setting.

Inner Join Behavior

Left table Right table
Key in both ✓ Kept ✓ Kept
Key in left only ✗ Dropped —
Key in right only — ✗ Dropped
Python 
inner = pd.merge(orders, customers,
                 on="CustomerID",
                 how="inner")         # only matching rows
print(inner.shape)                     # fewer rows than either table

Inner join is the default. If you omit how, Pandas uses inner automatically.

Left Join

Keep all rows from the left table

how

Left join keeps every row from the left table. If a row has no match in the right table, the right-side columns are filled with NaN.

Left Join Behavior

Left table Right table
Key in both ✓ Kept ✓ Kept
Key in left only ✓ Kept NaN
Key in right only — ✗ Dropped
Python 
left = pd.merge(orders, customers,
                on="CustomerID",
                how="left")           # keep all orders
print(left.shape)                      # same row count as orders

Left join is the most common choice. Use it when you want to keep every row from your main table and add extra columns from a lookup table.

Right / Outer Join

Other join options

how

A right join keeps all rows from the right table. An outer join keeps all rows from both tables, filling gaps with NaN.

Right vs Outer

Right join Outer join
Left-only rows ✗ Dropped ✓ Kept + NaN
Right-only rows ✓ Kept ✓ Kept + NaN
Both matched ✓ Kept ✓ Kept
Python 
right = pd.merge(orders, customers,
                 on="CustomerID",
                 how="right")          # keep all customers
outer = pd.merge(orders, customers,
                 on="CustomerID",
                 how="outer")          # keep everything

Outer joins can create many NaN values. Run df.isna().sum() after an outer join to see how many gaps appeared.

Code Examples

Hands-on code snippets to explore the concepts

01

Merge Two Tables

Intermediate Orders pd.merge() on

What This Does

This script joins two DataFrames on a shared column using pd.merge(). Matching rows are combined into a single table.

merge_basic.py 
import pandas as pd  # load pandas

orders = pd.read_csv("orders.csv")        # order details
customers = pd.read_csv("customers.csv")  # customer details

merged = pd.merge(orders, customers, on="customer_id")  # join on key
print(merged.head())  # show combined table
Terminal $ python merge_basic.py
order_id customer_id total customer_name
0 1001 101 250 Alice

The on= column must exist in both DataFrames with the same name — check spelling and case before merging.

02

Left Join

Intermediate Products how="left" NaN matches

What This Does

This script uses a left join to keep every row from the left table. Rows with no match in the right table get NaN values.

left_join.py 
import pandas as pd  # load pandas

products = pd.read_csv("products.csv")      # all products
suppliers = pd.read_csv("suppliers.csv")    # supplier info

A left join never drops rows from your main table — use it when you want to add extra columns without losing data.

Join on Multiple Keys

Intermediate Sales on=[list] composite key

What This Does

This script merges on two columns at once — both must match for rows to join. Use this when a single column is not unique enough.

sales = pd.read_csv("sales.csv")      # monthly sales

04

Validate Merge Result

Intermediate Employees .shape isna()

What This Does

This script compares row counts before and after a merge, then checks for NaN values. These two checks catch silent data loss or unexpected row multiplication.

validate_merge.py 
import pandas as pd  # load pandas

employees = pd.read_csv("employees.csv")      # 120 employees
departments = pd.read_csv("departments.csv")  # department info

merged = pd.merge(employees, departments,
                  on="dept_id", how="left")  # keep all employees

# Check 1 — row count should match the left table
print("Before:", len(employees), " After:", len(merged))

# Check 2 — NaN means unmatched keys
print(merged.isna().sum())  # any gaps?
Terminal $ python validate_merge.py
Before: 120 After: 120
dept_name 3
dtype: int64

If the row count increases after a merge, you have duplicate keys. If NaN appears, some keys had no match in the other table.

05

Different Key Names

Intermediate Invoices left_on right_on

What This Does

This script merges two tables when the key column has a different name in each table. Use left_on and right_on instead of on.

diff_keys.py 
import pandas as pd  # load pandas

invoices = pd.read_csv("invoices.csv")    # uses "cust_id"
customers = pd.read_csv("customers.csv")  # uses "customer_id"

merged = pd.merge(invoices, customers,
                  left_on="cust_id",       # key in left table
                  right_on="customer_id")  # key in right table
print(merged.head())
Terminal $ python diff_keys.py
cust_id invoice_total customer_id customer_name
0 201 350 201 Alice

After merging with left_on / right_on, both key columns remain in the result. Drop the duplicate with .drop(columns="customer_id").

SQL / Excel Comparison

How this topic compares across Python, SQL, and Excel

If you already join tables in SQL with JOIN or use VLOOKUP in Excel to combine two datasets, pandas merge() does the same thing — it matches rows from two DataFrames on a shared key column.

The SQL clause that combines rows from two tables based on a matching key column.

The Excel function that looks up a key in one table and returns a column from another.

Returns only rows that have a matching key in both tables.

Python how="left"

Keeps all rows from the left DataFrame even if there is no match on the right.

Keeps all rows from the left table and fills NULL for unmatched right rows.

The condition defining which columns to match when joining two tables.

The column in one table whose values are used to find matching rows in the other.

Same customer join, three tools

Python code

result = pd.merge(orders, customers,
    on="customer_id", how="left")

SELECT *
FROM orders
LEFT JOIN customers
  ON orders.customer_id = customers.id;

merge() is SQL JOIN and Excel VLOOKUP expressed in Python. The concept — match rows from two tables on a shared key — is exactly the same in all three tools.

Common Mistakes

Pitfalls beginners hit when working with merging DataFrames

Merging on a Column That Does Not Match

The result contains zero rows or many NaN values because no keys aligned.

Pitfall

If the join key has different names in each DataFrame, merge() finds no matches. Use left_on and right_on when the column names differ.

Wrong — no match found 

merged = orders.merge(clients, on="ID")
# KeyError — orders has "CustomerID", not "ID"

Correct — specify both keys 

merged = orders.merge(clients,
                      left_on="CustomerID",   # left DataFrame
                      right_on="ID")           # right DataFrame
print(merged.shape)                             # confirm match count

Print the columns of both DataFrames before merging. If the names differ, spell each one out with left_on and right_on.

Using the Default Inner Join When You Need All Rows

Rows without a match are silently dropped from the result.

Pitfall

The default how="inner" keeps only rows that match in both DataFrames. If you need all rows from the left table, use how="left".

Wrong — inner drops rows 

merged = orders.merge(clients, on="CustomerID")
# rows with no matching client are gone

Correct — use left join 

merged = orders.merge(clients,
                      on="CustomerID",
                      how="left")       # keep all orders
print(merged.shape)                      # same row count as orders

Ask yourself: "Do I need every row from my main table, even if there is no match?" If yes, set how="left".

Getting Unexpected Duplicate Rows After Merge

A one-to-many relationship multiplies rows in the result.

Pitfall

If the right table has multiple rows per key, each left row is duplicated for every match. Check for duplicates in the join key with .duplicated() before merging.

Wrong — duplicates appear 

# clients has 2 rows for CustomerID 101
merged = orders.merge(clients, on="CustomerID")
# order row is now duplicated

Correct — check first 

# verify no duplicates in the join key
dupes = clients["CustomerID"].duplicated().sum()  # count dupes
print(f"Duplicates: {dupes}")                      # should be 0
merged = orders.merge(clients, on="CustomerID")    # safe merge

Always check df["key"].duplicated().sum() on the right table before merging. Unexpected duplicates are the number-one cause of inflated row counts.

Ignoring _x and _y Suffix Columns

Both tables share a column name and the result has confusing _x / _y suffixes.

Pitfall

When both DataFrames have a column with the same name (for example, amount), merge() appends _x and _y to disambiguate them. These cryptic names make the result hard to read and easy to confuse.

Wrong — confusing suffixes 

merged = orders.merge(returns, on="order_id")
# columns: amount_x, amount_y — which is which?

Correct — descriptive suffixes 

merged = orders.merge(returns, on="order_id",
                      suffixes=("_order", "_return"))
# columns: amount_order, amount_return — clear

Set suffixes=("_orders", "_returns") to name the source table directly. This makes the merged DataFrame self-documenting.

Not Checking NaN After a Left Join

Unmatched rows silently get NaN values that corrupt downstream calculations.

Pitfall

A left join keeps every row from the left table, but when a key has no match, the right-side columns are filled with NaN. If you skip the null check and immediately calculate a sum or mean, those NaN values silently skew your result.

Wrong — skip null check 

merged = orders.merge(prices, on="sku", how="left")
total = merged["price"].sum()  # NaN rows excluded silently

Correct — check nulls first 

merged = orders.merge(prices, on="sku", how="left")
print(merged["price"].isna().sum())  # check gaps first
total = merged["price"].sum()        # now you know the gap

After every left join, run merged.isna().sum() to see how many rows had no match. Decide whether to fill those NaN values or investigate the missing keys before continuing.
