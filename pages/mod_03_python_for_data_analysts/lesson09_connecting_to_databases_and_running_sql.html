Lesson Objective

The goal and expected outcome of this lesson

Build a connection string

A connection string includes the database type, host, port, user, and password needed to open a live connection.

Connect with SQLAlchemy

SQLAlchemy provides a Python object that manages your database connection so pandas can query it directly.

Load a table into pandas

The read_sql_table function retrieves an entire database table and returns it as a fully typed DataFrame.

Close connections safely

Always closing a database connection after querying prevents resource leaks that can crash long-running scripts.

Run a custom SQL query

The read_sql function accepts a raw SQL string so you can filter, join, and aggregate data before it reaches pandas.

Use parameterized queries safely

Passing user values through the params argument prevents SQL injection and keeps your queries secure.

This lesson replaces the manual step of exporting data from a database to CSV before loading it in Python.

Overview

A high-level summary of the topic

Py

A database connection lets Python talk directly to a live database — no CSV export needed.

Think of a database connection like a phone call — you dial the number (connection string), ask your question (query), get the answer (DataFrame), then hang up.

Connection string

The phone number — address and login details

A connection string is a URL that contains the database type, hostname, port, username, and database name so Python knows exactly where to dial and how to log in.

SQLAlchemy engine

The phone handset — manages the connection

The engine is the handset you pick up before making any call. It remembers the number you dialed and manages a pool of open lines so multiple queries can share the same connection efficiently.

read_sql_table()

The question — loads a full table as a DataFrame

This is like asking the person on the other end to read you the entire contact list. You get every row and column back, ready to work with in pandas.

close()

The hang-up — releases the connection cleanly

Hanging up the phone is essential — if you leave the line open, the database keeps waiting and eventually runs out of available connections for other callers.

read_sql()

The custom question — runs your own SQL

Instead of asking for the entire list, you ask a specific question — "Give me only the customers in New York." The database filters the data before sending it back, saving time and memory.

Parameterized queries

The safe question — prevents injection attacks

Rather than reading a name directly into your question (which a bad actor could hijack), you use a blank to fill in later. The database fills the blank safely, blocking any attempt to sneak in extra commands.

SQLAlchemy supports SQL Server, Oracle, Snowflake, and Teradata — the same connection code works across all of them.

Key Takeaways

The most important ideas to remember

Use Context Managers to Close Connections

Wrapping the engine in a with block guarantees the connection is closed even if your script crashes mid-run. Without it, the database keeps holding open connection slots that can exhaust pool limits on long-running scripts.

with block Auto-close Connection pool

Never Hard-Code Credentials in Scripts

A connection string with a username and password embedded should never be committed to version control. Store credentials in a .env file that Git ignores and load them at runtime with os.environ.

.env file No hard-code Git safe

Connect with a Read-Only Role

Using a database account that has only SELECT permissions during development prevents accidental writes to production data. This is a standard requirement in most enterprise data teams and protects you from running an unintended UPDATE or DELETE.

Read-only SELECT only Least privilege

Never Build SQL with f-Strings

Inserting a filter value directly into an f-string SQL query creates a SQL injection vulnerability that attackers can exploit. Always pass filter values through the params argument with a :placeholder instead.

SQL injection params= Parameterized

Store Long Queries in .sql Files

Saving a large SELECT statement as a separate .sql file and reading it with open() keeps your Python script clean and focused on transformation logic. It also lets SQL tools like DBeaver run the query independently for debugging.

.sql files Separation DBeaver

LIMIT Rows During Development

Adding LIMIT 500 to your development query loads just enough rows to test your pandas transformation code without waiting for a full production data pull. Once your code is correct, remove the limit for the final run.

LIMIT Dev speed Test rows

Key Concepts

Connection strings, SQLAlchemy engines, and database types.

Connection Strings

Address that locates your database

concept

A connection string is a URL that tells Python where the database is and how to log in. It follows the pattern dialect://user:password@host/dbname.

Connection String Parts

dialect Database type — mssql, oracle, snowflake
user:password Login credentials
host Server address — localhost or an IP
dbname Name of the database
Python 
# SQL Server — uses pyodbc driver
conn = "mssql+pyodbc://admin:pw@sqlserver01/company?driver=ODBC+Driver+18+for+SQL+Server"
# Oracle — uses oracledb driver
conn = "oracle+oracledb://admin:pw@oracle01:1521/sales"

Always include the driver parameter for SQL Server. The ?driver= part tells SQLAlchemy which ODBC driver is installed on your machine.

SQLAlchemy Engine

Python object that manages the connection

object

An engine is the object SQLAlchemy uses to talk to the database. Create it once with create_engine(), then reuse it for every query.

Engine vs Connection

Engine Connection
Created by create_engine() engine.connect()
Lifetime Whole script One query block
Reusable ✓ Yes ✗ Close after use
Python 
from sqlalchemy import create_engine
engine = create_engine("mssql+pyodbc://admin:pw@sqlserver01/company?driver=ODBC+Driver+18+for+SQL+Server")
print(engine)                                    # Engine object

Create the engine once at the top of your script. Pass the same engine variable to every pd.read_sql() call.

Database Types

SQL Server, Oracle, Snowflake, and Teradata

reference

Database types (dialects) determine the connection string format and the pip package you need. Each database needs its own driver package.

Common Dialects

Dialect Database pip install Connection String
mssql+pyodbc SQL Server pyodbc mssql+pyodbc://u:p@host/db
oracle+oracledb Oracle oracledb oracle+oracledb://u:p@host/db
snowflake Snowflake snowflake-sqlalchemy snowflake://u:p@acct/db
teradatasql Teradata teradatasqlalchemy teradatasql://u:p@host/db
Python 
# SQL Server — pip install pyodbc
engine = create_engine("mssql+pyodbc://admin:pw@sqlserver01/sales?driver=ODBC+Driver+18+for+SQL+Server")

# Oracle — pip install oracledb
engine = create_engine("oracle+oracledb://admin:pw@oracle01:1521/sales")

# Snowflake — pip install snowflake-sqlalchemy
engine = create_engine("snowflake://admin:pw@acct123/analytics")

# Teradata — pip install teradatasqlalchemy
engine = create_engine("teradatasql://admin:pw@teradata01/warehouse")

Install the driver first, then connect. Run pip install pyodbc, oracledb, snowflake-sqlalchemy, or teradatasqlalchemy depending on your database, then use the same create_engine() pattern.

pd.read_sql()

Run SQL and get a DataFrame

function

pd.read_sql() sends a SQL query to the database and returns the results as a DataFrame. Pass two arguments: the query string and the engine.

Key Parameters

sql The SQL query as a string
con The SQLAlchemy engine
params Values for parameterised queries
index_col Column to use as the DataFrame index
Python 
df = pd.read_sql("SELECT * FROM staff", engine)  # all rows
print(df.shape)                                    # (rows, cols)
print(df.head())                                   # first five

Always test queries with LIMIT 5 first. A table with millions of rows will freeze your script if you fetch everything at once.

SQL Query Strings

Write SQL inside Python strings

syntax

A SQL query string is a regular Python string that contains SQL commands. Use triple quotes for multi-line queries. Pandas sends it to the database unchanged.

Single-line vs Multi-line

Single line Triple quotes
Best for Short queries Long or complex queries
Readability Compact Easier to read
Example "SELECT ..." """SELECT ..."""
Python 
query = """
SELECT name, salary
FROM staff
WHERE salary > 50000
"""
df = pd.read_sql(query, engine)  # run multi-line query

Use triple quotes for anything over one line. They make SQL easier to read and you can add line breaks freely.

Parameterised Queries

Safe way to pass values into SQL

security

A parameterised query uses placeholders instead of pasting values directly into the SQL string. This prevents SQL injection — a serious security risk.

String Formatting vs Parameters

f-string (unsafe) params (safe)
SQL injection ✗ Vulnerable ✓ Protected
Placeholders None — value pasted in :name or %s
Values passed via f-string interpolation params={} dict
Python 
from sqlalchemy import text                              # wrap SQL safely
query = text("SELECT * FROM staff WHERE dept = :dept")   # :dept is a placeholder
df = pd.read_sql(query, engine, params={"dept": "Sales"})# value passed safely

Never paste user input into SQL with f-strings. Use params to pass values safely. This stops attackers from rewriting your query.

Create a Connection

Load a Table

03

Close Safely

Beginner Customers with statement context manager

What This Does

This script uses a with block to open and automatically close the connection, even if an error occurs.

safe_connect.py 
import pandas as pd
from sqlalchemy import create_engine

engine = create_engine("mssql+pyodbc://admin:pw@sqlserver01/customers?driver=ODBC+Driver+18+for+SQL+Server")

with engine.connect() as conn:               # auto-close on exit
    df = pd.read_sql("SELECT * FROM customers", conn)
print(df.shape)  # connection is already closed here
Terminal $ python safe_connect.py
(1500, 6)

Always use a with block for database connections — it guarantees cleanup even when your code hits an unexpected error.

Run a Query

02

Safe Parameters

Beginner Orders params SQL injection

What This Does

This script uses parameterised queries to insert values safely. Never paste user input directly into a SQL string.

safe_params.py 
import pandas as pd
from sqlalchemy import create_engine, text  # text() wraps raw SQL

engine = create_engine("mssql+pyodbc://admin:pw@sqlserver01/orders?driver=ODBC+Driver+18+for+SQL+Server")

Common Mistakes

Pitfalls beginners hit when working with database connections

Using an Incorrect Connection String Format

SQLAlchemy raises an ArgumentError because it cannot parse the URL.

Pitfall

Each database type needs its own URL prefix and driver. SQL Server uses mssql+pyodbc:// and requires a ?driver= parameter. Omitting the driver triggers ArgumentError.

Wrong — missing driver param 

engine = create_engine("mssql+pyodbc://admin:pw@server/sales")
# ArgumentError — missing ?driver= parameter

Correct — include driver 

from sqlalchemy import create_engine                               # import first
engine = create_engine("mssql+pyodbc://admin:pw@server/sales"     # dialect + driver
                       "?driver=ODBC+Driver+18+for+SQL+Server")   # driver param
print(engine)                                                      # confirms connection

Always add the ?driver= parameter for SQL Server. Run pyodbc.drivers() in Python to see which drivers are installed on your machine.

Opening a Connection Without Closing It

Open connections pile up and may lock the database or exhaust server resources.

Pitfall

Calling engine.connect() without a with block leaves the connection open. Use a context manager to close it automatically.

Wrong — no close 

conn = engine.connect()
df = pd.read_sql("SELECT * FROM sales", conn)
# connection stays open forever

Correct — use with block 

with engine.connect() as conn:                # auto-closes
    df = pd.read_sql("SELECT * FROM sales", conn)
print(df.shape)                                # connection closed here

A with block is like a self-closing door — you walk through, do your work, and the door shuts behind you without you remembering to close it.

Forgetting to Install the Database Driver Package

Python raises a ModuleNotFoundError or SQLAlchemy cannot find the dialect.

Pitfall

SQLAlchemy needs a separate driver for each database — pyodbc for SQL Server, oracledb for Oracle. Without the driver installed, create_engine() fails.

Wrong — driver missing 

engine = create_engine("mssql+pyodbc://admin:pw@server/sales?driver=ODBC+Driver+18+for+SQL+Server")
# ModuleNotFoundError: No module named pyodbc

Correct — install driver 

# first, in terminal: pip install pyodbc
from sqlalchemy import create_engine                                               # import
engine = create_engine("mssql+pyodbc://admin:pw@server/sales"                     # connect
                       "?driver=ODBC+Driver+18+for+SQL+Server")                   # works now

Check the SQLAlchemy docs for your database type. SQL Server needs pip install pyodbc, Oracle needs pip install oracledb.

Building SQL with an f-String Instead of params

The query runs fine but is vulnerable to SQL injection — no error is raised.

Pitfall

Pasting a variable directly into the SQL string with f"..." works in testing, but an attacker can inject malicious SQL through that variable. This is the single most common web-application vulnerability.

Wrong — f-string injection risk 

dept = input("Department? ")
q = f"SELECT * FROM staff WHERE dept = '{dept}'"
df = pd.read_sql(q, engine)  # attacker can inject SQL

Correct — use params 

from sqlalchemy import text
q = text("SELECT * FROM staff WHERE dept = :dept")
df = pd.read_sql(q, engine, params={"dept": dept})  # safe

If you see f"SELECT" anywhere in your code, replace it immediately with text() + params. No exceptions.

Fetching All Rows Without a LIMIT During Development

The script freezes or runs out of memory — no helpful error message.

Pitfall

A production table can have millions of rows. During development, fetching everything loads it all into memory at once, which can freeze your notebook or crash the kernel without any useful error.

Wrong — fetches everything 

df = pd.read_sql("SELECT * FROM transactions", engine)
# 50 million rows — script hangs for 10+ minutes

Correct — use LIMIT 

q = "SELECT * FROM transactions LIMIT 500"  # dev sample
df = pd.read_sql(q, engine)                  # finishes fast
print(df.shape)                              # (500, cols)

Always add LIMIT 500 while building your code. Remove it only for the final production run after you know the script works.

Two Habits That Prevent Every Connection Bug

1

Always use a with block

Wrap every engine.connect() call inside a with statement. The connection closes automatically when the block ends — even if your script hits an unexpected error halfway through.

2

Keep credentials out of your script

Store database passwords in environment variables or a .env file. Use os.getenv() at runtime so no sensitive data appears in your code, your Git history, or a pull request review.

The same create_engine() + pd.read_sql() pattern works across SQL Server, Oracle, Snowflake, and Teradata — you only change the connection string prefix and driver.
