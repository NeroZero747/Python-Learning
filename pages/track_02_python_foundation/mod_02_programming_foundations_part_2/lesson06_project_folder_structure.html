Overview

A high-level summary of the topic

Py

A project folder structure is the system of subfolders that gives every file in your Python project a clear, predictable home.

Think of your Python project like a filing cabinet: the whole cabinet is your project folder, each drawer holds a different type of file, and every document inside has a labeled home so you can find it in seconds. A project folder structure is that cabinet, and Python is the language you use to create everything inside it.

Why structure matters

The cabinet itself — tidy cabinets find things; messy desks don't

The bigger the project, the more a clear layout saves you time.

Organizing into folders

The labeled drawers — one for scripts, one for data, one for config

Each drawer holds one type of file so you always know where to look.

One script, one job

The document in each drawer — focused on a single task

Focused files let you fix one step without touching everything else.

The standard layout

The cabinet model — a format any colleague navigates instantly

One shared layout means any new project starts with a familiar structure.

If you already create folders on your desktop to separate work by project or date, you already understand the thinking — this lesson shows you the standard way Python projects apply that same habit.

Key Concepts

Project root, modules, data folder, and entry point — with examples for each.

Project Root

The master folder that holds everything

folder

The project root is the single top-level folder that contains every file and subfolder in your project. You create one root folder per project, and every path in your code is relative to it.

Folder Naming Rules

No spaces — use _ instead Sales Analysis ✗ → sales_analysis ✓
Describe the project, not the date project_march ✗ → customer_churn ✓
Use underscores, not hyphens hr-data → prefer hr_data ✓
Bash 
sales_analysis/           # ← the project root folder
├── main.py               # entry point — run this to start everything
├── modules/              # one .py file per task (cleaning, exports …)
│   ├── cleaning.py
│   ├── calculations.py
│   └── exports.py
├── data/                 # all data files live here — never inside modules/
│   ├── raw/              # original source files — never overwrite these
│   └── processed/        # output your scripts produce
└── config.env            # file paths, passwords, settings

Keep your root at a stable path. Python builds file paths from the root, so moving the root folder will break your import statements. Put it in your Documents folder and leave it there.

Module

A .py file with one clear responsibility

.py file

A module is a single Python file that handles one specific task, such as cleaning data or calculating totals. You import it into your main script instead of copying and pasting the same code into every project.

Module Naming Rules

All lowercase letters Cleaning.py ✗ → cleaning.py ✓
Separate words with _ datacleaning.py → prefer data_cleaning.py
Name by task, not by person johns_script.py ✗ → exports.py ✓
Python 
# modules/cleaning.py — handles data cleaning only
def remove_blanks(df):
    return df.dropna()        # drops every row that has an empty cell

# main.py — imports and uses the cleaning module
from modules.cleaning import remove_blanks  # bring in the function
clean_df = remove_blanks(raw_df)            # use it without copy-pasting

One module, one job. If cleaning.py starts calculating totals, split it. A module that does too much is just as hard to navigate as one enormous file.

Data Folder

Separate raw data from processed results

data/

The data folder keeps all your CSV and Excel files separate from your Python scripts. Split it into a raw/ subfolder for original files and a processed/ subfolder for cleaned outputs — so you never overwrite your source data by accident.

raw/ vs processed/

raw/ processed/
What goes here Unedited original files Files your scripts produced
Write to it? ✗ Never ✓ Every run
Example file sales_jan.csv sales_jan_clean.csv
Python 
import pandas as pd

# Read from raw/ — treat this folder as read-only
df = pd.read_csv("data/raw/sales_jan.csv")

# Save cleaned result to processed/ — safe to overwrite each run
df.dropna().to_csv("data/processed/sales_jan_clean.csv", index=False)

Never write back to raw/. Treat those files as backup originals — if a script corrupts its output, you can always reprocess from data/raw/ without losing anything.

Entry Point

The one script that starts your pipeline

main.py

The entry point is the single script you run to start your entire project — usually called main.py. It imports functions from your modules and calls them in order. You never run the individual module files directly.

main.py vs modules/

main.py modules/
Purpose Coordinates the pipeline Does one specific task
Contains Imports + function calls Function definitions
Run directly? ✓ Yes ✗ No — imported only
Python 
# main.py — the entry point; run: python main.py
from modules.cleaning import remove_blanks   # step 1: import the cleaner
from modules.calculations import get_totals  # step 2: import the calculator
from modules.exports import save_to_csv      # step 3: import the exporter

df     = remove_blanks("data/raw/sales.csv") # clean the data
totals = get_totals(df)                       # calculate totals from clean data
save_to_csv(totals, "data/processed/")        # write results to processed/

main.py should read like a checklist. If you open it and can't tell what the project does in 10 seconds, split more tasks into separate module files.

The __init__.py File

What it is and why every modules/ folder needs one

Python tells the difference between a plain folder and an importable package by looking for a special file called __init__.py inside that folder. When Python sees it, it treats the whole folder as a package and lets you use from modules.cleaning import …. Without it, Python has no way to know the folder holds code you want to import, and the statement fails with a ModuleNotFoundError.

Without __init__.py

modules/
└── cleaning.py

→ import fails with ModuleNotFoundError

With __init__.py

modules/
├── __init__.py
└── cleaning.py

→ import works correctly

What goes inside __init__.py?

Almost always — nothing. The file exists purely so Python can recognize the folder. Leave it completely empty. The easiest way to create it in your setup script is one line of Python:

# one line — creates an empty file and closes it immediately
open("modules/__init__.py", "w").close()

Python 3.3+ note: Modern Python sometimes allows imports without __init__.py through a feature called namespace packages. Even so, always creating it is best practice — it makes your intention clear and prevents subtle import errors that can be very hard to trace.

Code Examples

Hands-on code snippets to explore the concepts

These three examples form one project. Step through them in order and you will have a working sales_report/ project — folders set up, a reusable module written, and a pipeline that runs from a single command.

01

Set Up the Project

Beginner os.makedirs() __init__.py Project Setup

What This Does

This script creates the standard sales_report/ folder structure using Python's built-in os module, then creates the empty modules/__init__.py file that tells Python the modules/ folder is a package. Run it once before you write any analysis code.

setup_project.py 
import os

os.makedirs("data/raw",       exist_ok=True)  # folder for raw source files
os.makedirs("data/processed", exist_ok=True)  # folder for cleaned output files
os.makedirs("modules",        exist_ok=True)  # folder for your reusable scripts

open("modules/__init__.py", "w").close()      # marks modules/ as importable

print("Folders created!")
Terminal $ python setup_project.py
Folders created!

exist_ok=True means the script is safe to re-run — Python won't raise an error if a folder already exists. Add a new folder path any time your project grows.

02

Write the Cleaning Module

Beginner def return modules/cleaning.py if __name__

What This Does

This file defines a remove_blanks() function and saves it as modules/cleaning.py. Any script in your project can import and call it — you only ever write the logic once.

modules/cleaning.py 
# modules/cleaning.py — save this inside the modules/ folder

def remove_blanks(names):
    result = []
    for name in names:
        if name != "":         # skip any blank entry
            result.append(name)
    return result
Terminal $ python modules/cleaning.py
No output — modules just define functions. The result appears in Step 3 when main.py imports this file.

Give each module one job. If cleaning.py starts doing calculations too, move that logic to a separate calculations.py file.

03

Run the Full Pipeline

Beginner from … import main.py Pipeline

What This Does

This is main.py — the only file you run. It imports remove_blanks from the module you created in Step 2 and calls it on a short list of names. You can see the module doing its job in the output below.

main.py 
# main.py — run this file: python main.py

from modules.cleaning import remove_blanks  # import the function from our module

names = ["Alice", "", "Bob", "", "Carol"]   # raw list — some entries are blank

clean = remove_blanks(names)               # remove the blanks using the module

print("Clean list:", clean)                # show the result
print("Count:", len(clean))                # how many names remain
Terminal $ python main.py
Clean list: ['Alice', 'Bob', 'Carol']
Count: 3

main.py contains no cleaning logic — it just calls the module. If you change the rules in modules/cleaning.py, every script that imports it gets the update automatically.

Practice Exercises

Guided exercises to reinforce your learning

01

Create Project Folders

Beginner Analytics os.makedirs() Folders

Your Task

You are starting a new analytics project. Write a Python script called setup.py that creates three folders: data/raw, data/processed, and modules. When you run it, it should print Folders created! to confirm everything is in place.

setup.py 
import os                                      # import the os module for folder tools

os.makedirs("data/raw",       exist_ok=True)   # create the raw data folder
os.makedirs("data/processed", exist_ok=True)   # create the processed data folder
os.makedirs("modules",        exist_ok=True)   # create the modules folder

print("Folders created!")                      # confirm the setup is done
Terminal $ python setup.py
Folders created!

Running setup.py once at the start of every new project makes sure all your folders exist — and exist_ok=True means you can run it again later without getting an error.

02

Write a Module

Beginner Students def Modules

Your Task

You are building a reusable module for a student grading project. Create a file called modules/grades.py that defines a function called pass_fail(score). It should return "Pass" if the score is 50 or above, and "Fail" if it is lower. Add two print() calls at the bottom to test that it works with a passing and a failing score.

modules/grades.py 
def pass_fail(score):         # define the function — accepts a student score
    if score >= 50:           # check if the score meets the pass threshold
        return "Pass"         # return Pass for scores 50 and above
    return "Fail"             # return Fail for anything below 50

print(pass_fail(72))          # test the function — should print Pass
print(pass_fail(35))          # test the function — should print Fail
Terminal $ python modules/grades.py
Pass
Fail

Defining the logic inside a function means any other script in your project can import pass_fail and use it without copying the code again.

03

Import and Use It

Beginner Sales import Functions

Your Task

You already have modules/grades.py with a pass_fail function. Create a new file called main.py in the project root that imports that function and uses it on two sales-target scores: 91 and 44. Print each result on its own line.

main.py 
from modules.grades import pass_fail    # import the function from the module

target_1 = 91                           # first sales-target score
target_2 = 44                           # second sales-target score

print("Target 1:", pass_fail(target_1)) # call the function and print the result
print("Target 2:", pass_fail(target_2)) # call the function again for the second score
Terminal $ python main.py
Target 1: Pass
Target 2: Fail

Using from modules.grades import pass_fail tells Python exactly where to find the function — your main.py stays focused on running the pipeline, not on how the logic works.

Common Mistakes

Frequent errors and how to avoid them

Importing a Module Without Marking its Folder as a Package

Python raises a ModuleNotFoundError because modules/ is not recognized as a package without an __init__.py file.

Pitfall

Python treats a folder as a package only when it contains an __init__.py file — without it, from modules.cleaning import remove_blanks raises a ModuleNotFoundError. Create an empty __init__.py inside modules/ to fix it.

Wrong — no package marker 

# modules/ has no __init__.py file yet
from modules.cleaning import remove_blanks   # raises ModuleNotFoundError

Correct — add the package marker 

open("modules/__init__.py", "w").close()    # mark modules/ as a Python package
from modules.cleaning import remove_blanks  # now Python can find the module

Think of __init__.py as a name badge on the folder — without it, Python doesn't recognize modules/ as a package it can import from.

Running a Script from Inside a Subfolder Instead of the Project Root

Python builds import paths from the folder where you run the script — running from the wrong folder causes a ModuleNotFoundError.

Pitfall

When you run python main.py from inside modules/ instead of the project root, Python searches for modules/cleaning.py relative to the wrong starting point and raises a ModuleNotFoundError. Always run your script from the top-level project folder.

Wrong — run from inside a subfolder 

cd modules
python main.py   # ModuleNotFoundError: No module named 'modules'

Correct — run from the project root 

cd sales_report          # navigate to the project root folder
python main.py           # Python finds modules/ from here

Think of the project root as the front door of the building — Python always starts looking for rooms (modules) from that front door, so that is where you need to be when you run your script.

Putting All Project Logic into a Single Script

A single large script runs but becomes hard to read, test, and reuse as the project grows.

Pitfall

Writing all data loading, cleaning, and reporting logic into one main.py runs without errors but creates a file that grows unmanageable fast. Move reusable functions into separate files in modules/ so each file has one clear purpose.

Wrong — all logic in one file 

def remove_blanks(names):         # cleaning logic buried in main.py
    return [n for n in names if n]
def generate_report(clean_data):  # reporting logic also in main.py
    print("Report:", clean_data)
generate_report(remove_blanks(["Alice", "", "Bob"]))

Correct — split into modules 

from modules.cleaning import remove_blanks   # import cleaning from its own file
from modules.reports import generate_report  # import reporting from its own file
raw_names = ["Alice", "", "Bob"]             # the raw data to process
generate_report(remove_blanks(raw_names))    # run the full pipeline

Think of it like a workbook with one sheet that holds every formula, every raw data dump, and every chart — it works, but nobody wants to open it. Separate sheets (separate modules) keep each piece easy to find and update.

Real-World Use

How project folder structure is used across real-world workflows

A data analytics team might run a dozen scripts spread across one flat folder — loading data, cleaning it, running calculations, and building reports all jumbled together. When a new file lands in the folder, nobody knows which script to run or whether editing one formula will break something else. A standard project layout puts each piece in a named location, so the whole team can find, update, and reuse any part of the pipeline without confusion.

12 scripts in one folder —
which one cleans the data?

Three analysts each have a slightly different copy of the same cleaning logic. Store it once in modules/cleaning.py and every analyst imports the same tested version — one edit fixes it for everyone.

returns shared logic

Raw file updated overnight —
processed output gone

Data files and scripts in one flat folder mean a fresh download can silently overwrite a finished report. Split into data/raw/ and data/processed/ and each folder has one clear purpose — source files never touch output files.

returns safe data paths

New teammate joins —
which file runs first?

Without a clear structure, a new team member has to read every script before finding the right starting point. Place one main.py at the project root and the entry point is unmistakable — open it, run it, done.

returns entry point

Without project structure

Three analysts each maintain a different copy of the same cleaning logic — one update means three files to edit, and one is always missed.

Raw and processed data share one folder — a fresh download silently overwrites a finished report.

A new team member reads every script to find the entry point — guessing which file to run wastes time every time.

With project structure

Store the logic once in modules/cleaning.py — every analyst imports the same tested version and one edit updates everyone's result.

data/raw/ holds source files and data/processed/ holds outputs — a fresh download never touches finished reports.

main.py at the project root is the unmistakable entry point — any team member can open it and run the full pipeline immediately.
