Overview

A high-level summary of the topic

Py

Git is a tool that remembers every change you make to your code, so you can always go back to a version that worked.

Think of your code project as a notebook — Git is the diligent note-taker who records every change, dates each entry, and keeps every page so nothing is ever truly deleted. A repository is that notebook, and Git is the system that manages it.

Repository

The notebook — holds the whole project history

The notebook itself, containing every page ever written.

Commit

A dated entry — a snapshot you can return to

A finished, dated page filed in the notebook permanently.

Staging Area

The draft page — choose what to record

The draft page where you gather changes before filing them.

Remote (GitLab)

The shared shelf — your team's copy of the notebook

The shared copy your teammates can read and add to.

If you've used "Track Changes" in Word or version history in Google Sheets, Git gives you that same safety net for your code — with far more control over what gets saved and why.

Key Concepts

Repository, commit, staging area, and remote — the four building blocks of Git.

Repository

Your project's tracked home folder

repo

A repository is the folder where Git watches your project. Run git init once inside your project folder and Git creates a hidden .git folder that records every change you make from that moment on.

Repository Rules

One repo per project sales_project/ web_app/ ✓
Never commit secrets .env ✗ → .gitignore
Use lowercase names my-project over My Project
Bash 
# Create a new folder for your project
mkdir my_project

# Move into that folder
cd my_project

# Tell Git to start tracking this folder
git init

Run git init only once per project. A hidden .git folder will appear — this is where Git stores its entire history. Never delete it or you will lose all your saved snapshots.

Commit

A saved snapshot of your project

snapshot

A commit is a permanent snapshot of all your files at one moment in time. Every commit records what changed, who changed it, and when — like a dated entry in a project diary you can always return to.

Commit Commands

Command What it does Result
git add . Stage all changes files ready to save
git commit -m Save a snapshot new entry in history
git log View past commits list of snapshots
Bash 
# Stage all changed files — prepare them to be saved
git add .

# Save a snapshot with a message describing what changed
git commit -m "Add revenue calculation function"

# View the list of commits you have made so far
git log --oneline

Write commit messages in plain English. A good message finishes the sentence "This commit will…" — for example, git commit -m "Fix total price rounding error" tells your future self exactly what changed and why.

Staging Area

The waiting room before a commit

stage

The staging area is a preparation zone between your files and a commit. You place files there with git add. Only staged files are included in your next commit, so you stay in full control of exactly what gets saved.

Unstaged vs Staged

Unstaged Staged
Git sees it? ✓ Yes ✓ Yes
In next commit? ✗ No ✓ Yes
How to get here (edit a file) git add
Bash 
# Check which files have changed since your last commit
git status

# Stage one specific file — only this file will be saved
git add analysis.py

# Stage every changed file at once
git add .

git add . stages everything. If you only want to save one file, use git add filename.py instead — this keeps unfinished work out of your commit and makes your history easier to read.

Remote / GitLab

Sharing your commits online

remote

A remote is an online copy of your repository, usually hosted on GitLab. After you commit locally, you use git push to upload those commits so teammates can see and download your work.

Local to Remote

Local Repo Your machine
→
git push Upload commits
→
GitLab ✓
Bash 
# Link your local repo to a GitLab repository (one-time setup)
git remote add origin https://gitlab.com/you/my_project.git

# Upload your commits to GitLab for the first time
git push -u origin main

# Upload subsequent commits after the first push
git push

git push only uploads commits. Unsaved file edits stay on your machine until you run git add and git commit first — then push to share your work on GitLab.

Decision Flow

A visual walkthrough of the logic and process

Every time you use Git, you follow the same repeatable path known as the stage → commit → push cycle: you edit files on your computer, tell Git exactly which changes to include, save a labeled snapshot, then upload it to GitLab — just like drafting a report, marking the sections worth keeping, printing a final version, and filing it away.

Git’s Three Zones — Where Your File Lives at Each Step

Working Directory

Zone 1

Your normal file system. When you edit a file here, Git notices the change but does not save a snapshot yet — the file is simply marked as “modified.”

modified: report.py ← what git status shows
git add report.py — moves the file to the staging area

Staging Area

Zone 2

A holding zone inside Git where you choose which changes go into the next snapshot. You can stage some files and leave others out before committing.

Changes to be committed: report.py ← what git status shows
git commit -m "Add Q1 sales report" — freezes staged files into a permanent snapshot

Local Repository

Zone 3

Your personal snapshot library stored on your own machine. Every commit lives here permanently — you can rewind to any of them at any time before pushing.

commit abc1234 — “Add Q1 sales report” ← what git log shows
git push origin main — uploads all local commits to GitLab

Remote Repository (GitLab)

Your team now sees all commits. One shared source of truth.

Zone 4

One Direction — You Cannot Skip a Zone

A file always moves through the zones in order: Working Directory → Staging Area → Local Repository → Remote. Running git push before git add and git commit will upload nothing new because no snapshots exist yet in Zone 3. Use git status any time to check which zone your changes are currently in.

Here is what the full cycle looks like in the terminal, starting from a folder that is already a Git repository:

git_workflow.sh 
# Step 1 — Check what files have changed
git status                       # shows modified and untracked files

# Step 2 — Stage the changes you want to include
git add report.py                # stage one specific file
git add .                        # or stage everything at once

# Step 3 — Save a snapshot with a clear, descriptive message
git commit -m "Add Q1 sales report script"   # locks in the snapshot

# Step 4 — Upload to GitLab so your team can access it
git push origin main             # sends all commits to the remote repository

The Three Commands That Drive the Git Cycle

Each command plays a distinct role — swapping the order or skipping one will leave your work either unsaved or unshared with your team.

git add

The staging area — what goes in?

Running git add moves your changed files into the staging area — a holding zone where you decide which edits go into the next snapshot. Use git add . to stage every change at once.

git commit

The snapshot — locked in forever

Running git commit -m "message" freezes everything in the staging area into a permanent, labeled snapshot. Write your message in the present tense and describe what changed, for example: "Fix login form validation".

git push

The upload — share your work

Running git push origin main uploads your local commits to GitLab so your team can access them. Git only pushes committed snapshots — you must complete git add and git commit first.

The most common beginner mistake is editing files and then trying to jump straight to git push without running git add and git commit first. Run git status any time you are unsure where you are in the cycle — it tells you exactly what Git has noticed and what is still waiting to be staged or committed.

Code Examples

Hands-on code snippets to explore the concepts

01

Your First Commit

Beginner Reports git init, git add, git commit

What This Does

These four commands turn a plain folder into a tracked project and save your first snapshot. git init creates the hidden .git folder that Git uses to remember every future change.

first_commit.sh 
git init                               # turn this folder into a Git repository
git add report.py                      # tell Git to start tracking this file
git commit -m "Add report script"      # lock in the first snapshot with a label
git log --oneline                      # show the snapshot you just created
Terminal $ bash first_commit.sh
e4f5a12 Add report script

You only run git init once per project — after that, every new snapshot needs only git add and git commit.

02

Stage the Right Files

Beginner Analysis Scripts git status, git add

What This Does

This runs git status before and after staging so you can see exactly which files Git is tracking. Staging one file at a time gives you precise control over what goes into each snapshot.

stage_files.sh 
git status                             # show all changed and untracked files
git add analysis.py                    # stage this one file only
git status                             # confirm the file is now in the staging area
git commit -m "Add data analysis script"  # save the staged snapshot
Terminal $ bash stage_files.sh
[main 9c3d2e1] Add data analysis script

Run git status before every commit — it shows exactly what Git has noticed so you never accidentally stage the wrong file.

03

Write a Clear Message

Beginner Dashboards git commit -m, git log

What This Does

This commits one changed file with a descriptive message, then shows the history. A good commit message tells your team (and your future self) exactly what changed and why — not just that something was “updated”.

commit_message.sh 
git add dashboard.py                         # stage the changed file
git commit -m "Fix column total formula"      # describe what changed, not what you did
git log --oneline                            # review the snapshot history
Terminal $ bash commit_message.sh
7a3b2c1 Fix column total formula
e4f5a12 Add report script

Write your message to complete the sentence "If applied, this commit will…" — that forces you to describe the change rather than just say "misc fixes" or "updated file".

04

Push to GitLab

Beginner Team Projects git remote add, git push

What This Does

This runs the full stage → commit → push cycle and uploads your work to GitLab. git push origin main only uploads committed snapshots — you must run git add and git commit first.

push_to_gitlab.sh 
git add .                                               # stage every changed file at once
git commit -m "Add Q1 forecast model"                   # save the snapshot locally
git remote add origin https://gitlab.com/you/project.git  # link to GitLab (one-time setup only)
git push origin main                                    # upload all commits to GitLab
Terminal $ bash push_to_gitlab.sh
Branch 'main' set up to track remote branch 'main' from 'origin'.

You only need git remote add once per project — after that, git push origin main is all you need to share new commits with your team.

SQL / Excel Comparison

How this topic compares across Python, SQL, and Excel

If you already save multiple copies of an Excel file or rely on SQL backups to recover data, you already understand why version control matters. Git gives you the same safety net for your code — but automatically, with a clear history of every change and the author who made it.

Excel's built-in tool highlights edits so you can accept or reject each one before saving the final copy.

Locks your staged changes into a permanent, labeled snapshot you can return to at any time.

All team members query the same live database, so committed changes are immediately visible to everyone.

All three save a permanent record of a change to a project — the same idea, just written in a different tool.

The version control idea is the same whether you use Git, SQL, or Excel — you are always making deliberate, labeled saves so you can recover earlier states. Git just replaces manual file naming and copy-pasting with a single command that records who changed what and when.

01

Save Your First Commit

Beginner Reports git init git commit

Your Task

You have just written a quarterly sales report and saved it as q1_report.md. Initialize a Git repository in the project folder, stage the file, and commit it with a clear message. Run git log --oneline at the end to confirm Git recorded your snapshot.

first_commit.sh 
git init                                   # create a new Git repository in this folder
git add q1_report.md                       # stage the report so Git will include it

Running git log --oneline after each commit confirms Git saved your work. Every line in the log is a permanent record — if you need to undo a change later, this list tells you exactly where to go back to.

02

Stage One File

Beginner Analysis Scripts git add git status

Your Task

You edited two scripts today — clean_data.sh and explore_data.sh — but only clean_data.sh is ready to save. Stage just that one file and commit it with a clear message. Use git status before and after staging to see exactly which files Git is tracking.

stage_one.sh 
git status                                        # check which files have changed
git add clean_data.sh                             # stage only the data-cleaning script
git status                                        # confirm only one file is staged

[main 4f5e6d7] Clean null values in source data

Staging files individually keeps each commit focused on one change. Your team can read the history later and understand exactly what changed and why — without digging through unrelated edits bundled into the same snapshot.

03

Push to GitLab

Beginner Team Projects git push git remote

Your Task

You have committed a sales pipeline script locally and need to share it with your team on GitLab. Stage and commit the file, link your local repository to a remote GitLab project, and push your commits. Your teammates can then pull your changes and continue the work.

push_to_gitlab.sh 
git add sales_pipeline.sh                                    # stage the updated pipeline script
git commit -m "Add monthly sales aggregation step"           # save the snapshot locally
git remote add origin https://gitlab.com/team/pipeline.git   # link this repo to GitLab
git push -u origin main                                      # upload all commits to GitLab
Terminal $ bash push_to_gitlab.sh
Branch 'main' set up to track origin/main.
* [new branch] main -> main

The -u flag in git push -u origin main sets GitLab as the default destination. After the first push, typing just git push is enough — Git remembers where to send your commits every time.

Common Mistakes

Frequent errors and how to avoid them

Using a Vague Message Like "update" or "fix"

Git records your message forever — a vague message makes it impossible to understand what changed weeks later.

Pitfall

Writing git commit -m "update" creates a permanent snapshot with no useful information — your future self and your teammates cannot tell what actually changed. Replace the message with a short imperative phrase that names the specific change, such as "Add Q1 sales aggregation step".

Wrong — vague message 

git commit -m "update"

Correct — specific imperative phrase 

git commit -m "Add Q1 sales aggregation step"  # names the exact change

Think of a commit message like the subject line of an email — you can open the commit to see the full diff, but a subject of "stuff" makes the history impossible to scan. Aim for a message that tells you what changed without having to open it.

Running git add . Without Checking What Will Be Staged

git add . stages every changed file — including scratch files, credentials, and large data exports you never meant to commit.

Pitfall

git add . stages all modified files at once — including temporary files, .env files containing API keys, or multi-gigabyte data exports. Run git status first to review which files are modified, then stage only the files you intend to commit using git add FILENAME.

Wrong — stage everything blindly 

git add .                   # stages all files — including secrets
git commit -m "Add changes"

Correct — check then stage selectively 

git status                            # review what has changed
git add sales_report.sh               # stage only the intended file
git commit -m "Add Q1 sales summary"  # save just that change

Think of git add . like forwarding an entire email thread — you might accidentally share something in the history that was meant to stay private. Use targeted git add FILENAME to keep each commit clean and intentional.

Treating a Local Commit as if the Team Can Already See It

git commit saves a snapshot on your machine only — teammates see nothing on GitLab until you run git push.

Pitfall

git commit writes a snapshot to your local repository and does not contact the remote server at all — teammates checking GitLab will still see the old version. Run git push after committing to upload your snapshots so the team can see them.

Wrong — commit but never push 

git add analysis.sh
git commit -m "Add cleaning step"
# team still sees the old version on GitLab

Correct — commit then push 

git add analysis.sh
git commit -m "Add cleaning step"  # save snapshot locally
git push origin main               # upload so teammates can see it

Think of git commit as saving a document to your own hard drive and git push as uploading it to a shared folder. Your colleagues can only open the file after the upload — a commit alone is never enough to share your work.

Real-World Use

How Git is used across real data team workflows

Every data team produces dozens of scripts, queries, and reports that change week after week. Without version control, a single accidental overwrite can erase hours of work — and no one can tell what changed or who changed it. Git gives every file a full edit history you can search, compare, and rewind at any point.

Three Moments These Commits Saved Time

Finding the broken formula on Friday afternoon

a4f2c1b

The revenue total looked wrong in the weekly report. Because every change was committed with a clear message, running git log showed exactly which commit changed the formula — no manual file-by-file scanning needed.

A teammate joins mid-week and gets up to speed instantly

git log

A second analyst needed to understand what had already been built. Instead of reading every script line by line, she ran git log --oneline and read five commit messages — the full project story in under 30 seconds.

Rolling back to a working version in two minutes

git checkout

A new cleaning rule accidentally broke an older report. Because the previous working state had a clean commit, the analyst could restore the exact version that worked — without losing any of the other changes that had already been committed separately.
